01文書概要

1.1 目的

本書は、帯域制御デバイス（FITELnet FX2）が送信するトラヒック統計ファイル（FLOW・SUBPORT）を TimescaleDB に定期取り込みし、時系列で蓄積するシステムの要求仕様を定義する。本仕様を満たすシステムを設計・実装すること。

1.2 適用範囲

  • FLOW ファイル（CSV/GZ）の取り込み処理
  • SUBPORT ファイル（CSV/GZ）の取り込み処理
  • TimescaleDB へのデータ蓄積・保持・圧縮
  • 処理スケジュール（cron）・冪等性制御・エラーハンドリング
  • 監視・ログ出力

02用語定義

用語定義例
FLOW ファイル 帯域制御デバイスが5分ごとに送信するトラヒック統計ファイル。命名: FLOW-<hostname>-<YYYYMMDDHHmmss>.csv.gz FLOW-router01-20260515140000.csv.gz
SUBPORT ファイル サービス（subport）単位のトラヒック統計ファイル。命名: SUBPORT-<hostname>-<YYYYMMDDHHmmss>.csv.gz SUBPORT-router01-20260515140000.csv.gz
flow_id 回線識別子。フォーマット: <回線番号>@<IPv6プリフィックス>@<subport> 0123456789@2001:db8::/32@AB12-03-0456
subport サービス識別子。フォーマット: XX00-00-0000（ISP コード-サービスコード-地域コード） AB12-03-0456
service_id subport から地域コード（末尾4桁）を除いた部分。別名：帯域制御グループ AB12-03（= AB12-03-0456 から -0456 を除いた部分）
flow_id_hash flow_id を BIGINT にハッシュ化した値。PRIMARY KEY として使用 hashtext('0123456789@2001:db8::/32@AB12-03-0456')
バケット 時刻を切り捨てた集計単位。flow_stats: 1時間、subport_stats: 5分 2026-05-15 14:00:00（1時間バケット）
stable_sec ファイル転送完了判定の待機秒数 60（秒）
冪等性 同一ファイルを複数回処理しても結果が変わらない性質 同一ファイル再実行時に import_log でスキップ
import_log 処理済みファイルを記録し冪等性を確保するテーブル filename = 'FLOW-router01-20260515140000.csv.gz'

03flow_id フォーマット定義

flow_id は FLOW ファイルの各行に含まれる回線識別子です。以下の 3 要素を @ 区切りで連結した文字列です。

回線番号
0123456789
@
IPv6 プレフィックス
2001:db8::/32
@
subport（地域コードを含む）
AB12-03-0456
第3セグメント = 地域コード

フィールド仕様

フィールド名位置説明例
回線番号 @ 区切り 第1フィールド 加入者回線を識別する数字列 0123456789
IPv6 プレフィックス @ 区切り 第2フィールド CIDR 表記の IPv6 プレフィックス 2001:db8::/32
subport @ 区切り 第3フィールド 帯域制御ルール名。ETL は flow_id をこのフィールドで分割して subport を抽出する AB12-03-0456
ETL は flow_id を @ で分割し、第3フィールドを subport として抽出します。さらに subport の末尾セグメント（- 区切り）を除いた部分を service_id として導出します。詳細は「04 subport フォーマット定義」を参照。

04subport フォーマット定義

subport は帯域制御ルール名です。ISP コード・サービスコード・地域コードの 3 要素をハイフンで連結した文字列です。

ISP コード
AB12
英大2桁 + 数字2桁
-
サービスコード
03
数字2桁
-
地域コード
0456
数字4桁

フィールド仕様

セグメントフォーマット説明例
第1セグメント XX00（英大文字2桁+数字2桁） ISP コード。ISP を識別する固定値。 AB12
第2セグメント 00（数字2桁） サービスコード。プランや契約種別を識別する。 03
第3セグメント 0000（数字4桁） 地域コード。FX2 装置の設置エリアを識別する。 0456
service_id との関係： subport から第3セグメント（-0456）を除いた部分が service_id（例: AB12-03）です。service_id は SUBPORT ファイルおよび flow_stats の集計キーとして使用されます。

8.1 ファイル転送完了検知

ファイルの最終更新時刻が 60秒（stable_sec）以上経過したものを転送完了と判定します。WAN での転送を想定し、余裕を持たせています。

8.2 冪等性設計

  • import_log テーブルにファイル名を記録し、再実行時の二重取り込みを防止すること
  • status に関わらずファイル名だけで二重処理チェックを行うこと
  • 処理完了ファイルは import_log 記録後に削除すること（再処理防止）

8.3 FLOW 処理要件

インプット

  • ファイル配置場所: data/incoming/（FLOW-*.csv.gz）イメージ
  • 転送完了判定: ファイルの最終更新から 60秒以上経過したものを処理対象とする
  • ファイル形式: 第6章「入力ファイル仕様」参照

処理要件

1
import_log による重複処理防止
処理済みファイルは import_log テーブルで管理し、重複処理を防止すること。
↓
2
遅延ファイル検出・隔離
flow_stats に既存バケットが存在するファイル（遅延ファイル）を検出し、隔離ディレクトリ（別途要相談）へ退避の上 WARNING ログを出力すること。
↓
3
gzip 展開・1時間バケット集計
各ファイルを gzip 展開・CSV 読込し、flow_id と 1時間バケットで集計すること（volume_in・volume_out・dropped_packets_in・dropped_bytes_in を合算）。
↓
4
subport 抽出
flow_id から subport を抽出すること（@ 区切りの第3フィールド）。
↓
5
import_log 記録・ファイル削除
処理済みファイルを import_log に記録後、削除すること。
↓
6
flow_stats 確定書き込み（毎時 :05）
毎時 :05 に、確定済みバケット（現在時刻の1時間バケット未満）のデータを flow_stats へ書き込むこと（冪等性: 既存データを削除してから INSERT）。

アウトプット

  • 書き込み先: flow_stats（PRIMARY KEY: flow_id_hash, bucket）
  • 集計キー: flow_id_hash（flow_id の BIGINT ハッシュ値）+ 1時間バケット
  • 導出カラム: subport（flow_id の @ 区切り第3フィールド）
  • 現在時バケットは毎時 :05 時点でスキップし、翌時 :05 に確定すること

SLA

  • 毎5分バッチ受信処理: SLA 5分以内（警告閾値 3分超）
  • 毎時:05 flow_stats 確定書き込み: SLA 10分以内（警告閾値 5分超）

10想定 SELECT クエリ

カスタマーコントローラーが TimescaleDB に対して発行する代表的な SELECT クエリを示します。パラメータは :subport 等のプレースホルダーで表記しています。

10.1 flow_stats クエリ例

#フィルタ粒度取得カラム使用インデックス
Q1subport 指定1時間バケット（生値）flow_id ごとの volume_in / volume_outix_flow_stats_subport_bucket
Q2service_id 指定1時間バケット（生値）flow_id ごとの volume_in / volume_outix_flow_stats_service_id_bucket
Q3subport 指定1日集計（time_bucket）flow_id ごとの dropped_packets_in 合計ix_flow_stats_subport_bucket
Q4service_id 指定1日集計（time_bucket）flow_id ごとの dropped_packets_in 合計ix_flow_stats_service_id_bucket
 
-- Q1: subport ごとに各 flow_id の 1時間バケット volume_in / volume_out
SELECT bucket, flow_id, volume_in, volume_out
FROM   flow_stats
WHERE  subport = :subport
  AND  bucket >= now() - INTERVAL '7 days'
ORDER BY bucket DESC, flow_id;

-- Q2: service_id ごとに各 flow_id の 1時間バケット volume_in / volume_out
SELECT bucket, flow_id, volume_in, volume_out
FROM   flow_stats
WHERE  service_id = :service_id
  AND  bucket >= now() - INTERVAL '7 days'
ORDER BY bucket DESC, flow_id;

-- Q3: subport ごとに各 flow_id の 1日単位 dropped_packets_in 合計
SELECT time_bucket('1 day', bucket) AS day,
       flow_id,
       SUM(dropped_packets_in) AS dropped_packets_in
FROM   flow_stats
WHERE  subport = :subport
  AND  bucket >= now() - INTERVAL '30 days'
GROUP BY day, flow_id
ORDER BY day DESC, flow_id;

-- Q4: service_id ごとに各 flow_id の 1日単位 dropped_packets_in 合計
SELECT time_bucket('1 day', bucket) AS day,
       flow_id,
       SUM(dropped_packets_in) AS dropped_packets_in
FROM   flow_stats
WHERE  service_id = :service_id
  AND  bucket >= now() - INTERVAL '30 days'
GROUP BY day, flow_id
ORDER BY day DESC, flow_id;

#粒度取得カラム使用インデックス

-- Q5: 各 subport の 5分バケット volume_in / volume_out（全ホスト合算）
SELECT bucket,
       subport,
       SUM(volume_in)  AS volume_in,
       SUM(volume_out) AS volume_out
FROM   subport_stats
WHERE  subport = :subport
  AND  bucket >= now() - INTERVAL '1 hour'
GROUP BY bucket, subport
ORDER BY bucket DESC;

-- Q6: 各 subport の 1時間単位 dropped_packets_in 合計
SELECT time_bucket('1 hour', bucket) AS hour,
       subport,
       SUM(dropped_packets_in) AS dropped_packets_in
FROM   subport_stats
WHERE  subport = :subport
  AND  bucket >= now() - INTERVAL '24 hours'
GROUP BY hour, subport
ORDER BY hour DESC;

11SLA・非機能要件

11.1 SLA 定義

処理SLA（最大）警告閾値違反時対応
FLOW バッチ受信処理（毎5分）CRITICAL 5分（300秒） 3分超（180秒） 即座に確認
FLOW flow_stats 確定書き込み（毎時:05）CRITICAL 10分（600秒） 5分超（300秒） 即座に確認
SUBPORTNORMAL 5分（300秒） 3分超（180秒） 即座に確認

11.2 非機能要件

要件説明
可用性 ETL 障害時の手動リカバリ手段を用意（import_log 削除 → ファイル移動 → 再実行）
耐久性 93日間のデータ保持（TimescaleDB 保持ポリシー）。それ以前は自動削除
スケーラビリティ 最大 600万 flow_id に対応。ホスト数増加時は並列処理ワーカー数で線形スケールすること
監視 すべてのログを Syslog メッセージとして送信可能なこと。WARNING・ERROR を分離
冪等性 同一ファイル再実行は安全（import_log チェック）。データは DELETE + INSERT で上書き

11.3 ログ出力要件

条件レベル
処理失敗 ERROR
FLOW バッチ受信処理 3分超 / FLOW flow_stats 確定 5分超 / SUBPORT 3分超 WARNING
FLOW バッチ受信処理 5分超 / FLOW flow_stats 確定 10分超 / SUBPORT 5分超（SLA 超過） WARNING
DB 接続失敗 ERROR
遅延ファイル検出（既存バケット） WARNING
正常完了 INFO

12システムのポイント

  • A-01
    FLOW ファイル 1,200ファイル/時間を5分 SLA で処理できること
    判定: 性能テスト実行、処理時間 ≦ 300秒を確認
  • A-02
    SUBPORT ファイル 100ファイル/時間を5分 SLA で処理できること
    判定: 性能テスト実行、処理時間 ≦ 300秒を確認
  • A-03
    import_log で冪等性確保（同一ファイル再実行が安全にスキップ）
    判定: 同一ファイルを2回実行し、2回目がスキップされることを確認
  • A-04
    flow_stats: 1時間バケット、93日保持
    判定: TimescaleDB ハイパーテーブル・保持ポリシー設定を DDL で確認
  • A-05
    subport_stats: 5分バケット、93日保持
    判定: TimescaleDB ハイパーテーブル・保持ポリシー設定を DDL で確認
  • A-06
    ログ出力: WARNING・ERROR を区別し、SLA 違反時に WARNING ログが出力される
    判定: SLA 超過条件を再現し、ログで WARNING を確認
  • A-07
    遅延ファイル検出: 既存バケットのファイルを隔離ディレクトリへ退避し WARNING ログを出力
    判定: 既存バケットのファイルを投入し、隔離ディレクトリへの退避と WARNING ログを確認
  • A-08
    カスタマーコントローラーが SELECT 文で flow_stats・subport_stats から統計情報を取得できること
    判定: SQL クライアントで flow_stats・subport_stats をクエリし、期待するデータが返ることを確認
