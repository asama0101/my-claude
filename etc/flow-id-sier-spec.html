01文書概要

1.1 目的

本書は、bwc-provisioner（flow_id プロビジョニング API サービス）の要求仕様を定義する。本仕様を満たすシステムを設計・実装すること。

1.2 適用範囲

  • flow_id バルク追加 API（POST /api/v1/flow-ids/bulk）
  • flow_id バルク削除 API（POST /api/v1/flow-ids/bulk/delete）
  • flow_id 一覧取得 API（GET /api/v1/flow-ids）
  • FITELnet FX2 連携仕様・スコープ解決ロジック
  • 期待値ストア管理・バックアップ世代管理
  • エラーハンドリング・非機能要件・システムのポイント

02用語定義

用語定義例
flow_id 回線番号・IPv6プレフィックス・subport を @ で連結したユーザー識別子。FX2 上の帯域制御の最小単位。 0123456789@2001:db8::/56@AB12-03-0456
subport 帯域制御ルール名。ISP コード・サービスコード・region_id の 3 要素をハイフンで連結した文字列。 AB12-03-0456
region_id ローカルエリア識別子（4 桁数字）。subport 名の末尾 4 桁と一致する。同一 region_id に属する FX2 に一括投入される。 0456
service_id subport 名から region_id（末尾4桁）を除いた部分。別名：帯域制御グループ。複数 region にまたがるサービス全体を指す際に使用する。 AB12-03（= AB12-03-0456 から -0456 を除いた部分）
address flow_id に紐づく IPv6 プレフィックス（CIDR 表記）。 2001:db8::/56
bwc-provisioner カスタマーコントローラーと FITELnet FX2 の間に位置するプロビジョニング API サービス。flow_id の追加・削除・一覧取得リクエストを受け付け、対象 FX2 へ並列投入する。 https://{host}/api/v1

03flow_id フォーマット定義

flow_id は以下の 3 要素を @ 区切りで連結した文字列です。

回線番号
0123456789
@
IPv6 プレフィックス
2001:db8::/56
@
subport（region_id を含む）
AB12-03-0456
末尾 4 桁 = region_id

フィールド仕様

フィールド名型追加時削除時説明
flow_id string 必須 必須 {回線番号}@{IPv6プレフィックス}@{subport} 形式の一意識別子
subport string 必須 不要 紐づく subport 名。FX2 への登録に使用する。省略時は空文字列で登録される。
address string 必須 不要 IPv6 プレフィックス（CIDR 表記）。FX2 への登録に使用する。省略時は空文字列で登録される。
注意： subport と address は API スキーマ上 optional ですが、FX2 正常動作のため追加操作では必ず指定してください。省略すると FX2 に空文字列が登録されます。

04subport フォーマット定義

subport は帯域制御ルール名です。ISP コード・サービスコード・地域コード（region_id）の 3 要素をハイフンで連結した文字列で、FX2 上の帯域制御グループを識別します。

ISP コード
AB12
英大2桁 + 数字2桁
-
サービスコード
03
数字2桁
-
地域コード（region_id）
0456
数字4桁

フィールド仕様

セグメントフォーマット説明例
第1セグメント XX00（英大文字2桁+数字2桁） ISP コード。ISP を識別する固定値。 AB12
第2セグメント 00（数字2桁） サービスコード。プランや契約種別を識別する。 03
第3セグメント 0000（数字4桁） 地域コード（= region_id）。この値から対象 FX2 装置を解決する。 0456
service_id との関係： subport から第3セグメント（-0456）を除いた部分が service_id（例: AB12-03）です。service_id は複数 region にまたがるサービス全体を指す際に使用します。

07flow_id バルク追加 Must

flow_id のリストを受け取り、各 flow_id に含まれる subport の region_id から対象 FX2 を自動解決して一括追加します。

POST https://{host}/api/v1/flow-ids/bulk
flow_id を一括追加する。同一 flow_id が既に存在する場合は上書き（重複登録は無視）。

リクエストボディ（JSON）

フィールド型必須説明
flow_ids array[FlowIdItem] 必須 追加する flow_id のリスト（1件以上）

FlowIdItem

フィールド型必須説明
flow_id string 必須 追加する flow_id（{回線番号}@{IPv6プレフィックス}@{subport}）
subport string | null 推奨 紐づく subport 名。FX2 登録に使用。省略時は空文字列。
address string | null 推奨 IPv6 プレフィックス（CIDR 表記）。FX2 登録に使用。省略時は空文字列。
Request Example
{
  "flow_ids": [
    {
      "flow_id":  "0123456789@2001:db8:1::/48@AB12-03-0456",
      "subport":  "AB12-03-0456",
      "address":  "2001:db8:1::/48"
    },
    {
      "flow_id":  "0987654321@2001:db8:2::/48@AB12-03-0456",
      "subport":  "AB12-03-0456",
      "address":  "2001:db8:2::/48"
    },
    {
      "flow_id":  "1111111111@2001:db8:3::/48@CD34-05-0789",
      "subport":  "CD34-05-0789",
      "address":  "2001:db8:3::/48"
    }
  ]
}

レスポンス（200 OK）

フィールド型説明
batch_id string (UUID) このバッチ操作の一意 ID
action string 常に "add"
total integer 処理対象 FX2 装置数
success integer 投入成功装置数
failed integer 投入失敗装置数（FX2 通信エラー等）
skipped integer スキップ装置数（メンテナンスモード中）
results array[DeviceResult] 装置ごとの処理結果

DeviceResult

フィールド型説明
hostname string 対象 FX2 装置のホスト名
status string "applied"（成功）/ "skipped"（メンテ中）/ "failed"（失敗）
reason string | null スキップ・失敗の理由。成功時は null。
Response Example
{
  "batch_id": "3fa85f64-5717-4562-b3fc-2c963f66afa6",
  "action":   "add",
  "total":    2,
  "success":  1,
  "failed":   0,
  "skipped":  1,
  "results": [
    {
      "hostname": "fx2-0456-01",
      "status":   "applied",
      "reason":   null
    },
    {
      "hostname": "fx2-0456-02",
      "status":   "skipped",
      "reason":   "maintenance"
    }
  ]
}

業務ロジック

  • 各 flow_id から subport（末尾 @ 以降）を抽出し、region_id（subport 末尾 4 桁）を取得する。
  • devices.yaml の region → FX2 マッピングを参照して対象装置リストを決定する（スコープ解決ロジックを参照）。
  • 同一 hostname への重複投入は 1 回にまとめる。
  • 装置が メンテナンスモード中の場合、FX2 への投入をスキップし、期待値ストアのみ更新する。
  • FX2 への投入成功後に期待値ストアを更新し、バックアップを保存する。
  • FX2 への投入失敗時は期待値ストアを変更しない。
  • 一部装置への投入が失敗しても、他の装置への投入は続行する（部分成功を許容）。
認証：必須 Content-Type：application/json 200：部分失敗含む全ケースで返却 400：バリデーションエラー 401：認証失敗 409：装置キュー満杯

08flow_id バルク削除 Must

flow_id のリストを受け取り、各 flow_id に含まれる subport の region_id から対象 FX2 を自動解決して一括削除します。

POST https://{host}/api/v1/flow-ids/bulk/delete
flow_id を一括削除する。存在しない flow_id は無視される（冪等）。

リクエストボディ（JSON）

フィールド型必須説明
flow_ids array[string] 必須 削除する flow_id の文字列リスト（1件以上）
削除リクエストは flow_id 文字列のみ。追加と異なり、subport / address の指定は不要です。
Request Example
{
  "flow_ids": [
    "0123456789@2001:db8:1::/48@AB12-03-0456",
    "0987654321@2001:db8:2::/48@AB12-03-0456",
    "1111111111@2001:db8:3::/48@CD34-05-0789"
  ]
}

レスポンス（200 OK）

バルク追加と同一スキーマ。"action" フィールドのみ "delete" になる。

Response Example
{
  "batch_id": "7b2c1f45-8a3e-4d2b-9c7f-1e8d4a5b6c0f",
  "action":   "delete",
  "total":    2,
  "success":  2,
  "failed":   0,
  "skipped":  0,
  "results": [
    {
      "hostname": "fx2-0456-01",
      "status":   "applied",
      "reason":   null
    },
    {
      "hostname": "fx2-0789-01",
      "status":   "applied",
      "reason":   null
    }
  ]
}

業務ロジック

  • 各 flow_id から region_id を抽出し、対象 FX2 を決定する（バルク追加と同一スコープ解決）。
  • 装置がメンテナンスモード中の場合、FX2 への削除投入をスキップし、期待値ストアのみ更新する。
  • FX2 に存在しない flow_id の削除は FX2 が無視するため、エラーにならない。
  • FX2 への投入失敗時は期待値ストアを変更しない。
  • 一部装置への投入が失敗しても、他装置への投入は続行する。
認証：必須 Content-Type：application/json 200：部分失敗含む全ケースで返却 400：バリデーションエラー 401：認証失敗 409：装置キュー満杯

09flow_id 一覧取得 Must

期待値ストアから flow_id の一覧を取得します。スコープは hostname・region_id・service_id の 3 種類から 1 つを指定します。

注意： このエンドポイントは期待値ストア（サーバー内部の状態ファイル）を参照します。FX2 への通信は発生しません。
GET https://{host}/api/v1/flow-ids
クエリパラメータでスコープを指定して flow_id 一覧を取得する。3つのうち必ずいずれか1つを指定すること。

クエリパラメータ（いずれか1つ必須）

パラメータ型説明例
hostname string 特定の FX2 装置の flow_id を取得 ?hostname=fx2-0456-01
region_id string 指定 region_id に属する全 FX2 の flow_id を統合して取得（重複除去済み） ?region_id=0456
service_id string 全 FX2 から指定 service_id に一致する flow_id を取得（全 region 横断） ?service_id=AB12-03
クエリパラメータを1つも指定しない場合、400 Bad Request が返ります。

フィールド型説明
hostname string 検索対象のホスト名
flow_ids array[FlowIdInfo] 該当装置の flow_id 一覧。未登録 or 不明 hostname の場合は空配列。
GET /api/v1/flow-ids?hostname=fx2-0456-01
{

    },
    {

    }
  ]
}

フィールド型説明
region_id string 検索対象の region_id

GET /api/v1/flow-ids?region_id=0456

10FX2 連携仕様（確認中）

bwc-provisioner が FITELnet FX2 の REST API を直接操作する際の仕様です。FX2 の API ベース URL は https://{fx2-host}/api/v1 です。

API 認証

bwc-provisioner は各 FX2 への全リクエストを HTTP Basic 認証で送信します。

Request Header（bwc-provisioner → FX2）
Authorization: Basic <base64(username:password)>
credentials.yaml フィールド型説明
username string FX2 Basic 認証のユーザー名（装置ごとに設定）
password string FX2 Basic 認証のパスワード（装置ごとに設定）

エンドポイント一覧

操作メソッドパスContent-Type
flow_id 追加 POST /api/v1/dynamic-queueing/flows application/jsonl
flow_id 削除 DELETE /api/v1/dynamic-queueing/flows application/jsonl
flow_id 取得 GET /api/v1/dynamic-queueing/flows —

JSONL フォーマット

POST / DELETE のリクエストボディは JSONL 形式（1行1 JSON オブジェクト、改行区切り）で送信します。

POST ボディ（追加）

1行1 flow_id（改行区切り）
{"flow_id": "0123456789@2001:db8:1::/48@AB12-03-0456", "subport": "AB12-03-0456", "address": "2001:db8:1::/48"}
{"flow_id": "0987654321@2001:db8:2::/48@AB12-03-0456", "subport": "AB12-03-0456", "address": "2001:db8:2::/48"}
{"flow_id": "1111111111@2001:db8:3::/48@AB12-03-0456", "subport": "AB12-03-0456", "address": "2001:db8:3::/48"}

DELETE ボディ（削除）

削除時は flow_id キーのみ指定します。

1行1 flow_id（flow_id キーのみ）
{"flow_id": "0123456789@2001:db8:1::/48@AB12-03-0456"}
{"flow_id": "0987654321@2001:db8:2::/48@AB12-03-0456"}

GET レスポンス

FX2 から返却される現在登録済みの flow_id 一覧（JSONL 形式）。行ごとに flow_id・subport・address を含む JSON オブジェクト。

FX2 API ステータスコードの扱い

FX2 レスポンスbwc-provisioner の扱い
200 OK 成功。期待値ストアを更新してバックアップを保存する。
200 以外 エラーとして扱い、最大 3 回リトライ後に失敗。期待値ストアは変更しない。
subport との違い： subport 操作では FX2 の 3-step commit プロトコル（PATCH → refresh → save）が必要ですが、flow_id 操作は POST / DELETE の 1 ステップで完結します。

11スコープ解決ロジック

バルク追加・削除では、リクエストに含まれる flow_id の文字列から対象 FX2 装置リストを自動的に決定します。呼び出し元は region_id や hostname を別途指定する必要はありません。

1
flow_id から subport を抽出
末尾 @ 以降を取得 → "0123456789@2001:db8::/56@AB12-03-0456" → AB12-03-0456
↓
2
subport から region_id を抽出
末尾 4 桁を取得 → AB12-03-0456 → 0456
↓
3
devices.yaml で region_id → FX2 リストを解決
region_id: "0456" → [fx2-0456-01, fx2-0456-02]
↓
4
対象装置リスト確定（重複除去済み）
複数の flow_id が同じ region_id を持つ場合、同一装置への投入は 1 回にまとめる。
必須設定ファイル：devices.yaml
ステップ 3 の解決には /app/config/devices.yaml が必要です。 devices セクションに FX2 装置一覧、regions セクションに region_id と所属 FX2 のマッピングを定義します。
devices.yaml — 構造例
devices:
  - hostname: fx2-0456-01
    ip:       192.168.1.1
    maintenance: false
  - hostname: fx2-0456-02
    ip:       192.168.1.2
    maintenance: false
  - hostname: fx2-0789-01
    ip:       192.168.2.1
    maintenance: false

regions:
  - id:   "0456"
    name: "東京"
    devices:
      - fx2-0456-01
      - fx2-0456-02
  - id:   "0789"
    name: "大阪"
    devices:
      - fx2-0789-01
フィールド説明
devices[].hostname FX2 装置のホスト名（一意識別子）。FX2 への接続ホスト名と一致させること。
devices[].ip FX2 の管理 IP アドレス。
devices[].maintenance メンテナンスフラグ（true 時は FX2 への投入をスキップし、期待値ストアのみ更新する）。
regions[].id region_id（4 桁数字）。subport 末尾 4 桁と一致させること。
regions[].name リージョンの表示名（任意文字列）。
regions[].devices この region_id に属する FX2 ホスト名のリスト。devices[].hostname と一致させること。

200 OK 正常完了（部分失敗含む）。results[].status で各装置の結果を確認すること。 —
400 Bad Request リクエストボディの検証エラー（必須フィールド欠落・型不正・空配列等）、または GET でクエリパラメータなし リクエスト内容を修正して再送
401 Unauthorized HTTP Basic 認証失敗（認証情報なし or 誤り） 正しい認証情報で再送
409 Conflict 対象装置の処理キューが満杯（同一装置への同時リクエスト超過） しばらく待ってから再送
422 Unprocessable Entity FastAPI の Pydantic バリデーションエラー リクエスト内容を修正して再送

13SLA・非機能要件

12.1 SLA 定義

処理SLA（最大）警告閾値備考
バルク追加・削除（1バッチ全体）CRITICAL FX2 台数 × 30 秒 FX2 台数 × 20 秒超 並列処理のため直列合計より大幅に短縮
呼び出し頻度NORMAL 1時間に 1 回（毎時 40 分） — カスタマーコントローラーからの呼び出しタイミング

12.2 非機能要件

要件説明
タイムアウト FX2 API 1 リクエストあたり 30 秒
リトライ FX2 API 失敗時に最大 3 回リトライ。全リトライ失敗時は当該装置を "failed" として記録し、他装置への投入は継続する
バックアップ世代 region_id 単位で最大 10 世代 FIFO ローテーション。11 世代目書き込み時に最古世代を自動削除
TLS FX2 への通信は HTTPS（自己署名証明書可）
冪等性 同一 flow_id の重複追加は FX2 が上書き（エラーなし）。削除時に存在しない flow_id は FX2 が無視（エラーなし）

条件レベル
FX2 への投入成功 INFO
FX2 リトライ発生（1〜3回目） WARNING
FX2 全リトライ失敗（装置単位の最終失敗） ERROR
メンテナンスモードによるスキップ INFO
devices.yaml 読み込み失敗 ERROR
バルク操作正常完了（部分失敗含む） INFO

14システムのポイント

  • A-01
    バルク追加: リクエストの flow_id が対象 FX2 装置すべてに登録されること
    判定: バルク追加後に GET /api/v1/flow-ids?hostname=fx2-XXXX で登録済みを確認
  • A-02
    バルク削除: リクエストの flow_id が対象 FX2 装置から削除されること
    判定: バルク削除後に GET /api/v1/flow-ids?hostname=fx2-XXXX で対象が消えていることを確認
  • A-03
    スコープ解決: flow_id の subport から region_id を抽出し、devices.yaml に基づいた正しい FX2 リストへ投入されること
    判定: 異なる region_id を持つ flow_id を混在させたバルク追加を実行し、各装置への投入結果を results[] で確認
  • A-04
    部分失敗耐性: 一部 FX2 装置への投入が失敗しても、他装置への投入が継続されること
    判定: 1台の FX2 を意図的に停止し、残り装置への投入が成功（200 OK + results[].status で確認）することを確認
  • A-05
    バックアップ世代管理: region_id 単位で最大 10 世代が FIFO で保持され、11 世代目で最古が削除されること
    判定: バルク追加を 11 回実行し、バックアップファイル数が 10 のままであることを確認
  • A-06
    冪等性: 同一 flow_id の重複追加がエラーにならず、削除時に存在しない flow_id が無視されること
    判定: 同一 flow_id を2回追加して 200 OK を確認。存在しない flow_id を削除して 200 OK を確認
