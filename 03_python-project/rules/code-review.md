# コードレビュー規約

PR レビュー・セルフレビュー時に参照する。

## レビュー観点

- **ロジックの正確性**: 正常系・異常系・境界値を正しく処理しているか
- **型安全性**: 型ヒントが適切で `mypy --strict` を通過するか
- **テスト網羅性**: 追加・変更したロジックに対応するテストがあるか
- **パフォーマンス**: N+1 クエリ・ループ内 DB アクセス・不必要な全件取得がないか
- **セキュリティ**: IDOR・インジェクション・機密情報漏洩がないか（`rules/security.md` 参照）
- **ドキュメント更新**: 公開 API の docstring・CLAUDE.md の機能一覧・docs/ が最新か

## レビューコメントの形式

種別を先頭に明記する。

| 種別 | 意味 | 対応 |
|---|---|---|
| `[blocking]` | マージ前に修正必須 | 必ず対応する |
| `[nit]` | 軽微な提案・好み | 対応は任意 |
| `[question]` | 動作・意図の確認 | 回答 or 対応する |

例: `[blocking] この関数は None を返す可能性があるが、呼び出し元で未処理です。`

## セルフレビューチェックリスト

マージ前に以下を確認する。

- [ ] `uv run ruff check src/` がパスする
- [ ] `uv run mypy src/` がパスする
- [ ] `uv run pytest tests/` がパスする
- [ ] 不要なコメント・デバッグ用 print が残っていない
- [ ] 命名規則（`rules/coding.md`）に従っている
- [ ] 公開 API に docstring がある
- [ ] DBスキーマを変更した場合、マイグレーションファイルが揃っている
- [ ] 外部入力のバリデーション・認可チェックが実装されている（`rules/security.md` 参照）

## マージ基準

- CI の全チェックがパスしていること
- `[blocking]` コメントが全て解消されていること
