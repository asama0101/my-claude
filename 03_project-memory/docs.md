# ドキュメント作成ルール

仕様検討ルール、ドキュメント配置、処理フロー設計、図示・記法ガイドライン、仕様レビュー観点をまとめる。
ドキュメント作成・設計レビュー時に参照する。

## ドキュメント形式

`docs/` 配下のドキュメントは原則 **HTML** とする。

- 例外: 外部仕様書（`docs/spec.md` 等）のみ Markdown とする。
- 理由: 外部仕様書は実装中に頻繁に参照・編集するため、ビルドを介さず直接読める形式の方が実用的。

`.claude/` 配下の Claude 向け参照ドキュメントは Markdown とする(本ガイドラインの対象外)。

## ドキュメント作成の前提

### 想定レビュアー

本プロジェクトのドキュメントは、以下のレビュアーへの提示を想定する。

| レビュアー | 想定するレビュー観点 |
|---|---|
| 自分自身(セルフレビュー) | 設計判断の振り返り、実装との整合 |
| 外部SIer | 自社システムとの I/F 整合性、責任分界点、運用引継ぎ |
| 第三者レビュアー(技術監査・契約上のレビュー等) | 設計の妥当性、セキュリティ、運用継続性 |

そのため、**社内の暗黙知・前提知識に依存せず、ドキュメント単体で内容が伝わる**ように記述する。

### ドキュメント単体での自己完結性

各ドキュメントの冒頭に以下を明記する。

- **対象読者**: 誰が読むことを想定したドキュメントか(例: 「外部SIerの開発担当者」「運用担当者(オンコール)」)
- **前提知識**: 読み手に求める知識(例: 「BGP・帯域制御の基礎知識」「Python・Docker の基本」)
- **本ドキュメントで伝えたいこと**: 1〜3行で要約
- **関連ドキュメントへのリンク**: 上位ドキュメント(要件・基本設計)、関連する詳細設計、運用手順

### 専門用語・略語の扱い

社内・装置ベンダー固有の用語は初出時に定義する。

- 略語は初出時に正式名称を併記（例: 「API（Application Programming Interface）」）
- ドメイン固有の概念は説明を付ける（例: 「〇〇（説明）」）
- 用語集を `docs/glossary.html` に集約し、各ドキュメントから参照する

### 設計判断のトレーサビリティ

要件 → 基本設計 → 詳細設計 → 実装 の各段階で、**前段への参照**を明示する。

- 基本設計の各コンポーネントは、要件定義のどの機能要件を満たすかを記載
- 詳細設計の各クラス・関数は、基本設計のどのコンポーネントに対応するかを記載
- 実装(コード)のクラス docstring に、対応する詳細設計の節番号を記載

これにより、外部レビュアーが「この実装は何の要件を満たすか」を追跡できる。

### 設計判断の根拠の明示

「何をするか(What)」だけでなく「なぜそうするか(Why)」を書く。特に以下は省略しない。

- 採用した技術・手法の理由
- 検討した代替案と却下理由(重要なものは ADR に分離)
- 既知の制約・トレードオフ

## 仕様検討ルール

- 仕様の検討・変更は docs/ 配下のドキュメントで議論し、合意後にコードへ反映
- 重要な設計判断は ADR(`docs/design/decisions/`)に記録(コンテキスト / 決定 / 理由 / 代替案 / 結果、500語以内)
- 機能追加・変更は対応する要件・設計ドキュメントの更新を同じPRに含める

ドキュメントの粒度:

- **要件定義**: What を書く(How は書かない)。機能要件は ID を振り、後段からトレース可能にする(例: `FR-001`)
- **基本設計**: コンポーネントと I/F、責任分界点(実装詳細は書かない)
- **詳細設計**: クラス・関数レベル、処理フロー(コード断片は最小限)

## プロジェクトドキュメント

各種ドキュメントは以下に配置する。

| ドキュメント | 配置先 | 役割 | 作成タイミング |
|---|---|---|---|
| 用語集 | docs/glossary.html | 専門用語・略語の定義 | 要件定義 |
| 装置仕様書 | docs/spec.md | REST APIなど装置の詳細仕様(例外的にMarkdown) | 基本設計 |
| 性能しきい値 | docs/performance_thresholds.html | Load/Stress Testの基準値 | 基本設計 |
| 処理フロー設計 | docs/design/flows/*.html | 機能ごとの処理フロー図(レビュー時の参照用) | 詳細設計 |
| テスト計画書 | docs/test_plan.html | テスト範囲・合格基準・スケジュール | 基本設計完了時 |
| Scenario Test手順書 | docs/scenario/*.html | Acceptance Test の手動手順 | 詳細設計〜実装 |
| 構築・デプロイ手順書 | docs/operations/setup.html | 初期構築・更新・ロールバック | 実装中盤 |
| 日常運用手順書 | docs/operations/daily.html | 日次確認・メンテモード・装置追加 | 実装完了直前 |
| 障害対応手順書 | docs/operations/incident.html | 症状別一次対応・エスカレーション | 実装完了直前 |
| バックアップ・リストア手順書 | docs/operations/backup.html | 取得・保管・復元・装置交換時の移行 | バックアップ機能実装時 |
| ADR | docs/design/decisions/*.html | 設計判断の記録 | 設計判断発生時 |

### 各ドキュメントの必須セクション

すべてのドキュメントに共通する必須項目:

- 対象読者・前提知識
- 本ドキュメントの目的(1〜3行の要約)
- 関連ドキュメントへのリンク
- 改訂履歴(日付・変更概要・変更者)

ドキュメント別の必須セクション:

- **要件定義書**: 背景・スコープ(対象範囲と対象外)・機能要件(ID付き)・非機能要件(定量化)・制約条件・用語定義
- **基本設計書**: システム構成・コンポーネント一覧と責務・I/F定義・責任分界点・要件とのトレース表
- **詳細設計書**: クラス・関数レベルの設計・処理フロー図・基本設計とのトレース表・設計判断の根拠
- **テスト計画書**: 目的・対象範囲・テストレベル一覧・合格基準・テスト環境・テスト観点・スケジュール・バグ管理フロー
- **構築・デプロイ手順書**: 前提条件・初期構築・起動停止・Smoke Test・通常デプロイ・ロールバック・設定変更時の対応
- **日常運用手順書**: 定期確認項目・ログの読み方・メンテナンスモード・装置追加撤去・認証情報ローテーション・緊急停止
- **障害対応手順書**: 障害対応フロー・影響範囲特定・緊急度判定・症状別一次対応・エスカレーション・復旧後確認・報告
- **バックアップ・リストア手順書**: 対象と取得方法・保管・リストア判断基準・リストア手順・装置交換時の移行・DR・定期検証
- **ADR**: コンテキスト・決定・理由・代替案・結果

### 運用マニュアル共通ルール

- 対象読者を冒頭に明記
- コマンドは実行可能な形(プレースホルダは `[xxx]`)
- 各章末尾にチェックリスト
- 重複は避け、相互リンクを使う

### ドキュメント作成・更新の指示

- 既存有無を確認し、あれば差分追記、なければ「必須セクション」を満たすテンプレートを作成
- ドキュメント間で同じ情報を重複記載しない(リンクを使う)
- 機能追加・変更PRには影響を受けるドキュメント更新を同梱
- 必須セクションが不要な場合は削除せず「N/A: 理由」と明記
- レビュー資料として通読を前提にせず、参照されることを前提に詳細を記載する
- 外部レビュアー向けに、社内固有の前提・略語を初出時に説明する

## 処理フロー設計

コードレビュー時、コードのみでは処理の全体像が把握しにくく、設計意図との整合性を確認できない。これを防ぐため、詳細設計フェーズで **処理フロー図** を作成し、レビュー時の参照とする。

### 作成単位(3階層)

機能単位で1ファイルを作り、ファイル内を節分けして異なる粒度の図を含める。

| 粒度 | 配置 | 内容 |
|---|---|---|
| **システム全体** | アーキテクチャ書(基本設計) | コンポーネント間のやりとり。1枚 |
| **機能単位の概略**(必須) | `docs/design/flows/[機能名].html` の §1 | 機能の入力・出力と主要分岐を1枚で俯瞰。レビュアーが最初に読む |
| **機能内のサブ処理単位**(複雑な処理に必須) | 同ファイルの §2 以降に節分けして複数 | リトライ・状態遷移・並列処理・エラー伝播など、特定の処理を詳細化 |

「機能内のサブ処理」を独立節に分けることで、概略図に情報を詰め込みすぎず、詳細を書きたい部分は別図として深掘りできる。

### 配置とファイル構成

- ファイル: `docs/design/flows/[機能名].html`(例: `subport_register.html`)
- 命名は機能単位の英小文字スネークケース
- 1ファイル内の構成例(subport登録の場合):

```
# subport登録の処理フロー

## 対象読者・前提知識・関連ドキュメント
## 1. 全体フロー(概略)
   ※ activity図1枚で機能の入出力と主要分岐を示す
## 2. 装置選定ロジック
   ※ region_id → 対象装置群の決定を詳細化
## 3. 並列送信処理
   ※ asyncio による全装置同時送信を sequence(par) で詳細化
## 4. リトライ処理
   ※ 失敗時の3回リトライと状態遷移を state図で詳細化
## 5. エラー時のSyslog通知
   ※ 全リトライ失敗時の通知経路
```

### 必須要素

各フロー図に以下を含める。

- **タイトル**: 何の処理か(例: 「subport登録の処理フロー」)
- **入力・出力**: 開始点と終了点
- **分岐条件**: if/else, try/except, リトライ, タイムアウト
- **外部呼び出し**: HTTP, docker run, ファイルI/O, ログ出力
- **エラーハンドリング**: 例外発生時の経路
- **凡例**: 記号・線種の意味
- **対応コード位置**: ファイル・関数・行番号(例: `src/launcher/main.py: handle_request()`)

各節(サブ処理単位の図)には以下も追加する。

- **この節で詳細化する範囲**: 全体フローのどの部分を深掘りしているか(例: 「§1 のステップ3〜5を詳細化」)

### 推奨する図の種類

| 内容 | 図の種類 | 主な使用箇所 |
|---|---|---|
| 単一機能の流れ(分岐含む) | Mermaid `flowchart` | §1 全体フロー |
| コンポーネント間のやりとり | Mermaid `sequenceDiagram` | サブ処理(I/F詳細) |
| 状態を持つ機能の遷移 | Mermaid `stateDiagram-v2` | サブ処理(リトライ・メンテモード等) |
| 並列処理(asyncio) | Mermaid `sequenceDiagram` の `par` 構文 | サブ処理(並列送信) |
| 複雑な分岐(スイムレーン) | Mermaid `flowchart` の `subgraph`、または draw.io | §1または複雑なサブ処理 |
| Mermaid で表現困難な複雑な構造 | draw.io(`.drawio.svg` 形式) | 例外的に使用 |

### コードとの整合性

- フロー図はコードと一緒に変更する(機能変更時、フロー図も同じPRで更新)
- フロー図と実装が乖離している場合、レビューで指摘する
- フロー図には対応コード位置を記載し、追跡可能にする

### Claudeへの指示

- 詳細設計フェーズで新規機能を実装する前に、`docs/design/flows/[機能名].html` に処理フロー図を作成する
- ファイル冒頭に対象読者・前提知識・関連ドキュメントを記載する
- §1 に機能単位の概略フロー図を必ず含める
- 機能内に複雑な処理(リトライ・並列・状態遷移など)があれば、§2 以降に節分けして詳細フロー図を含める
- 既存機能の変更時、対応する処理フロー図を同じPRで更新する
- レビュー時、フロー図と実装の不整合を発見したら指摘する
- 処理フロー図のないPRがレビューに回る場合、追加を提案する(機能単位の変更がない軽微な修正は除く)

## 図示・視覚化ガイドライン

ドキュメントの可読性向上のため、図示には **Mermaid を基準** とする。Mermaid で表現困難な場合のみ draw.io を使用する。

### 図示を推奨する場面

- 構造・関係性が文章では伝わりにくい
- 時系列・処理順序が重要
- 分岐・状態が複数ある
- スケジュール・期間を示す
- 複数の対象を比較・分類する

### Mermaid を基準にする理由

- GitHub・MkDocs Material・VSCode などが標準対応しており、**追加のビルド環境なしで図がレンダリングされる**
- PRレビュー時、GitHub の差分画面で図がそのまま見える
- 学習コストが低く、テキストでバージョン管理できる

### Mermaid と draw.io の使い分け

**基本は Mermaid を使う**。以下の図種をカバーできる。

- `flowchart`(処理フロー、構造図)
- `sequenceDiagram`(コンポーネント間やりとり、`par` で並列処理表現)
- `stateDiagram-v2`(状態遷移)
- `classDiagram`(クラス図)
- `erDiagram`(ER図)
- `gantt`(スケジュール)

**draw.io は以下の場合に限り使用**:

- Mermaid のスイムレーン表現(`subgraph`)では構造が伝わらない複雑なフロー
- 画面レイアウト・ネットワーク構成図など、自由配置が必要な図
- 既存の draw.io 資産を流用する場合

draw.io を使う場合は **`.drawio.svg` 形式で保存**する(編集可能なSVGとして HTML/Markdown に直接埋め込め、Git でバージョン管理できる)。

### 図作成時の注意

- 1つの図に詰め込みすぎない(15要素超なら分割)
- 日本語ラベル可
- 色を多用しない(モノクロでも判別できる構造)
- 図には **タイトル・凡例・読み取りポイント** を必ず添える
- Mermaid のコードブロックには言語指定 ` ```mermaid ` を必ず付ける
- draw.io ファイルはレポジトリに `.drawio.svg` で保存し、HTMLからは `<img>` または埋め込みSVGで参照する

## ドキュメント記法ガイドライン

ドキュメントはレビュー資料として詳細に記載する。通読されることは前提にせず、レビュアーが必要箇所を素早く参照できる構造を優先する。

記法は以下の目的で使い分ける。

- **構造化**: 表・箇条書き・見出しで情報を整理し、スキャンしやすくする
- **強調**: 注意ボックス(admonition)で見落としてはいけない情報を目立たせる
- **再現性**: コードブロック・コマンド例で実行可能な形式を残す

装飾は上記の目的に沿う場合のみ使用する。見栄え目的の装飾は禁止。

### 注意ボックス(admonition)

各章で特に強調したい箇所のみ使用(目安: 各章1〜2個まで)。

| 種別 | 用途 |
|---|---|
| `note` | 補足情報 |
| `tip` | 推奨事項・コツ |
| `warning` | 注意喚起 |
| `danger` | 重大な警告・不可逆操作 |

```markdown
!!! warning "タイトル"
    本文
```

### コードブロック

- 言語指定必須(`bash`, `python`, `yaml` 等)
- 20行超は行番号付与(`linenums="1"`)
- 重要行はハイライト(`hl_lines="3 5"`)
- ファイルパス明示時はタイトル付き(`title="src/main.py"`)

### 表

- 比較・属性一覧・パラメータ説明に積極使用
- 4列超 または 各セルが長文 になる場合は箇条書きに切り替え
- セル内改行は `<br>`
