# セッション終了改善フロー

軽量ワークフロー。**過剰な更新は行わない**。Step 1 で振り返り、寝かせる学びは
Step 2 の薄れないバックログへ積み、人間が選んだものだけ Step 3 で実行する。

---

## Step 1: 振り返り（サブエージェント中心）

**主軸: サブエージェント使用の徹底チェック**

以下の必須エージェントが、必要な場面で使われたか:

| エージェント | 必須場面 | 使用できたか |
|------------|---------|-----------|
| tdd-guide | 新機能・バグ修正 | ✅ / ❌ |
| reviewer-*（run-reviewers スキル） | コード変更後 | ✅ / ❌ |
| python-dev | Python 実装 | ✅ / ❌ |

追加チェック:
- 汎用 `claude` エージェントで代替した場面はなかったか
- 独立したタスクを並列起動できた場面を逐次実行していないか

**副軸: Skills の適切な使用**
- `brainstorming` → `writing-plans` → `subagent-driven-development` のフローが守られたか
- 省略されがちなステップ（spec 作成・タスクごとレビュー）が守られたか

**第3軸: 使った agent/skill の「質」評価（定義の改善要否）**

「使えたか」だけでなく、**今セッションで実際に使った** agent/skill それぞれの定義に改善余地がないか点検する:

- **実害**: 誤誘導した／チェックを漏らした／古い情報・存在しない参照を持っていた／期待した出力形式でなかった
- **先回り**: 今回はうまくいったが「この観点・チェックを定義に足せば次回さらに良くなる」と気づいた点

### Step 1 → Step 2/Step 3 への引継ぎ

❌（使用規律）と 実害/先回り（質評価）を列挙。`[実害]`/`[先回り]` の質評価項目には対象 agent/skill 名と汎用/固有の見立てを付す（`❌` 項目には不要）:

```
【Step 2/Step 3 への引継ぎ】
- ❌ <省略されたエージェント/スキル>: <根本原因> → <対策候補>
- [実害] <agent/skill名>: <事象（何を誤った/漏らした）> → <定義への対策候補>
- [先回り] <agent/skill名>: <足したい観点> → <汎用 or 固有 の見立て>
```

各引継ぎ項目に「今やる／寝かせる」の処置を付す:
- **今やる**（Step 3 の実害3条件／先回り3条件のいずれかを満たす見込みのもの）→ Step 3 で実行。
- **寝かせる**（条件を満たさない・判断保留・様子見・新規作成の種）→ Step 2 でバックログへ append。
  ＝「メモで寝かせて忘れる」をしない。薄れないバックログに必ず残す。

引継ぎ項目も、バックログ追記対象も無ければ Step 2 / Step 3 をスキップして Step 5 へ。

---

## Step 2: 改善バックログ（薄れない保留リスト）

保留中の改善を1か所に溜め、薄れさせず人間が選んで実行する閉ループ。
ファイル: `~/.claude/improvement-backlog.md`（グローバル1枚・sync 対象外）。
無ければ作成。3セクション（`## 新規作成候補` / `## 保留更新` / `## 未実行同期`）。

**(a) 追記**: Step 1 で「寝かせる」とした項目を、該当セクションへ1行 append。
重複排除キー＝同 type＋同対象（既存があれば追記しない。任意で末尾に `／再: YYYY-MM-DD`）。
行形式: `- [type] <一行で何を> — <理由/証拠> (初出: YYYY-MM-DD)`
type 例: `skill` / `agent:<name>` / `reference:<name>` / `sync` / `proj:<name>`。

**(b) 提示と選択**:
- バックログが空なら「保留なし」1行でこの提示・選択を飛ばす（Step 1 の「今やる」項目があれば Step 3 の更新は実行する）。
- 空でなければ **全件をコンパクトに列挙**（初出から60日超の項目は `(陳腐化?)` を表示だけ付す。昇格・削除には関与しない目印）。
  「今やる項目」を番号でユーザーに選ばせる。**未選択はそのまま残す**。

**(c) ルーティング**: 選ばれた項目を Step 3 の更新機構（反・増殖ラダー②〜⑥）へ渡す。

**(d) 完了＝行削除（閉ループの要）**: 反映できた項目はバックログから該当行を削除。
やり残しは残す＝次回また出る。ユーザーが「不要」と言えば即削除（自動プルーンはしない）。

## Step 3: 最小更新

Step 1 の引継ぎ（今やる項目）と Step 2 で選ばれたバックログ項目について、ゲートを満たすものだけ更新する。**過剰更新しない**。

### 反・増殖の昇格ラダー（最初に必ず通す）

学びは**軽い置き場から順（①→⑥）に検討**し、新しい入れ物（agent/skill 新設）は最終手段:

① ゲート不充足なら Step 2 のバックログへ送る（後退は一方向。本 STEP の実効選択肢は②〜⑥）→ ②対象 repo の CLAUDE.md/spec へ固有ルール追記 → ③既存 agent/skill 定義へ 1〜数行追記 → ④既存 `~/.claude/agents/references/*.md` へ追記 → ⑤新規 reference 作成 → ⑥新規 agent/skill 作成。

> 「単なる知識の塊」は ④⑤（reference）へ。⑥ は「reference では満たせない理由＝新トリガー・独立フローを 1 行で言える」時のみ。

### ゲート（満たすものだけ今すぐ更新。満たさない＝寝かせる項目は Step 2 のバックログへ）

- **実害**（❌・[実害] 項目）: ①実際に問題が起きた ②解決策が 1 行で言える ③次回も再現しうる ―― **3 つ全て**。
- **先回り**（[先回り] 項目）: 実害①が無いので歯止めとして ―― **汎用に有用・1〜数行で重複なく追記・肥大/汎用性低下を招かない** の全て。

### 更新の種類と閾値

| 種類 | 実施する条件 |
|------|------------|
| CLAUDE.md 修正/追記 | revise-claude-md→improver で実施（後述）。古い情報・誤記・フェーズ変化、または実害/先回りゲート充足分のみ |
| agent/skill 定義の更新 | 実害 **または** 先回りゲートを満たすもののみ |
| references/*.md 追記・新規 | 既存 agent が実行時に読む知識の塊。**新規 agent/skill より優先**（ラダー④⑤）。agent 定義側は参照行のみ |
| Hook 新規作成 | 「自動でなければ確実に忘れる」構造的問題のみ |
| agent/skill 新規作成（最終手段） | reference で満たせない=新トリガー・独立フローが必要な場合のみ（ラダー⑥・却下理由を 1 行で） |

### ルーティング（汎用 vs 固有）

- **汎用**（別プロジェクトでも有用）: 既存 agent が読む知識の塊 → `~/.claude/agents/references/<name>.md`（ラダー④⑤・agent 側は参照行のみ）。そうでなければ `~/.claude/agents/<name>.md` / `~/.claude/skills/<name>/SKILL.md` を更新。
- **固有**: 原則 対象 repo の CLAUDE.md/spec へ。例外（複数セッションで繰り返す完結ワークフロー）のみ `<project>/.claude/...` に新設。

> グローバル agent は汎用に保ち、固有ルールは混ぜない。先回りで「固有」と判明したものは固有ルーティングへ回す。

### 実装時の注意

**CLAUDE.md を変更する場合（ラダー② 固有ルール追記・CLAUDE.md 修正）は 2 段フロー（変更時必須）**:
1. `/claude-md-management:revise-claude-md` で学びを CLAUDE.md へ取り込む
2. `/claude-md-management:claude-md-improver` で更新結果をレビューする

（フォールバック: revise が使えない時のみ直接 Read→Edit。CLAUDE.md は現状直接 Edit 可＝`hard-block-workarounds` メモリ。）

agent/skill/reference 定義の更新は上記フロー対象外（revise/improver は CLAUDE.md 専用）→ 従来どおり提示→承認→直接 Edit/Write（HARD BLOCK はサブエージェントのみ）。

Hook を追加した場合は `settings.json` への登録と CLAUDE.md 「アクティブなHooks」テーブルへの記載も必須。

**ミラー同期**: グローバル定義（`~/.claude/agents|skills|hooks`、`agents/references/`）を直接編集すると設定管理 repo（my-claude）の `claude/` ミラーが古くなる。編集したら `未実行同期` を Step 2 のバックログに積む（記録だけで終わらせない）。`bash scripts/sync.sh` を実行できたらその行を削除。

ユーザーに変更案を提示し、承認を得てから実装する（テーブル形式・対象パスと汎用/固有を明記）:

| # | 種類 | 対象/箇所 | 汎用/固有 | 反映先パス | 解決する問題 | ゲート |
|---|------|---------|---------|----------|------------|------|
| 1 | CLAUDE.md修正 | アクティブなHooks | ― | （対象 repo）CLAUDE.md | 古い記述の修正 | ①②③ |
| 2 | agent定義更新 | reviewer-correctness | 汎用 | ~/.claude/agents/reviewer-correctness.md | 古い参照 | ①②③ |

### Reference / インベントリ棚卸（今セッションで agent/skill/reference を触った場合のみ）

増殖の対抗力。**今セッションで agent/skill/reference を作成・編集していなければスキップ**。触った場合のみ軽量に実施し、整理候補を提示する。

- **孤立 reference 検出**: 今回触れた `references/*.md` が、いずれかの agent 定義の参照行（`~/.claude/agents/references/<name>.md` を Read 等）から実際に参照されているか確認。被参照ゼロなら削除候補に。
- **重複・肥大**: 今回追記/新規した reference が既存と内容重複していないか、肥大していないか（本当に全部要るか）を点検。重複は 1 本へ統合候補。
- **未使用 agent / 重複 skill**: CLAUDE.md の表からの誘導が無い・目的が重複する agent/skill を、統合・廃止・reference 降格の候補として提示。
- **非破壊原則**: 候補を表で提示 → **ユーザー承認** → 整理。削除は CLAUDE.md「削除制限」に従い方法提示＋承認必須（グローバル定義の編集自体は本セッションで可・`hard-block-workarounds`）。整理後は `未実行同期` をバックログに積む。

## Step 4: spec/plan アーカイブ（docs/superpowers/ がある時のみ）

**スキップ条件**: `docs/superpowers/` が無ければ本 STEP 全体をスキップ。本 STEP は他 STEP と独立した housekeeping。

cwd に `docs/superpowers/specs/` も `docs/superpowers/plans/` も無ければ本ステップ全体をスキップ。ある場合のみ、完了状態を可視化し完全完了分を `done/` へアーカイブする。

1. **状態判定**: spec はファイル存在で ✅。plan はチェックボックス集計で進捗を出す:
   ```bash
   done=$(grep -cE '^[[:space:]]*- \[x\]' <plan.md>); todo=$(grep -cE '^[[:space:]]*- \[ \]' <plan.md>)
   # todo==0 かつ done>0 → ✅完全完了 / todo>0 → 🔄進行中（据え置き） / done==0 → 未着手
   ```
   完了/進行中を一覧表示する。

2. **アーカイブ（承認必須・非破壊）**: 100% 完了の plan を候補提示し、**承認を得てから** `git mv <file> docs/superpowers/plans/done/`。対応 spec も完了なら同様に確認のうえ移動。未完了ファイルは絶対に動かさない。
