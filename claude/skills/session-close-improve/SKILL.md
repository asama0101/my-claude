# セッション終了改善フロー

3ステップで完了する軽量ワークフロー。**過剰な更新は行わない**。

---

## Step 1: 振り返り（サブエージェント中心）

**主軸: サブエージェント使用の徹底チェック**

以下の必須エージェントが、必要な場面で使われたか:

| エージェント | 必須場面 | 使用できたか |
|------------|---------|-----------|
| tdd-guide | 新機能・バグ修正 | ✅ / ❌ |
| reviewer-*（run-reviewers スキル） | コード変更後 | ✅ / ❌ |
| python-dev | Python/FastAPI 実装 | ✅ / ❌ |
| api-designer | API 設計 | ✅ / ❌ |

追加チェック:
- 汎用 `claude` エージェントで代替した場面はなかったか
- 独立したタスクを並列起動できた場面を逐次実行していないか

**副軸: Skills の適切な使用**
- `brainstorming` → `writing-plans` → `subagent-driven-development` のフローが守られたか
- 省略されがちなステップ（spec 作成・タスクごとレビュー）が守られたか

**第3軸: 使った agent/skill の「質」評価（定義の改善要否）**

「使えたか」だけでなく、**今セッションで実際に使った** agent/skill それぞれの定義に改善余地がないか点検する:

- **実害**: 誤誘導した／チェックを漏らした／古い情報・存在しない参照を持っていた／期待した出力形式でなかった
- **先回り**: 今回はうまくいったが「この観点・チェックを定義に足せば次回さらに良くなる」と気づいた点

### Step 1 → Step 2 への引継ぎ

❌（使用規律）と 実害/先回り（質評価）を列挙。`[実害]`/`[先回り]` の質評価項目には対象 agent/skill 名と汎用/固有の見立てを付す（`❌` 項目には不要）:

```
【Step 2 への引継ぎ】
- ❌ <省略されたエージェント/スキル>: <根本原因> → <対策候補>
- [実害] <agent/skill名>: <事象（何を誤った/漏らした）> → <定義への対策候補>
- [先回り] <agent/skill名>: <足したい観点> → <汎用 or 固有 の見立て>
```

引継ぎ項目がなければ Step 2 をスキップして Step 3 へ（ただし **Step 1.5 は有無に関わらず必ず実施**する）。

---

## Step 1.5: superpowers spec/plan の完了確認とアーカイブ

superpowers の spec（brainstorming）/ plan（writing-plans）が「終わったのか」を後から確認できるよう、成果物ファイルベースで完了状態を可視化し、完全完了したものを `done/` へアーカイブする。

1. **対象の有無を確認**: cwd の `docs/superpowers/specs/` と `docs/superpowers/plans/` を確認。どちらも無ければ本ステップをスキップ（`done/` 配下のファイルは集計対象外）。

2. **状態を判定して一覧表示**:
   - **spec**: ファイルが存在すれば ✅（spec フェーズ完了）。
   - **plan**: チェックボックスを集計して実装進捗を出す。
     ```bash
     done=$(grep -cE '^[[:space:]]*- \[x\]' <plan.md>); todo=$(grep -cE '^[[:space:]]*- \[ \]' <plan.md>)
     # todo==0 かつ done>0 → ✅完全完了 / todo>0 → 🔄進行中 done/(done+todo) / done==0 → 未着手
     ```

   表示フォーマット:
   ```
   === superpowers spec/plan 状態 ===
   spec フェーズ : ✅ docs/superpowers/specs/2026-05-25-foo-design.md
   plan フェーズ : ✅ docs/superpowers/plans/2026-05-25-foo.md  実装 8/8 → done/ へ移動可
   plan フェーズ : 🔄 docs/superpowers/plans/2026-05-20-bar.md  実装 3/8（進行中・据え置き）
   =================================
   ```

3. **完了分のアーカイブ（承認必須）**: plan が 100%（todo==0 かつ done>0）のものを「アーカイブ候補」として提示し、**ユーザー承認を得てから** `git mv` で移動する:
   ```bash
   mkdir -p docs/superpowers/plans/done
   git mv docs/superpowers/plans/<file>.md docs/superpowers/plans/done/
   ```
   - 対応する spec も完了とみなせる場合は、対応関係（日付/トピックが異なりうるため自動マッチしない）を**ユーザーに確認**のうえ `mkdir -p docs/superpowers/specs/done && git mv` で移動する。
   - **非破壊原則**: 移動前に必ず承認を得る。未完了（チェックボックス未消化）のファイルは絶対に動かさない。

active フォルダ＝進行中、`done/`＝完了済み、という状態が後から一目で分かるようになる。

## Step 2: 最小更新

Step 1 の引継ぎ項目について、ゲートを満たすものだけ更新する。**過剰更新しない**。

**実害ケース（❌ 項目・[実害] 項目）= 以下の3条件を全て満たすものだけ**:

```
① このセッションで実際に問題が起きた
② 解決策が明確（1行以内で言える）
③ 次回セッションでも再現する可能性が高い
```

**先回りケース（[先回り] 項目）= ①が原理的に成立しないため専用の先回りゲート（過剰更新の歯止め）**:

```
P① 汎用的に有用（その agent/skill を使うどのセッションでも効く観点）
P② 1〜数行で定義に追記でき、既存内容と重複しない
P③ 追記しても肥大化・汎用性低下を招かない（プロジェクト固有をグローバルに混入させない）
```

いずれのゲートも満たさなければ → Step 3 のメモリに気づきとして残すだけ。

### 更新の種類と閾値

| 種類 | 実施する条件 |
|------|------------|
| CLAUDE.md 修正 | 古い情報・誤記・フェーズ変化のみ。新規追記は3条件を全て満たす場合のみ |
| Hook 新規作成 | 「自動でなければ確実に忘れる」構造的問題のみ |
| Skill 新規作成 | 「複数セッションで繰り返す複雑なワークフロー」のみ |
| agent/skill 定義の更新 | 実害ゲート(①②③) **または** 先回りゲート(P①P②P③) を満たすもののみ |

### agent/skill 定義更新のルーティング（汎用 vs 固有）

```
その改善は別プロジェクトでも同じく有用か？
  YES（汎用） → グローバル定義を更新
                ~/.claude/agents/<name>.md  または  ~/.claude/skills/<name>/SKILL.md
  NO（固有）  → 原則: 対象 repo の CLAUDE.md / spec に固有ルールを書く（agent が実行時 Read する設計）
                例外（「複数セッションで繰り返す完結した再利用ワークフロー」の時のみ）:
                  <project>/.claude/agents/<name>.md  または
                  <project>/.claude/skills/<name>/SKILL.md を新規作成/更新
```

> 既存方針（グローバル agent は汎用に保ち、固有ルールは CLAUDE.md/spec へ）を崩さないこと。
> 先回りで「これは固有だ」と判明したものは **グローバルに足さず固有ルーティングへ**回す。

### 実装時の注意

`~/.claude/CLAUDE.md` は現状メインセッションから直接 Read→Edit 可（`hard-block-workarounds` メモリの実測）。万一ブロックされた場合は `claude-md-management:claude-md-improver` スキル経由で更新する。

Hook を追加した場合は `settings.json` への登録と CLAUDE.md 「アクティブなHooks」テーブルへの記載も必須。

**グローバル agent/skill 定義の編集について**:
- メインセッションから Read→Edit/Write で可（HARD BLOCK はサブエージェントのみ。CLAUDE.md も現状は直接 Edit 可。`hard-block-workarounds` メモリ参照）。
- **グローバル定義（`~/.claude/agents|skills`）を直接編集すると、設定管理 repo（my-claude）の `claude/` ミラーが古くなる**。次回 my-claude セッションで `bash scripts/sync.sh` を走らせ同期する旨を Step 3 のメモリに残す。

ユーザーに変更案を提示し、承認を得てから実装する（テーブル形式・対象パスと汎用/固有を明記）:

| # | 種類 | 対象/箇所 | 汎用/固有 | 反映先パス | 解決する問題 | ゲート |
|---|------|---------|---------|----------|------------|------|
| 1 | CLAUDE.md修正 | アクティブなHooks | ― | （対象 repo）CLAUDE.md | 古い記述の修正 | ①②③ |
| 2 | agent定義更新 | reviewer-correctness | 汎用 | ~/.claude/agents/reviewer-correctness.md | 古い参照 | ①②③ |

## Step 3: クローズ

### メモリ保存

次回セッションのために保存が必要な状態変化があれば記録（最小限に）:

保存先: `~/.claude/projects/<project-sanitized-path>/memory/`

- プロジェクトのフェーズ変化・次のステップ
- 次セッションで注意すべき非自明な制約

### CLAUDE.md 軽量確認

Step 2 で変更した場合のみ、目視で確認:
- 追加した記述が既存内容と重複していないか
- 古い情報を誤って残していないか

`claude-md-management:claude-md-improver` 呼び出しは任意（大きな変更をした場合のみ推奨）。
