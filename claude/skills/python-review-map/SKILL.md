# Python レビュー地図生成

Python コードベースを初めて読む、または久しぶりに読む人間のレビュアーのために、「地図＋仕様対応表
＋ロジック可視化」を、インデックスページ＋モジュールページ群のHTMLレポート一式にまとめて生成する。
**このスキル自身はバグの有無を判定しない。** 判定は人間、または `review-correctness` のような
別スキル/エージェントの仕事。このスキルの役割は「見るべき場所を具体的に指し示し、理解を
助けること」に徹する。

**レビュー観点チェックリストは作らない（全面廃止）。** チェックリストは「判定を促す問いかけ」
であり、このスキルの「判定はしない」という役割と相容れない。判定・チェックが要る場合は
`review-correctness` など別のスキル/エージェントに委ねる（ユーザーから依頼された場合も、
このスキルでは作らずその旨を伝える）。

## いつ使うか

- 初めて読むモジュール/パッケージの構造を把握したいとき
- PR や `git diff` のレビュー準備をしたいとき
- 仕様書とコードの対応が取れているか確認したいとき
- 「地図」「マップ」「俯瞰」「構造を整理して」のような言葉が出てきたとき

## ワークフロー

### 1. スコープを決める

呼び出し時の文脈から、次のどちらかを判断する。曖昧なら質問する。

- **diff/PR スコープ**: `git diff`（未コミット差分、またはユーザー指定のベースブランチとの差分）で
  変更されたモジュールを起点にする。それらの直接の呼び出し元/呼び出し先（1 hop）を `grep -rn` で
  洗い出し、地図に含める。リポジトリ全体は対象にしない。
- **全体スコープ**: ユーザーが指定したディレクトリ/パッケージ配下を対象にする。指定が無ければ
  対象の候補（例: `src/` 配下のトップレベルパッケージ）を挙げてユーザーに確認する。

対象が大きすぎる場合（数十モジュール超）は、一度に全部やらず範囲を絞ることをユーザーに提案する。
黙って一部だけ処理して「完了」と報告しない — 絞った場合は必ずその旨をレポート冒頭に明記する。

### 2. 仕様書/要件を探す

`references/spec-correspondence.md` の手順に従って自動探索し、見つけた候補をユーザーに提示して
確認を取ってから対応表作成に進む。確認を飛ばさない（誤検出のリスクがあるため）。

### 3. コードを読む

対象モジュールを実際に Read する（推測で書かない）。各モジュールについて:

- モジュール全体の役割
- 主要な関数/クラスのシグネチャと docstring
- 非自明な関数の分岐・例外・戻り値パターン
- 呼び出し元/呼び出し先の関係（`grep -rn` で確認）。このスキルの核心的な価値は
  「開発者が自分の目でコードを追える可視化」にある。曖昧な言葉で済ませず、後述の
  `callgraph` ブロックに落とし込める形（`関数A[file:line] -> 関数B[file:line]`）で
  関係を控えておく

範囲が広い場合は、複数のサブエージェント（Explore 等）に分担させて並行調査してよい。その場合、
各エージェントには対象モジュールのパスと `references/module-map-format.md` の要点を渡し、
`file:line` 付きの事実を持ち帰らせる。

### 4. Markdown 中間生成物を書く（モジュールごと＋インデックス）

**1つの巨大な Markdown/HTML にまとめない。** モジュールが2つ以上あるスコープでは、
モジュールごとに1つの Markdown ファイル＋インデックス用に1つの Markdown ファイル、
合計「モジュール数＋1」のファイルを書く（`references/module-map-format.md` の
「0. 出力構成」節を参照）。長い1ページに全部詰め込むと区切りが分からず読みづらくなるため。
モジュールが1つしか無いスコープでも、この構成（インデックス1つ＋モジュールページ1つ）で
統一する。

作業ディレクトリ（プロジェクト直下や `/tmp` 配下など、書き込み可能な場所）に、同じ
サブディレクトリ名の下にまとめて書く（例: `review-map-transform/index.md`,
`review-map-transform/common.md`, `review-map-transform/rules.md`,
`review-map-transform/subscribers.md`）。

- **インデックスページ**: 対象範囲の説明（スコープ・絞り込みがあればその理由）、全体構成図
  （`callgraph` ブロック、`<details open>` で開閉可能に）、モジュール一覧（各モジュールページ
  への相対リンク＋1行要約）、仕様書との対応表（`references/spec-correspondence.md` のテンプレ、
  対象があれば）を含める。レビューチェックリストは含めない（全面廃止）。
- **モジュールページ**（モジュールごとに1つ）: そのモジュールのモジュールブロック・関数ブロック・
  制御フロー可視化・難所解説・変数一覧（`references/module-map-format.md` の該当節のテンプレ）を
  含める。インデックスへ戻るリンクを先頭に置く。

見出しレベルは `references/module-map-format.md` の各節で指定された通りにする（TOCに拾われる
`##`/`###` を、ページ内の主要な区切りに正しく使うことが「区切りが分かる」ために重要）。

書き終えたら、使用した全ての `file:line` を `grep -n` で実ソースと突合する。一致しないものは
修正するか、確認できなかった旨を明記する。これを飛ばさない — このスキルの成果物の信頼性は
ここにかかっている。

### 5. HTML化

Markdown ファイルの数だけ `build_report.py` を実行する（インデックス1回＋モジュールごとに1回）。
出力先ディレクトリ名・ファイル名は入力の Markdown と対応させ、同じディレクトリに並べる
（相対リンクで行き来できるようにするため）。

```bash
python3 <このスキルのディレクトリ>/scripts/build_report.py \
  --input <Markdown中間生成物のパス> \
  --output <出力先HTMLのパス> \
  --title "<ページのタイトル。インデックスなら対象範囲名、モジュールページならモジュールのパス>" \
  --target "<対象範囲の説明。例: src/shaper_db/transform/ (diffスコープ)>"
```

出力先は対象プロジェクトのルート直下 `docs/codemap/<対象を表す短い語>/`（例:
`docs/codemap/review-map-transform/index.html` ＋同ディレクトリに `common.html`・
`rules.html`・`subscribers.html`）にする。ディレクトリが無ければ作成する。この地図は
使い捨てのレビュー成果物であり、`doc-updater` が管理する永続的なドキュメント（README・ガイド等）
とは別物 — 単に置き場所をプロジェクト直下から `docs/` 配下に集約しただけで、性質は
変わらない。プロジェクトルートに `.gitignore` があり `docs/codemap/` がまだ無視対象になって
いなければ、`docs/codemap/` の1行を追記する（使い捨て成果物が誤ってコミットされるのを防ぐ
ため）。追記した場合は最終報告でその旨を一言添える。Artifact 化はしない（対象コードの内容を
外部ホストへ公開しないため、常にローカルファイルとして保存する）。

### 6. 報告する

生成した **インデックス** HTML の保存先パスを一文で伝える（そこから各モジュールページへ辿れる
ため、これが入口になる）。レポート内容の要約（モジュール数・対応表の件数など）を簡潔に添えて
よいが、レポート本文の中身をここで繰り返し出力しない（レポートを見れば分かるため）。

## 参照ファイル

- `references/module-map-format.md` — モジュール/関数ブロック・難所解説・制御フロー可視化のテンプレ定義
- `references/spec-correspondence.md` — 仕様書の自動探索手順とユーザー確認フロー
- `scripts/build_report.py` — Markdown→HTML変換スクリプト（標準ライブラリのみ、`--help` で使い方確認可）
- `assets/report_template.html` — 上記スクリプトが差し込む自己完結HTMLテンプレート（直接編集する用途ではない）
