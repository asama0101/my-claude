# モジュール地図フォーマット

Markdown 中間生成物のうち「地図」パートを書くときのテンプレート集。対象は Python プロジェクトの
モジュール群。読者はそのコードを初めて読む、または久しぶりに読む人間のレビュアー。目的は「理解を
助けること」であり、正誤判定やバグ指摘はしない（判定が要る部分は `review-correctness` など
別のスキル/エージェントに委ねる。このスキルはレビューチェックリストを作らない）。

この地図は使い捨てのレビュー成果物であり、doc-updater が管理する永続的なドキュメント（README・ガイド等）
とは別物。ただし読みやすさの型は共通するため、以下のテンプレはその実績パターンを踏襲する。

## 大原則

- **コードから生成する。手書きで想像しない。** 各モジュール・関数は実際に Read してから書く。
- **`file:line` は実ソースと一致させる。** これが最大のリスク。書いたら `grep -n` で該当行を確認する
  （例: `grep -n "def sync_rules" src/shaper_db/sync/rules.py`）。一致しない参照を残さない。
- **本文は日本語、シンボル名・コマンド・パスは英語のまま。**
- **比例原則**: 自明なモジュール/関数（薄いラッパー・単純な getter・定型的な dataclass 等）は
  「平たく言うと」の1行で済ませてよい。難所解説・制御フロー可視化は非自明な箇所にだけ使う。
  すべての関数に同じ厚みのテンプレを機械的に当てはめない。
- **網羅性**: スコープ内のモジュールは漏れなく触れる（1行だけでも可）。省略した場合は
  「対象外: このモジュールは今回のスコープ外」のように明記し、黙って落とさない。
- **コードには必ず説明を添える**: 通常のコードブロック（```python 等）は、直前に最低1行、
  日本語で「何のコードか/何を示しているか」を説明する段落を置く。素のコードだけを貼って
  終わりにしない。このスキルの読者は初めてそのコードを読む人間であり、コードの意味を
  自分で読み解く前提を置かない。**説明を直前に、間に見出し等を挟まず置くと、HTML化の際に
  自動的に横並び（説明が左、コードが右）で表示される** — コードと解説を離して見づらくしない
  ための挙動なので、この並び順（説明→コードブロック）を守る。

## 0. 出力構成（インデックス＋モジュールページ）

**1ページに全モジュールを詰め込まない。** モジュール数が2つ以上のスコープでは、モジュールごとに
1つの Markdown/HTML（モジュールページ）＋全体を束ねる1つの Markdown/HTML（インデックスページ）
を作る。モジュールが1つしか無いスコープでも、同じ構成（インデックス1つ＋モジュールページ1つ）
で統一する。ファイルは全て同じディレクトリに置き、相対リンク（`[表示名](ファイル名.html)`）で
行き来できるようにする（このスキルの Markdown→HTML 変換は通常の Markdown リンク記法をそのまま
`<a href="...">` にするので、特別な書き方は不要）。

- **インデックスページ**（`index.md`）: 対象範囲の説明、全体構成図（「1. 全体構成図」）、
  モジュール一覧（各モジュールページへのリンク＋1行要約）、仕様書対応表を置く（レビュー
  チェックリストは全面廃止のため置かない）。複数モジュールを横断する内容はここにまとめ、
  モジュールページには重複させない。
- **モジュールページ**（モジュール名.md、例: `rules.md`）: そのモジュール1つ分の「2. モジュール
  ブロック」「3. 関数ブロック」「4. 制御フローの可視化」「5. 難所解説」「7. 変数一覧」を置く。
  ページ冒頭にインデックスへ戻るリンク（`[← インデックスへ戻る](index.html)`）を置く。

見出しレベルはページの主要な区切りとして機能させる（`##`/`###` は自動的にページ内目次に
拾われる）。モジュールページは1モジュールしか扱わないので、モジュール名そのものをページ見出し
として繰り返さず、`##` はページ内のセクション（概要・関数・変数一覧など）に、`###` は関数ごとに
使う（詳細は「2. モジュールブロック」「3. 関数ブロック」参照）。これによりページごとの内容が
コンパクトになり、区切りが視覚的にはっきりする。

## 1. 全体構成図（インデックスページの冒頭に1つ）

対象スコープ全体のアーキテクチャを `callgraph` ブロックで示す。依存の向き（矢印）が分かればよい。
このスキルの価値は「開発者が自分の目で処理の流れを追える」ことなので、静的な ASCII 図ではなく
HTML化時にパン/ズーム/クリックで追えるインタラクティブな図として描画される `callgraph` を使う
（文法は下記「6. callgraph ブロックの文法」参照）。ノードはモジュールページへのリンクにしたい
場合、`callgraph` のノード自体はリンクにできないので、図の直後に「モジュール一覧」として
Markdown リンクの箇条書きを添える（下記参照）。

全体構成図は表示/非表示を切り替えられるよう、`<details>`/`<summary>` で包む（変数一覧と同じ
固定書式。開いた状態をデフォルトにするため `<details open>` にする）。

````markdown
<details open>
<summary>全体構成図(クリックで表示/非表示)</summary>

```callgraph
cli/[src/shaper_db/cli.py] -> sync/[src/shaper_db/sync/]
sync/[src/shaper_db/sync/] -> transform/[src/shaper_db/transform/]
sync/[src/shaper_db/sync/] -> fx2/[src/shaper_db/fx2/]
```

</details>

## モジュール一覧

- [`common.py`](common.html) — 平たく言うと: 識別子/subport/flow_id の検証とSkipReason集約。
- [`rules.py`](rules.html) — 平たく言うと: ルール名生成・検証。
- [`subscribers.py`](subscribers.html) — 平たく言うと: ロンゲストマッチによる加入者ID生成・検証。
````

diff スコープの場合は、変更されたモジュールのノード id に `(変更)` を付け（例: `sync/rules.py(変更)`）、
その直接の呼び出し元/呼び出し先（1 hop）だけをブロックに含める。全体を網羅しようとしない
（1 hop 先のモジュールは、フルページを作らずインデックスの説明文中で軽く触れる程度でよい）。

## 2. モジュールブロック（モジュールページごとに1つ、`##` 見出し）

```markdown
[← インデックスへ戻る](index.html)

# `path/to/module.py`

## 概要

**平たく言うと:** 1〜2文でこのモジュールの役割。

**なぜ要るか:** 設計上の存在理由。何を1箇所に集約しているか、何を防いでいるか。

**呼ばれ方:** 実際の呼び出し元を `file:line` で。複数あれば主要なものを列挙。

（下記「7. 変数一覧」のテンプレ（`## 変数一覧` 見出し込み）をこの後に続ける）
```

`# path/to/module.py` は `build_report.py --title` に渡すタイトルと重複するので省略してもよい
（省略する場合は `## 概要` から始める）。自明なモジュール（例: 定数定義だけのファイル）は
「平たく言うと」の1行だけで次のセクションへ進んでよい（その場合「変数一覧」も省略してよい）。

## 3. 関数ブロック（非自明な関数のみ、`###` 見出し）

分岐・例外処理・ループ・外部呼び出しなど「読み手が誤読しうる」関数にのみ適用する。単純な
delegator・getter・dataclass のフィールド定義には使わない。`## 関数` という見出しの下に、
対象関数ごとに `###` を1つずつ続ける。

````markdown
## 関数

### `関数名(引数) -> 戻り値型` — `file:line`

**意図:** docstring または対応する仕様書の記述から読み取れる「本来やるべきこと」。
仕様書 ID があれば `(REQ-xx)` のように添える。docstring も仕様書もない場合は
「(docstring/仕様書の記述なし。呼び出され方から推測: …)」と明記する。

**呼び出し元:** `grep -rn` で確認した実際の呼び出し元を `file:line` で全て列挙する
（非公開のヘルパ関数（`_` 始まり等）や、同じモジュール内に複数の公開エントリポイントが
ある場合に特に重要 — 読み手は「同じような関数が並んでいるから共通ヘルパも両方から呼ばれて
いるはず」と誤って一般化しやすい。呼び出し元が1つだけなら「〜からのみ呼ばれる」と明記し、
複数の公開関数が並列に存在するモジュールでは特に省略しない）。

**実装の要点:** 実装から読み取れる分岐・例外・戻り値パターンを箇条書きで。
- 条件Aのとき: 〜する
- 条件Bのとき: 〜を送出する (`SomeError`)
- それ以外: 〜を返す

**コード抜粋:** 上記の要点に対応する実際のコード。関数が短ければ全体を、長ければ非自明な
箇所（分岐・ループ・例外処理の部分）だけを引用する（シグネチャや自明な代入行の羅列まで
律儀に全部貼らない）。実装の要点の箇条書きだけで十分説明できていそうでも省略しない —
このスキルの読者は実際のコードを見て自分の目で確認したいレビュアーである。

```python
def foo(...):
    ...
```

**確認ポイント:** 意図と実装の要点を読み比べて、レビュアーが目を留めるべき箇所があれば
一言添える。判定はしない（「ズレている」と断定しない）。ズレが無いと判断できるなら
このセクション自体を省略してよい。
````

コードブロックの直前には、必ず「コード抜粋:」のような一言の説明を置く（大原則の
「コードには必ず説明を添える」を参照）。HTML化されると、説明文とコードブロックは
横に並べず、説明→コードの順で全幅のまま縦に並ぶ（関数全体を貼ることもあるため、
横並びにすると片方の欄が間延びしたり、コードが横スクロール必須になり読みづらいため）。

## 4. 制御フローの可視化（非自明な分岐・ループを持つ関数のみ）

過度に複雑な関数（多分岐・多重ループ・複数の例外パス）にだけ使う。`callgraph` ブロックで、
関数内の分岐ステップをノードとして表す。分岐条件やエラーの種類はエッジラベル（`: ラベル`）に書く。
終端ノード（早期 return・例外送出など、file:line を持たないステップ）は `[...]` を付けず
`id` だけにする — 描画時に破線の終端ノードとして区別される。

`callgraph` ブロックは、それを説明する「実装の要点」の直後に、間に見出しや他のブロックを
挟まず連続して置く（図と対応する説明を離さないため）。横並びレイアウトにはしない — `callgraph`
自体がインタラクティブな図として十分な情報を持つので、幅の狭いカラムに押し込めず全幅で描画する。

```callgraph
現在状態取得[sync/rules.py:45] -> 差分判定[sync/rules.py:52]
差分判定[sync/rules.py:52] -> 早期return: 差分なし
差分判定[sync/rules.py:52] -> PATCH送信[fx2/client.py:120]: 差分あり
PATCH送信[fx2/client.py:120] -> リトライ[fx2/client.py:140]: 失敗
リトライ[fx2/client.py:140] -> 例外送出: 上限超過
PATCH送信[fx2/client.py:120] -> refresh_save[fx2/client.py:160]: 成功
```

実装の分岐条件と実際のコードが一致しているか `grep`/Read で確認してから書く。比例原則は
ここでも有効 — 単純な if-else 1つ程度なら「実装の要点」の箇条書きで十分であり、無理に
`callgraph` 化しない。

## 5. 難所解説（非自明な実装トリックがある場合のみ）

読み手が「なぜこんな書き方をしているのか」で詰まりそうな箇所（並行処理・ロック・リトライ・
シークレット処理・低レベル API の使用など）に使う。モジュールページに `## 難所解説` という
セクションを作り、該当箇所ごとに `###` を続ける。

```markdown
## 難所解説

### 難所: 〜（一言で何が難しいか）

**何が問題か:** 素朴に書くと何が壊れるか。

**どう解決しているか:** 対処方針。必要なら簡易図を添える。

**コード抜粋:** `file:line` 付きで実ソースから短く引用（長大な引用はしない。要点のみ）。

**ハマりどころ:** レビュアーがここを見落とすとどう困るか。
```

## 6. callgraph ブロックの文法

1行1エッジ。両端は `id[file:line]`（file:line が無い依存グラフ等は `id[path]` でも可）か、
終端は `id` のみ（file:lineを省略、破線ノードとして描画される）。ラベルは省略可。

```
<左ノード> -> <右ノード>[: <エッジラベル>]
```

- ノード id は空白を含んでよいが `[` `]` `->` は使えない。同じ id が複数行に出てきたら
  同一ノードとして自動的にまとめられる（ref は最初に出てきた非空のものが使われる。終端として
  先に出現しても、後で `[file:line]` 付きの出現があればそちらが採用される）。
- 1つの `callgraph` ブロックに書けるのは1つの図。複数の独立した図が要る場合はブロックを分ける。
- HTML化されると、パン・ズーム・ノードクリックでのハイライト（クリックしたノードの
  呼び出し元/呼び出し先を強調し、対応する見出しがあれば自動スクロール）・簡易フィルタ入力が
  使える単一のインタラクティブな SVG 図になる。外部ライブラリ・外部通信は使わない（このスキルの
  「自己完結HTMLレポート」の原則を維持するため）。
- 表示/非表示を切り替えたい `callgraph`（全体構成図など）は `<details>`/`<details open>`/
  `<summary>...</summary>`/`</details>` の固定書式（「7. 変数一覧」参照）で包んでよい。
  デフォルトで開いた状態にしたい場合は `<details open>` を使う。
- ノード数が多くなりすぎる場合（数十を超える）は、図を分割するか、比例原則に従って対象を
  絞ることをユーザーに確認する。読み取れないほど密な図を1つ作って終わりにしない。

## 7. 変数一覧（モジュールごとに1つ）

対象モジュールで使われている変数を、モジュールレベル（定数・設定値・グローバル）から関数内の
ローカル変数まで網羅的に一覧化する。自明なループカウンタ（`i`・`idx` 等）も省略しない —
表形式なら一覧化のコストは低く、このスキルの核心的価値である「可視化の正確性」を優先する。

一覧は長くなりがちなので、`<details>`/`<summary>` で折りたたんで埋め込む（クリックで展開）。
この2行3タグはこのスキル専用の固定書式であり、この通りに書く（他のHTML断片を書き足さない）。

```markdown
## 変数一覧

<details>
<summary>変数一覧(クリックで展開)</summary>

| 変数名 | スコープ | 型/値 | 役割 | file:line |
|---|---|---|---|---|
| MAX_RETRY | モジュール定数 | int = 3 | PATCH失敗時のリトライ上限 | fx2/client.py:12 |
| current_state | `sync_rules()` 内ローカル | dict | 装置から取得した現在状態 | sync/rules.py:46 |
| i | `sync_rules()` 内ローカル | int | ルール一覧のループカウンタ | sync/rules.py:58 |

</details>
```

- 「スコープ」列: モジュールレベルなら「モジュール定数」「モジュールグローバル」、関数内なら
  `` `関数名()` 内ローカル `` のように、どの関数に属するかを明示する。
- 「型/値」列: 型ヒントがあればそれを、無ければ代入されている値から型を読み取って書く。
- 変数名が同じでもスコープが違えば別の行として扱う（同じモジュール内で `i` が複数の関数に
  出てくる場合など）。

## 書き終えたら

- 使った `file:line` を全部リストアップし、`grep -n` で1つずつ実ソースと突合する。
- 「対象外」と明記したモジュール以外、スコープ内の全モジュールに触れているか確認する。
- インデックスページの「モジュール一覧」のリンク先ファイル名と、実際に生成した各モジュール
  ページのHTMLファイル名が一致しているか確認する（`build_report.py --output` の出力先ファイル名と
  Markdown内のリンクは別々に手で書くため、ずれやすい）。各モジュールページ冒頭の「← インデックスへ
  戻る」リンクも同様に確認する。
- 完成した Markdown 中間生成物を見直し、通常のコードブロック（```python 等）が1つでもあれば、
  その直前または直後に説明が実際に残っているか確認する。編集の過程で説明だけ削れて素の
  コードブロックだけが残る、あるいはコードブロック自体が意図せず全部消えることがある —
  「コードには必ず説明を添える」という大原則が最終稿でも守られているかを最後に必ず見直す。
