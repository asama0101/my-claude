# ゲート定義（gates.md）

10 の品質ゲート。オーケストレータ（SKILL.md）が順に駆動し、各ゲートを担当ロールへ委任する。
採点は `scoring.md` 準拠。**PASS したゲートだけが次に進める。**

3 ロール分離（自己承認の構造的排除）:
- **Planner** = `planner` エージェント
- **Generator** = `tdd-generator` エージェント（テストと実装を書く）
- **Evaluator** = `tdd-evaluator` エージェント（採点する）

**分離原則（正典）**: Generator と Evaluator は必ず別のサブエージェントとして起動する。実装した本人が採点してはならない。

---

## テスト種別の自動判定（言語プロファイル依存）

編集/対象ファイルのパスから、書くべきテスト種別を判定し、**最後にユーザーへ確認**する（誤判定防止）。
具体的なパスパターンとテストランナーは言語プロファイル（`profiles/pytest.md` 等）に定義。判定結果（unit / integration / e2e）に応じて Gate7(UI/UX) の要否も決まる。

---

## 各ゲート

### Gate 1: コンテキスト分析・要件整理（Planner 起草 → Evaluator 採点）
- **目的**: 依頼＋既存コード・ドメイン・制約を分析し、**要件を抽出・構造化して抜け漏れなく確定**する（受入基準の言語化・テスト設計は Gate2）。
- **内容**: コンテキスト（依頼意図・既存実装・制約・関連システム）の把握、要件の抽出と構造化、**変更影響範囲の整理**、スコープの単一性確認。small ルート適用可否の判定材料（差分規模見込み・公開IF変更有無）を `planner` が申告。
- **手段**: `planner` が起草し、`tdd-evaluator` が rubric で採点（Generator 起動前なので自己承認リスクは無いが、Main の独断で次へ進めないよう採点主体を明示）。**small ルート4条件の照合・簡略可否判定はこの Gate1 で行う**（スコープ判定のため。SKILL.md「段階導入の限界」を Gate1 に紐付け）。
  - **要件網羅の敵対的検査**: `tdd-evaluator` は要件集合の完全性を敵対的に点検し、**「抜け漏れ候補リスト（未記載の要件・暗黙要件・未対応エッジ）」を必ず出力**する（"書かれた要件の明確さ" とは別軸で "書かれていない要件" を能動的に探す）。
  - **抜け漏れは Stop & Ask**: 要件の真実源はユーザーにあるため、非自明な抜け漏れ候補が出たら**自動 FAIL せずユーザーに確認**する（グローバル CLAUDE.md「Stop & Ask」準拠）。確認結果を要件に反映して再採点。
- **Gate1 と Gate3 の違い（重複でない）**: Gate1 のコンテキスト分析は「要件を抽出・確定するための把握」。Gate3 の事前レビューは「テストを書く前に既存コードのセキュリティ・DB・設計リスクを監査する」。目的が異なるため両立する。
- **Critical**: ①コンテキスト把握が的確（既存実装・制約を踏まえている）。②**要件の網羅性**——抜け漏れ候補リストが提示され、非自明な欠落はユーザー確認で解消済み。③スコープが単一。
- **採点項目**: コンテキスト把握の的確さ / 要件抽出・構造化 / 要件の網羅性（抜け漏れ・暗黙要件）/ スコープの単一性 / small ルート判定根拠の妥当性。
- **証拠**: 整理済み要件リスト＋抜け漏れ候補リスト（＋ユーザー確認の結果）＋スコープ申告（`git diff --stat` 見込み・既存テスト一覧）＋参照した既存コードの `file:line`。

### Gate 2: 受入基準・テスト設計（Planner 起草 → Evaluator 採点）
- **目的**: Gate1 で確定した要件を、**検証可能な受入基準に言語化し、テストシナリオへ落とし込む**。
- **内容**: 各要件を検証可能な受入基準へ言語化、**想定不具合パターンの列挙**をテストシナリオ（正常・異常・境界）へ、**3層のどの層で守るか判定**（下記「テスト3層戦略」）、対象ファイルとテスト種別の対応付け、**E2E 要否の明示判断**、レビュー方針。
- **手段**: `planner` が起草し、`tdd-evaluator` が rubric で採点。
- **Critical**: 受入基準が検証可能な粒度で、**要件↔受入基準↔テストシナリオの対応に漏れがない**（各要件に最低1受入基準・各受入基準に最低1シナリオ＝トレーサビリティ）。
- **採点項目**: 受入基準の明確さ・検証可能性 / シナリオ網羅（正常・異常・境界）/ テスト種別の妥当性・3層割り当て / 要件↔受入基準↔シナリオのトレーサビリティ。
- **証拠**: 対応表（要件ID ↔ 受入基準 ↔ テストシナリオ ↔ 対象ファイル ↔ テスト種別/層）。

#### テスト3層戦略（Gate2 の層割り当て方針）

| 層 | 守るもの | 方針 |
|----|---------|------|
| unit | 業務ロジック | 業務ロジックを I/O・FW・UI から**分離**して高速・多数（分離の検査は Gate3） |
| integration | 機能品質 | **主軸**。機能の振る舞いを結合レベルで担保 |
| e2e | 品質の最後の砦 | **最小限**。重要導線のみ。むやみに増やさない |

Gate2 は各シナリオを「どの層で守るか」割り当て、E2E は要否を明示判断して最小化する。層別の実行コマンドは言語プロファイル（`profiles/*.md`）に定義。

### Gate 3: 事前レビュー（Evaluator）
- **内容**: テストを書く前に、既存コード・DB・セキュリティ（**権限漏れ含む**）・設計をレビュー。**業務ロジックが unit テスト可能な形に分離されているか**を検査し、未分離なら分割を提案する（分離は Gate2 の unit 層戦略の前提）。**ファイル肥大は `review-maintainability` の基準に従って**検出し分割を提案（数値はそちらを単一ソースとする）。
- **手段**: `tdd-evaluator` が計画と既存コードを直接レビュー・採点する（**既定は evaluator 単独・reviewer 起動なし**——テスト・差分が未作成の段階では多次元 reviewer の検査材料が乏しく、実差分への多次元フルレビューは Gate8 で行うため）。計画がセキュリティ敏感（認証・入力処理・機密データ）または性能敏感（大量データ・ホットパス）な場合のみ、該当する `review-security`／`review-performance` を条件付きで並列起動し（最大2本）、起動構成を台帳に記録する。
- **Critical**: 重大セキュリティ欠陥・設計破綻がない。
- **採点項目**: 既存構造の把握 / セキュリティ観点（権限漏れ含む）/ DB・I/O 観点 / **業務ロジックの分離（unit テスト可能性）** / ファイルサイズ・責務分割。
- **証拠**: レビュー所見（review-* の出力）。

### Gate 4: RED（Generator）
- **内容**: 期待する振る舞いに対して**失敗するテストを先に書く**。実装コードは書かない。
- **Critical（即FAIL）**:
  - **tdd-evaluator が自ら再実行**し、テストが実際に失敗する（プロファイルの失敗ログ形式に一致）。「おそらく失敗する」は 0 点。
  - **assert が対象の振る舞いを具体的に検証している**。`assert False`/`assert True`/例外 raise だけ/トートロジー/`assert ..., "TODO"` 等の無条件失敗は Critical 未達（意味のない RED は不可）。
  - **evaluator が自ら取得した `git diff` で、変更がテストファイルのみ**（実装ファイルの変更が無い）であること（実装先行→RED 後付けの排除）。
- **採点項目**: 失敗ログの提示 / assert の具体性 / エッジケース網羅 / テスト独立性。
- **証拠**: テストランナーの失敗ログ。

### Gate 5: GREEN（Generator）
- **内容**: テストを通す**最小限の実装**を書く。過剰実装（テストにない機能）をしない。
- **Critical（即FAIL）**:
  - **tdd-evaluator が自ら再実行**し、テストが通過する（対象 `passed` かつ全体で**ベースライン比の新規 `failed` 0**——台帳のベースライン記録に無い failed が 0。ベースラインが全緑なら従来どおり `failed` 0）。
  - **RED(Gate4) 時に台帳へ固定したテストと同一で、assert を弱めていない**（照合して差分があれば Critical 未達。台帳に記録済みの仕様変更による固定記録の更新は除く）。
  - 最小実装である（evaluator 取得の `git diff` にテスト非対応の実装 branch を作り込んでいない）。
- **採点項目**: 通過ログの提示 / 最小性 / 既存テストの緑維持 / エラーパス考慮。
- **証拠**: テストランナーの通過ログ（全既存テスト緑を含む）。

### Gate 6: REFACTOR（Generator）
- **内容**: 重複除去・命名改善・整理。**振る舞いは変えない**。
- **Critical（即FAIL）**:
  - **新機能・新しい振る舞いを追加していない**。evaluator が `git diff` を取得し、**テストファイルとテスト設定ファイル（`conftest.py`・`pytest.ini`・`pyproject.toml` の test 設定等、プロファイル定義のテスト構成に関わるファイル）が不変**（skip/xfail・deselect 等のテスト実行除外の追加を含め変更なし）かつ**既存テストが到達しない新規 branch/機能が加わっていない**ことを確認する（差分カバレッジで裏取り推奨）。
  - リファクタ後もプロファイル定義のテスト実行コマンドで全緑（Python プロファイルなら `pytest -q`）。
- **採点項目**: 重複除去 / 命名・可読性 / テスト不変の確認 / 緑維持ログ。
- **証拠**: `git diff` ＋ リファクタ後の通過ログ。

### Gate 7: UI/UX レビュー（frontend-design＋敵対的クロスレビュー・**E2E 時のみ・条件付き**）
- **スキップ条件**: テスト種別が e2e（ブラウザ）でなければ**ゲートごとスキップ**。ただし**テンプレート/ルーティング/ビュー層に変更がある場合は unit 申告でも Gate7 を必須化**（自己申告での回避を防ぐ）。スキップ理由は台帳に明記する。
- **内容**: `frontend-design` スキル＋敵対的クロスレビューで UI/UX を検査（AI 臭さ排除・レイアウト・導線）。最大 3 ラウンド（確認→修正）。ブラウザ不可環境は静的解析で代替。
- **Critical**: なし（非 E2E はスキップ）。
- **証拠**: スクショ / 静的解析結果。

### Gate 8: 差し戻し判定（Evaluator）
- **内容**: 変更差分全体を多次元で採点し、マージ可否を判定する最終ゲート。
- **手段**: オーケストレータが `review-*` 5 次元（correctness/security/performance/test/maintainability）を**並列起動**し、`tdd-evaluator` が 1 枚のスコアカードに集約＋Critical 判定。
- **Critical（即FAIL）**: 仕様不適合 / 既存回帰 / **偽装テスト検出**（assert なし・常に真・実装の写経。検出は目視に加えミューテーション検証を実施——徴候があれば必須・無くても代表1テストにスモーク。実施義務・トリガー・安全手順の正典は tdd-evaluator 定義）。
- **採点項目**: 5 次元それぞれ（正確性 / セキュリティ / 性能 / テスト品質 / 保守性）。
- **証拠**: 集約スコアカード（review-* 所見に裏付け）。

### Gate 9: CI 品質ゲート整備（Evaluator・条件付き・プロファイル駆動）
- **スキップ条件**: CI を運用しないプロジェクトは、台帳にスキップ理由を記して**ゲートごとスキップ**。
- **内容**: プロジェクトの CI が下記ステージを自動実行するよう整備・検証する。**必須ステージ（抽象・言語非依存）**: `lint` / `typecheck` / `build` / `unit test` / `integration test` / `主要E2E`。**任意**: `preview 環境デプロイ`（web 系のみ・既定 off）。
- **手段**: 各ステージの**具体コマンドは言語プロファイル（`profiles/*.md`）が供給**（グローバル skill にプロジェクト固有を書かない）。CI プロバイダは **GitHub Actions を既定**とし profile で差し替え可。ワークフロー実体（`.github/workflows/quality-gates.yml` 等）は**各プロジェクトに生成**する（未整備なら profile のコマンドから scaffold＝オーケストレータが `doc-updater` に委任、既存なら必須ステージの被覆と当該変更の新規テスト取り込みを検証）。`tdd-evaluator` が必須ステージの被覆を採点する。
- **同期性の限界**: 「CI がグリーン」は push 後の非同期事象。本ゲートは **CI 定義の存在・被覆をゲート**し、green-on-push は**マージ後の enforcement 前提**として台帳に記録する（agent ループ内で green を待たない）。
- **Critical**: 必須ステージのいずれかが CI 定義から欠落している（preview は任意なので対象外）。
- **採点項目**: 必須ステージの被覆 / 当該変更の新規テストが CI 対象に含まれる / プロファイル駆動（プロジェクト固有コマンドがグローバル skill に漏れていない）。
- **証拠**: 生成/検証したワークフロー定義の該当行＋被覆したステージ一覧。

### Gate 10: ドキュメント同期（条件付き・執筆/検証ペア）
- **スキップ条件**: リファクタのみ等、ドキュメント影響が無ければスキップ。
- **内容**: `doc-updater` で README・ガイド・コードマップ・API 仕様を変更に同期。AI 臭い定型表現を検出・修正。
- **手段（執筆・検証の分離）**: `doc-updater` が執筆・更新 → Main が**別コンテキストで `doc-verifier` を起動**し、記載事実（数値・コマンド・スキーマ・挙動）をソースと照合させる（doc-updater 定義の委譲規則と同型＝執筆者の自己承認排除）。不一致は doc-updater へ差し戻して修正させ、**再検証では前回不一致を1件ずつ 解消 / 部分解消 / 未解消 で判定**する（`scoring.md` の逐項目3値判定と同型）。
- **不発明制約（委任指示の定型文）**: doc-updater への委任プロンプトには「**ソースに根拠の無い数値・目標値・事実を発明しない。導出できる事実のみ記載し、根拠が無い項目は『未規定』と明示する**」を必ず含める。doc-verifier には「文書に新規登場した数値・事実の裏取り（発明された値の検出）」を照合観点として指示する。
- **Critical**: doc-verifier の照合で、裏付けの無い事実主張（実装に無い挙動・根拠の無い数値の記載）が未解消のまま残っている。
- **証拠**: 更新差分＋doc-verifier の照合結果（✓/不一致の表）。

## 文書・設計成果物の敵対レビューパターン（オプション）

要件定義書・設計文書・計画書など**文書・設計成果物**の品質を問うとき（Gate1 の要件整理レビュー・Gate10 の大規模 doc 整備・TDD 文脈外の文書レビュー）は、次元別チェックリスト（コード用の review-* 5次元）の代わりに**命題駆動の敵対レビュー**を使ってよい。**コード差分の Gate8 はこのパターンで置き換えない**（review-* 5次元のチェックリストが確立済みのため現行維持）。

- **相補的な逆方向2命題**: 「過剰の立証」と「欠落の立証」の対になる命題を立て、それぞれ独立エージェントに検察官スタンスで立証させる（例: 「この文書は要件定義書を名乗る設計書である」×「要件定義書として必須の要素を欠いている」）。過剰系と欠落系で挟むと、次元分割では出ない被覆が得られる。
- **正例・負例ペアのルーブリック埋め込み**: 各命題のプロンプトに具体例で判定基準を与える（例: 「flush→fsync→rename の手順は設計／書きかけが読者に見えないことは要件」）。あわせて**境界事例は安易に断ぜず理由付きで判定させ、「弁護側に譲った事例」も報告させる**（`agents/references/review.md` の過剰指摘抑制と同旨）。
- **同一レビュアーによる再判定**: 修正後の再レビューは新規エージェントでなく**同一レビュアーに（コンテキスト保持のまま）再判定を依頼**し、前回指摘の逐項目3値判定（解消 / 部分解消 / 未解消、`scoring.md` と同型）＋修正による新規混入の検査をさせる。所見の回帰テストに相当し、新規レビューより収束が速く確実。
- **規範検証と事実検証の分離並列**: 「あるべき姿か」（敵対レビュー）と「記載事実が実体と一致するか」（`doc-verifier` の照合）は検出領域が重ならないため、**別エージェントで分離し並列**に走らせる。

## ゲート順序と分岐

フロー図の正典は SKILL.md「ゲート駆動ループ」（重複掲載しない）。分岐ルール:

- 各ゲートは PASS で次へ。CONDITIONAL は同ゲートを再評価（最大 2 回、`scoring.md`）。
- FAIL かつ再評価上限到達なら、ユーザーへ差し戻して停止。
- Gate4→5→6 は同一 Generator が担うが、**採点は毎回 Evaluator（別コンテキスト）が自力再実行して**行う。
- **Gate8 PASS 後、Gate9(CI) の前にユーザーの受け入れ確認**を取る（グローバル CLAUDE.md 準拠）。この受け入れ確認の際、`tdd-evaluator` が **受け入れチェックリスト**を生成する——受入基準（Gate2）＋品質観点（Gate8 の5次元）＋観点（例外処理・権限漏れ・変更影響範囲）から機械的に導出し、ユーザーの手動確認を**再現可能**にする。台帳に保存。
