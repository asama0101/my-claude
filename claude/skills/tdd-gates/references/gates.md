# ゲート定義（gates.md）

9 つの品質ゲート。オーケストレータ（SKILL.md）が順に駆動し、各ゲートを担当ロールへ委任する。
採点は `scoring.md` 準拠。**PASS したゲートだけが次に進める。**

3 ロール分離（自己承認の構造的排除）:
- **Planner** = `planner` エージェント
- **Generator** = `gate-generator` エージェント（テストと実装を書く）
- **Evaluator** = `gate-evaluator` エージェント（採点する。Generator とは必ず別コンテキスト）

Generator と Evaluator は**必ず別のサブエージェント**として起動する。実装した本人が採点してはならない。

---

## テスト種別の自動判定（言語プロファイル依存）

編集/対象ファイルのパスから、書くべきテスト種別を判定し、**最後にユーザーへ確認**する（誤判定防止）。
具体的なパスパターンとテストランナーは言語プロファイル（`profiles/pytest.md` 等）に定義。判定結果（unit / integration / e2e）に応じて Gate7(UI/UX) の要否も決まる。

---

## 各ゲート

### Gate 1–2: 計画（Planner 起草 → Evaluator 採点）
- **内容**: 受入基準の言語化、テストシナリオ設計、対象ファイルとテスト種別の対応付け、レビュー方針。
- **手段**: `planner` が計画を起草し、`gate-evaluator` が rubric で採点して PASS/CONDITIONAL/FAIL を判定（Generator 起動前なので自己承認リスクは無いが、Main の独断で次へ進めないよう採点主体を明示）。
- **Critical**: 受入基準とテスト対象の対応に漏れがない。
- **採点項目**: 受入基準の明確さ / シナリオ網羅（正常・異常・境界）/ テスト種別の妥当性 / スコープの単一性。
- **証拠**: 計画書（テスト対象一覧・シナリオ表）。

### Gate 3: 事前レビュー（Evaluator）
- **内容**: テストを書く前に、既存コード・DB・セキュリティ・設計をレビュー。**ファイル肥大は `reviewer-maintainability` の基準に従って**検出し分割を提案（数値はそちらを単一ソースとする）。
- **手段**: オーケストレータが `reviewer-*` を**並列起動**し、`gate-evaluator` が所見を集約・採点。**Gate3 のデフォルトセットは correctness / security / performance / maintainability の4本**（テスト未作成のため `reviewer-test` は含めず、Gate8 でのみ使う）。
- **Critical**: 重大セキュリティ欠陥・設計破綻がない。
- **採点項目**: 既存構造の把握 / セキュリティ観点 / DB・I/O 観点 / ファイルサイズ・責務分割。
- **証拠**: レビュー所見（reviewer-* の出力）。

### Gate 4: RED（Generator）
- **内容**: 期待する振る舞いに対して**失敗するテストを先に書く**。実装コードは書かない。
- **Critical（即FAIL）**: **gate-evaluator が自ら再実行して**テストが実際に失敗する（プロファイルの失敗ログ形式に一致）。「おそらく失敗する」は 0 点。**assert が対象の振る舞いを具体的に検証している** —— `assert False`/`assert True`/例外 raise だけ/トートロジー/`assert ..., "TODO"` 等の無条件失敗は Critical 未達（意味のない RED は不可）。
- **採点項目**: 失敗ログの提示 / assert の具体性 / エッジケース網羅 / テスト独立性。
- **証拠**: テストランナーの失敗ログ。

### Gate 5: GREEN（Generator）
- **内容**: テストを通す**最小限の実装**を書く。過剰実装（テストにない機能）をしない。
- **Critical（即FAIL）**: **gate-evaluator が自ら再実行して**テストが通過（対象 `passed` かつ全体 `failed` 0）。**RED(Gate4) 時に台帳へ固定したテストと同一で、assert を弱めていない**（照合して差分があれば Critical 未達）。最小実装である（evaluator 取得の `git diff` にテスト非対応の実装 branch を作り込んでいない）。
- **採点項目**: 通過ログの提示 / 最小性 / 既存テストの緑維持 / エラーパス考慮。
- **証拠**: テストランナーの通過ログ（全既存テスト緑を含む）。

### Gate 6: REFACTOR（Generator）
- **内容**: 重複除去・命名改善・整理。**振る舞いは変えない**。
- **Critical（即FAIL）**: **新機能・新しい振る舞いを追加していない**。evaluator が `git diff` を取得し、**テストファイルが不変**かつ**既存テストが到達しない新規 branch/機能が加わっていない**ことを確認（加わっていれば新機能追加＝Critical 未達。差分カバレッジで裏取り推奨）。かつ `pytest -q` 全緑。
- **採点項目**: 重複除去 / 命名・可読性 / テスト不変の確認 / 緑維持ログ。
- **証拠**: `git diff` ＋ リファクタ後の通過ログ。

### Gate 7: UI/UX レビュー（Evaluator・**E2E 時のみ・条件付き**）
- **スキップ条件**: テスト種別が e2e（ブラウザ）でなければ**ゲートごとスキップ**。ただし**テンプレート/ルーティング/ビュー層に変更がある場合は unit 申告でも Gate7 を必須化**（自己申告での回避を防ぐ）。スキップ理由は台帳に明記する。
- **内容**: `frontend-design` スキル＋敵対的クロスレビューで UI/UX を検査（AI 臭さ排除・レイアウト・導線）。最大 3 ラウンド（確認→修正）。ブラウザ不可環境は静的解析で代替。
- **Critical**: なし（非 E2E はスキップ）。
- **証拠**: スクショ / 静的解析結果。

### Gate 8: 差し戻し判定（Evaluator）
- **内容**: 変更差分全体を多次元で採点し、マージ可否を判定する最終ゲート。
- **手段**: オーケストレータが `reviewer-*` 5 次元（correctness/security/performance/test/maintainability）を**並列起動**し、`gate-evaluator` が 1 枚のスコアカードに集約＋Critical 判定。
- **Critical（即FAIL）**: 仕様不適合 / 既存回帰 / **偽装テスト検出**（assert なし・常に真・実装の写経）。
- **採点項目**: 5 次元それぞれ（正確性 / セキュリティ / 性能 / テスト品質 / 保守性）。
- **証拠**: 集約スコアカード（reviewer-* 所見に裏付け）。

### Gate 9: ドキュメント同期（条件付き）
- **スキップ条件**: リファクタのみ等、ドキュメント影響が無ければスキップ。
- **内容**: `doc-updater` で README・ガイド・コードマップ・API 仕様を変更に同期。AI 臭い定型表現を検出・修正。
- **証拠**: 更新差分。

---

## ゲート順序と分岐

```
Gate1-2(計画) → Gate3(事前レビュー) → Gate4(RED) → Gate5(GREEN) → Gate6(REFACTOR)
  → [e2e/view変更なら Gate7(UI/UX)] → Gate8(採点判定) → 受け入れ確認 → [doc影響あれば Gate9(同期)] → 完了
```

- 各ゲートは PASS で次へ。CONDITIONAL は同ゲートを再評価（最大 2 回、`scoring.md`）。
- FAIL かつ再評価上限到達なら、ユーザーへ差し戻して停止。
- Gate4→5→6 は同一 Generator が担うが、**採点は毎回 Evaluator（別コンテキスト）が自力再実行して**行う。
- **Gate8 PASS 後、Gate9 の前にユーザーの受け入れ確認**を取る（グローバル CLAUDE.md 準拠）。
