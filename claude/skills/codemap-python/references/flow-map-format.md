# フロー地図フォーマット

Markdown 中間生成物の書き方の正典。対象は Python プロジェクトの処理フロー群。読者はそのコードを
初めて読む、または久しぶりに読む人間のレビュアー。

**基準読者（この文書全体の合格ライン）**: プログラミング未経験に近いネットワークエンジニア。
Tera Term マクロ(TTL)や 20 行程度の Bash スクリプトを書いた経験はあり、**装置・CLI コマンド・
コンフィグ・show 系コマンド・SSH/Telnet・SFTP・cron・ログ・冗長構成(act/sby)といった運用の
言葉は日常語として通じる**。一方、**プログラミングと Web の言葉（関数・引数・戻り値・例外・
プロセス・終了コード・JSON・REST API・ハッシュ・データ構造など）は日常語ではない**。
`for`/`if`/`def`/`class`/`import`/`try-except` のような基礎文法にも、`dataclass` やクロージャの
ような Python 特有のイディオムにも不慣れである。この読者が誰の助けも借りずに通読できることを
合格ラインにする。ただしこの想定読者の引き下げは「line-notes の解説欄を毎回厚くする」方向では
実現しない — 解説欄は今まで通り簡潔な一言のままにし、基礎文法・イディオム・専門用語の説明は
「9. 用語集ブロックとインライン参照」で定義する用語集ページへの `[[用語]]` リンクに逃がす
（出現頻度の高い基礎文法を毎回 `<details>` で厚く説明すると、大原則の「簡潔性」「一本道原則」を
壊すため）。目的は「理解を助けること」であり、正誤判定やバグ指摘はしない（判定が要る部分は
`review-correctness` など別のスキル/エージェントに委ねる。このスキルはレビューチェックリストを
作らない）。

この地図は使い捨てのレビュー成果物であり、doc-updater が管理する永続的なドキュメント（README・
ガイド等）とは別物。

## 大原則

- **一本道原則（このフォーマットの存在理由）**: 読者がリンクを辿らずスクロールだけで処理全体を
  追えることを最優先する。ページはモジュール（ファイル）単位ではなく**処理フロー（エントリ
  ポイントから終わりまでの実行の流れ）単位**で作り、呼び出し先のコードはモジュール境界を越えて
  呼び出し箇所のその場に展開する。ページ間リンクはインデックス→フローページの1段だけにする。
  モジュール軸の分割は「1つの処理を追うのに5ページ以上を行き来させる」失敗構成だった —
  ファイル構造はコードの置き場所の都合であって、理解の順序ではない。
- **コードから生成する。手書きで想像しない。** 各フローが通る関数は実際に Read してから書く。
- **`file:line` は実ソースと一致させる。** これが最大のリスク。書いたら `grep -n` で該当行を
  確認する。一致しない参照を残さない。
- **本文は日本語、シンボル名・コマンド・パスは英語のまま。**
- **完全展開原則**: 呼び出し先の関数は、薄いラッパー・単純な getter・定型的な dataclass 等を
  含め、**省略せず全てコールフレームとして展開しソースコードを引用する**（旧版にあった
  「自明な呼び出し先は1行要約で済ませてよい」という比例原則は廃止した）。巨大な検証サブ
  システム（数十個のバリデータ群等）も例外ではない — 個々のバリデータを閉じたコールフレーム
  （脇道）に入れることで読みやすさは保つが、**閉じることと省略することは別**であり、開けば
  必ずソースが読める状態にする。難所解説・制御フロー可視化（callgraph による分岐図）は
  引き続き非自明な箇所にだけ使う（これらはコードの省略とは別の、補助的な可視化）。
- **簡潔性**: 説明文は長い段落でだらだら続けず、短文・箇条書きを優先する。**3文を超える説明は
  箇条書きや表に分解できないか検討する**（「実装の要点」「難所解説」「7a のなぜこう書くか」等、
  プレーンな文章で書きがちな箇所ほど、項目化できないか見直す）。
- **網羅性**: スコープ内の全エントリポイント（＝全フロー）をカバーする。どのフローにも登場しない
  コード（未使用ヘルパ・テスト専用関数等）はインデックスページに「どのフローにも登場しない
  コード」として一覧を明記し、黙って落とさない。
- **コードは1行ごとに解説を添える**: コードを引用するときは後述の `line-notes` フェンスを使い、
  各行の右側にその行の解説と具体例を書く。素のコードブロックだけを貼って終わりにしない。
- **運用のことばで先に説明する**: 説明文（Step 見出し・「この段階でやること」・「実装の要点」・
  難所解説・概要）は、次の2文構成を基本形にする — **1文目は「装置・ファイル・ネットワークに
  何が起きるか（運用者が手作業でやるなら何に相当するか）」を基準読者の日常語で書き、2文目
  以降で「コードがそれをどう実現しているか」を実装の言葉で続ける**。実装の言葉（同期・差分・
  オーケストレーション・クロージャ・写像など）だけで説明を完結させない。
  - 悪い例（実装のことばだけ）: 「同期オーケストレーション開始と外部DB取得」
  - 良い例: 「ルール表の元データをDBサーバから取ってくる（手作業なら DB サーバに SSH して
    psql コマンドを打ち、結果を保存する作業に相当）」
  - Step 見出しも「〜を読み取る」「〜を組み立てる」「〜を突き合わせる」「〜を装置に流し込む」の
    ような、装置とデータに何が起きるかが伝わる動詞句にする。
- **用語説明の原則**: 専門用語が文書中に初めて登場する箇所では、括弧書きで一言、平易な言葉に
  言い換える。対象は Python のイディオム・設計パターン（`dataclass`・クロージャ・`fail-fast`・
  `flock`・`MappingProxyType` 等）に限らず、**プログラミングと Web の一般用語（関数・引数・
  戻り値・例外・プロセス・終了コード・JSON・REST API・ハッシュ等）と、このプロジェクトの
  ドメイン語（指紋ゲート・差分同期・望ましい状態等）も含む** — 基準読者にとってはどれも同じ
  「知らない言葉」である。逆に基準読者の日常語（SSH・SFTP・cron・コンフィグ・show コマンド・
  act/sby 等）には言い換えを付けない（過剰な説明はかえって読みにくい）。同じ用語の2回目以降の
  登場では説明を省略してよい（繰り返して冗長にしない）。用語自体を避けて回りくどく書くのでは
  なく、「正確な用語＋簡潔な補足」を基本形にする。
- **構文の非自明性**: 「自明/非自明」は**処理内容**の自明さの話であり、
  **構文**の自明さとは別軸で判断する（完全展開原則によりコード自体は常に引用されるが、
  それとは別に構文への言及が要るかどうかの判断）。想定読者は for/if/try-except 等の基本構文は理解できるが
  それ以外の構文には不慣れという前提（本ページ冒頭）に立つと、以下のような構文が使われている
  行は、処理の意図そのものは自明でも構文自体に一言触れる。「何をしているか（意図）」だけ書いて
  「どういう書き方でそれをしているか（構文の仕組み）」への言及を省略しない:
  - タプル/シーケンスのアンパック代入（例: `a, b = pair`）
  - 内包表記（list/dict/set/ジェネレータ式）
  - 条件式（三項演算子、`x if cond else y`）
  - walrus 演算子（`:=`）
  - `*`/`**` によるアンパック展開
  - 拡張スライス（`[::-1]` 等）
  - デコレータ
  - `with` 文のコンテキストマネージャ
  - 悪い例: `naming, device_profile, source_schema = profiles.naming, ...` の解説を
    「Step 1 で束ねた ProfilesCfg を分解」とだけ書き、複数値を複数変数へ一括代入する
    アンパック代入という構文自体に触れない。
  - 良い例: 「`profiles.naming, profiles.device_profile, ...` の各値を対応する変数へ
    一括代入する（タプルアンパック代入）。Step 1 で束ねた `ProfilesCfg` を分解している」
    のように、構文の仕組み＋処理の意図の両方に触れる。

## 0. 出力構成（インデックス＋フローページ）

```
<出力ディレクトリ>/
  index.md          対象範囲・全体構成図・フロー一覧・仕様書対応表
  flow_<名前>.md     フローページ（エントリポイント1つ = 1ページ。例: flow_sync_rules.md）
```

ファイルは全て同じディレクトリに置く。フローページのファイル名は `flow_` 前置＋エントリポイント
を表す英語スラグにする。フローが1つしか無いスコープでも、この構成（インデックス1つ＋フロー
ページ1つ）で統一する。

**フローとは**: 1つのエントリポイント（CLI の `main()`・cron から呼ばれる関数・公開 API 関数等）
から始まり、処理の完了（正常終了・例外による終了）までの実行の流れ。フローの特定手順と
ユーザー確認は SKILL.md のワークフロー側に従う。

**共有関数の扱い**: 複数のフローから呼ばれる関数（共通オーケストレータ等）は、**各フローページに
重複して展開してよい**。各ページの自己完結（リンクなしで読み切れること）を優先する。生成物なので
重複の保守コストは無い。

## 1. インデックスページ

含めるもの（上から順に）:

0. **このレポートの読み方**: 初めてこの形式のレポートを見る読者向けに、3〜5行程度で書く。
   `<details>`/`<summary>` の展開・折りたたみの操作方法、line-notes の3列（コード/解説/例）の
   読み方、`<details>`（脇道・閉じた状態）と `<details open>`（本筋・開いた状態）の意味の違い、
   用語リンク（下線付き。ホバーでツールチップの要約が出て、クリックで用語集ページの詳しい説明へ
   飛ぶ）の使い方を説明する。わからない言葉が出てきたら、まず用語にカーソルを合わせて要約を
   確認し、それでも足りなければ用語集ページをブラウザ内検索（Ctrl+F/Cmd+F）する、という詰まった
   ときの動線も明記する。用語集セクション自体は「見出しへの明示的アンカー」機構を使い、
   `## 用語集 {#glossary}` のような見出しで実現する。
1. **対象範囲**: スコープ・絞り込みがあればその理由。
2. **全体構成図**: `callgraph` ブロックでモジュール間依存＋各フローの入口を示す。フローページを
   持つノードはブラケット内に `flow_<名前>.html` を書く（クリックでそのページへ遷移する。
   文法は「6. callgraph ブロックの文法」参照）。常に表示したままにする（`<details>` で畳まない）。
3. **フロー一覧**: 各フローページへの相対リンク＋「何を実現する流れか」の1行要約。
4. **仕様書との対応表**: `references/spec-correspondence.md` のテンプレに従う（対象があれば）。
   対応表の本体はこのインデックスに置き、各フローページの Step 本文には該当する仕様 ID
   （REQ-xx 等）をインラインで添える（両方やる — 表は俯瞰用、インラインは読みながら確認する用）。
5. **どのフローにも登場しないコード**: あれば `file:line` 付きで列挙し、「未使用か、別の入口が
   あるかはレビュアーの確認事項」と添える（判定はしない）。

## 2. フローページの構成

上から順に:

```markdown
[← インデックスへ戻る](index.html)

## 概要

**ひとことで言うと:** 2〜3文。このフローを運用者が手作業でやるなら何をすることに相当するか、
を基準読者の日常語だけで書く（例: 「装置に SSH でログインして帯域設定のコマンドを流し込む
作業を、cron が定期的に自動でやってくれるようにしたもの。投入すべき設定は DB の台帳から
自動で組み立て、前回と変わっていない装置には触らない」）。専門用語・仕様 ID はこの段落には
書かない。

**このフローが実現すること:** 1〜2文。対応する仕様 ID があれば添える（REQ-xx）。

**入口:** `main()` — `sync/rules.py:147`（起動方法: cron から `python -m ...` 等）

**出口:** 正常時は終了コード0。異常時の例外→終了コード写像は Step N で説明。

**通る主なモジュール:** sync/rules.py → sync/pipeline.py → fetch/db.py → …（読む前の心構え用）

## フロー全体図

（このフローが通る関数だけの callgraph。ノード id は関数名、ref は file:line。
クリックでページ内の該当コールフレームへスクロールする）

## Step 1: <段階名>
## Step 2: <段階名>
…
## Step N: <段階名>

## 変数一覧
```

- **概要は上記の項目のみで構成する**（ひとことで言うと／このフローが実現すること／入口／出口／
  通る主なモジュール）。
- **メタ情報はページ全体で書かない**（概要に限らない）: 生成の経緯・以前の版との差分・
  「旧版からの変更点」「〜は削除された」のようなリファクタ履歴への言及は、本文・難所解説・
  入出力表・変数一覧のどこにも書かない — この地図は毎回使い捨てで再生成される成果物であり、
  読者は常に「今のコード」だけを知りたい。過去との比較で書きたくなった内容は、「今のコードが
  どうであるか」の記述に言い換える。
- **Step** は処理の意味的な段階（設定読込・データ取得・変換・差分計算・装置適用…）で区切る。
  だいたい「エントリポイント直下の主要な関数呼び出し1つ＝1 Step」になる。Step 見出しは `##`
  （TOC に載せて目次から段階へ飛べるようにするため）。見出しの文言は大原則「運用のことばで
  先に説明する」に従う。
- 読者が上から下へスクロールするだけで、入口から出口まで実行順に読めることを常に確認しながら
  書く。「詳細は後述」「前述の通り」のような前後参照が増えたら、Step の切り方が実行順から
  ずれているサイン。

## 3. Step の書き方

各 Step は次の並びで書く:

````markdown
## Step 2: 外部DBからルール表を取得

**この段階でやること:** ルール表とエリア表を SSH+psql で取得し、件数検証まで行う（REQ-07・REQ-14）。

**実装の要点:**
- 3台のサーバへ act→sby→bcp の順でフェイルオーバーする
- 全台失敗なら `FetchError` を送出（→ Step 6 の終了コード写像で 1 になる）

（呼び出し元のコード抜粋を line-notes で。呼び出し行の解説にはページ内リンクを張る）

```line-notes
    rows = fetch_db(cfg.db) ::: DB取得は[fetch_db()](#fn-fetch_db)に委譲（直下に展開） ::: cfg.db=3台構成 → DbRows(rules=1200行, areas=40行)
```

#### `fetch_db(db_cfg) -> DbRows` — `fetch/db.py:93` {#fn-fetch_db}

<details open>
<summary>本筋: サーバを順に試し、最初に成功した結果を返す</summary>

```line-notes
def fetch_db(db_cfg): ::: フェイルオーバー付き取得の入口 ::: 3台構成 → 1台目成功なら即return
    for server in db_cfg.servers: ::: act→sby→bcp の固定順で試行 ::: servers=[act,sby,bcp] → 1回目: act
        …
```

（さらに内側の呼び出しがあれば、ここに次のコールフレームをネストする）

</details>
````

- **「この段階でやること」**: docstring または仕様書から読み取れる意図。仕様 ID をインラインで
  添える。docstring も仕様書も無い場合は「(docstring/仕様書の記述なし。呼び出され方から推測: …)」
  と明記する。
- **「実装の要点」**: 分岐・例外・戻り値パターンを箇条書きで。例外がどの Step で最終処理される
  かも書く（フローページは一本道なので「この例外はどこへ行くのか」を必ず追いきる）。
- **複数 Step にまたがる長い関数（オーケストレータ）の扱い**: `run_cli`/`run_sync` のような
  「フロー全体を貫く関数」は、全文を1箇所のコールフレームに押し込まない（後続 Step が毎回
  そのフレームへ戻るリンクになり、一本道が壊れる）。代わりに各 Step の冒頭で、**本体のうち
  その段階に該当する数行だけ**を line-notes で示し（h4 アンカーは初出 Step にだけ置き、2回目
  以降の抜粋では冒頭に「（`run_sync()` の続き — 指紋ゲート部分）」のように出どころを明示）、
  呼び出し先のフレームはその Step 直下に展開する。
- 過度に複雑な分岐（多分岐・多重ループ・複数の例外パス）を持つ Step では、「実装の要点」の直後に
  `callgraph` ブロックで分岐を図示してよい（分岐条件はエッジラベルに。終端ノード（早期 return・
  例外送出）は `[...]` を付けず `id` だけにする — 破線ノードとして描画される）。単純な if-else
  1つ程度なら箇条書きで十分であり、無理に図化しない。

## 4. コールフレーム（呼び出し先のその場展開）

呼び出し先関数を呼び出し箇所の直下に展開する仕組み。**h4 見出し＋`<details>`** の組で書く:

```markdown
#### `関数シグネチャ` — `file:line` {#fn-関数名}

<details open>
<summary>本筋: この関数が何をするかの1行要約</summary>

| 項目 | 内容 |
|---|---|
| 入力 | `db_cfg: DbConfig` — 3台のサーバ接続情報 |
| 処理 | act→sby→bcp の順に接続を試行し、最初に成功した結果を返す |
| 出力 | `DbRows`（`rules=1200行, areas=40行` のような実例を添える） |

（line-notes → さらに内側のコールフレーム）

</details>
```

- **h4 見出し**が展開の視覚的な区切りとアンカー（`{#fn-関数名}`）を兼ねる。h4 は TOC には
  載らない（TOC は h2/h3 のみ）ので、コールフレームが増えても目次が汚れない。見出しを
  `<details>` の外に置くのは、折りたたまれていてもアンカーへのスクロールが機能するようにするため。
- **モジュール色分けは自動**: HTML 化時に h4 の「` — `file:line`」部分からモジュールパスが
  抽出され、モジュールごとの色でバッジ・フレーム枠線・ページ内リンク・callgraph ノードが
  自動的に着色される（ページ冒頭に凡例も出る）。このため h4 は必ず
  `` #### `シグネチャ` — `path.py:line` {#fn-...} `` の形を崩さない（`—` と `file:line` を
  省略すると色分けが効かない）。
- **本筋（処理の背骨になる呼び出し）は `<details open>`**（開いた状態）、**脇道（バリデーション・
  ログ整形・エラーメッセージ組立等）は `<details>`**（閉じた状態）にする。読者のクリックが
  「脇道を深掘りしたいときだけ」になるのが狙い。閉じるフレームの `<summary>` には、開かなくても
  流れを見失わない1行要約を書く（「検証ヘルパ群 — 形式不正なら `ConfigError` を送出」等）。
  **本筋・脇道のどちらでも、開いたときに見えるソースコードは省略しない**（完全展開原則）。
  開閉は「読者が今すぐ読む必要があるか」を制御するためのものであり、「情報を載せるかどうか」
  の判断ではない。
- **入力/処理/出力の表**: h4 見出しの直後・`<summary>` の次の行に、上記の例のような「入力/処理/
  出力」の2列 Markdown テーブルを置く。「入力」は主要な引数とその意味（型・実例）、「処理」は
  この関数がすることを1〜2文（動作イメージが伝わる具体的な言い方。「〜を処理する」のような
  抽象的な言い換えで終わらせない）、「出力」は戻り値の型と具体的な実例を書く。副作用のみで
  戻り値が無い関数は「出力」に「なし（副作用: …）」のように書く。この表は「実装の要点」
  （Step 単位、分岐・例外パターンの箇条書き）とは役割が異なる — 表は関数単体の入出力契約、
  「実装の要点」は Step 全体の分岐・例外の流れを扱う。両方使ってよい（重複する場合は表を残し
  「実装の要点」側は省いてよい）。
- `<details>`/`<details open>`/`<summary>…</summary>`/`</details>` は各行単独で書く固定書式
  （HTML 変換器が許可する生 HTML はこの4パターンのみ）。
- **ネストの深さ**: コールフレームの中にさらにコールフレームを置いてよい（HTML 変換時に入れ子の
  `details` が深さとして視覚化される）。目安として開いたフレームのネストは3段まで — それより
  深い呼び出しは、本当に本筋なのか（Step の切り方を変えて浅くできないか）を考え直す。
- **持ち上げ（ネスト上限との両立）**: 本筋の呼び出しチェーンが深く、開フレーム3段を超えそうな
  場合は、途中のフレームをそこで閉じ、チェーンの続きを **Step 直下の兄弟位置に「持ち上げ」て**
  展開してよい。持ち上げる場合は、直前の line-notes の呼び出し行の解説欄に「（直下に展開）」と
  書き、持ち上げた先頭フレームが浮いて見えないようにする。
- **アンカー ID の命名**: ID に使えるのは英数字・ハイフン・アンダースコア（`[\w-]+`）。ドットは
  使えないため、メソッドや同名関数は `_` 連結で一意にする（例: `state.load` → `{#fn-state_load}`、
  `FX2Client.__init__` → `{#fn-FX2Client_init}`）。フロー全体図の callgraph ノード id は
  この変換後の文字列と**完全一致**させる（クリック時のスクロールは `fn-<ノードid>` の完全一致で
  解決されるため）。
- **関数でないコード（例外クラス群・dataclass 定義等）**: h4 シグネチャ書式は関数専用なので
  フレーム化しない。Step 本文の line-notes で直接扱うか、型の一覧は箇条書きで済ませる。クラスに
  非自明な実装（`__init__` 等）がある場合は、そのメソッドを関数と同様にフレーム化してよい。
- **`<summary>` の中身はプレーンテキスト限定**: summary 行は Markdown 処理されない生 HTML
  パススルーなので、バッククォート・リンク記法は効かない（そのまま表示されてしまう）。山括弧を
  含む文字列（`<id>` 等）は `&lt;` にエスケープするか、山括弧を使わない言い回しに変える。
- **同名関数の2回目以降の呼び出し**: 同じページ内で既に展開した関数を再び呼ぶ箇所では再展開
  しない。line-notes の解説欄で初出のフレームへ `[関数名()](#fn-関数名)` とページ内リンクを張り、
  例欄にはその呼び出しの結果の具体値を書く。
- **展開しない呼び出しは無い**（完全展開原則）。標準ライブラリ・外部ライブラリの関数呼び出し
  （`argparse.ArgumentParser(...)` 等、ソースを引用できないもの）だけは例外的に展開できない
  ため、line-notes の解説欄に「何をするか」を書き、必要なら「7a. 深掘り補足」でライブラリの
  一般的な使い方（シグネチャ・戻り値・落とし穴）を補う。

## 5. 難所解説（該当 Step の本文に埋め込む）

独立セクションにしない。読み手が「なぜこんな書き方をしているのか」で詰まりそうな箇所
（並行処理・ロック・リトライ・シークレット処理・低レベル API の使用など）では、該当する
line-notes またはコールフレームの直後に、次の型で短く挟む:

```markdown
**難所 — <一言で何が難しいか>:** 素朴に書くと<何が壊れるか>。ここでは<どう解決しているか>。
見落とすと<レビュアーがどう困るか>。
```

3〜4文で収める。長くなるなら「実装の要点」と重複していないか見直す。

## 6. callgraph ブロックの文法

1行1エッジ。両端は `id[ref]`（通常ノード）か、`id` のみ（終端ノード。file:line を省略、破線
ノードとして描画される）。ラベルは省略可。

```
<左ノード> -> <右ノード>[: <エッジラベル>]
```

- ノード id は空白を含んでよいが `[` `]` `->` は使えない。同じ id が複数行に出てきたら同一ノード
  として自動的にまとめられる（ref は最初に出てきた非空のものが使われる）。
- ブラケット内の `ref` には2種類の書き方がある。**`file:line`**（例: `[sync/rules.py:45]`）は
  ツールチップ表示＋クリックで同じページ内の「id と同じ `{#fn-...}` アンカーを持つ見出し」への
  自動スクロール。**`ページ名.html`**（例: `[flow_sync_rules.html]`、アンカー付きも可）は明示的な
  ページ遷移リンク。**インデックスの全体構成図は後者（flow_*.html）、フローページ内の図
  （フロー全体図・分岐図）は前者**を使う。
- フロー全体図ではノード id を関数名そのものにする — コールフレームの h4 に付けた
  `{#fn-関数名}` と一致していれば、図のノードをクリックするとそのフレームまで自動スクロールし、
  見出しが一瞬ハイライトされる。
- 1つの `callgraph` ブロックに書けるのは1つの図。複数の独立した図が要る場合はブロックを分ける。
- HTML 化されると、パン・ズーム・ノードクリック・簡易フィルタ入力が使える自己完結の
  インタラクティブ SVG になる（外部ライブラリ・外部通信なし）。
- ノード数が多くなりすぎる場合(数十超)は図を分割するか対象を絞ることをユーザーに確認する。
  読み取れないほど密な図を1つ作って終わりにしない。

## 7. line-notes フェンスの書き方（このスキルにおける唯一の正典）

- 1行 = `<コード1行> ::: <その行の解説>[ ::: <例>]`。区切りは半角スペース+コロン3つ+半角スペース
  （` ::: `）で、コードのインデントは変更せずそのまま左側に書く。表示は「コード/解説/例」の
  3列（HTML化されると列見出しが自動で付く）。
- 解説は日本語で、何をしている行かを短い動詞句で書く（「〜する」「〜を返す」等）。専門用語・
  イディオムを使う行では、大原則の「用語説明の原則」に従い、初出時に括弧書きで平易な補足を
  添える（例: `` `@dataclass(frozen=True)`(生成後にフィールドを変更できない設定用クラス) ``）。
  **解説の省略（区切りごと省略してコードだけの行にする）が許されるのは、空行と、閉じ括弧
  （`)`/`]`/`}`）だけの行に限る。** それ以外の行——`try:`・`return 2` のような制御構文単体の
  行、`x = 1` のような単一値の代入、シグネチャ行など——にも必ず一言の解説を付ける（このスキルの
  想定読者は Python 初心者であり、「制御構文だから自明」と Claude 側が判断して省略すると、
  読者にとっては読み解けない空白になる）。省略する場合は **` ::: ` 区切りごと省略してコードだけ
  の行にする**（変換器は区切りの無い行を解説・例なしとして扱う）。区切りを書いて右側を空にする
  書き方は行末スペースに依存して壊れやすいので使わない。いずれの場合もコード行そのものは
  省略しない。
  - 制御構文単体の行の解説例: `` try: ::: 直後のブロックで例外が起きるかもしれない処理を試みる
    (例外処理の開始) ::: 正常終了 → そのまま次の行へ／例外発生 → 直後の`except`節へジャンプ ``、
    `` return 2 ::: 終了コード`2`(設定/状態エラー)を返して関数を抜ける ::: → 呼び出し元に`2`が
    返る ``。
  - 単一値の代入の解説例: `` parser = argparse.ArgumentParser(prog=batch_name) ::: コマンドライン
    引数パーサを作る(`argparse`は標準ライブラリのCLI引数解析機能) ::: `batch_name="rules"` →
    ヘルプ表示等で`prog`名として使われる ``。
- **例（3列目）は、解説欄（2列目）が空でない行には必須で書く。** 例が無いと読み手はその行が
  具体的にどう動くかをイメージできない。解説欄を空欄にした自明な行は例も合わせて空欄にして
  よい — 比例原則が働くのは「解説を書くかどうか」の判断までであり、一度解説を書くと決めた行に
  ついて「例だけ省略する」ことはできない。以下の形式・禁止事項に従って書く。

  **形式（必須）:** `<具体的な入力> → <具体的な結果・出力>` の形にする（矢印は `→` を使う。
  複数ステップを経る場合は矢印を連ねてよい）。入力に相当する値が無い行（戻り値の生成そのもの等）
  は左側を省略してよいが、右側は必ず引用符・数値・具体的なリテラルを伴う値にする。

  **禁止1（片手落ち）:** 入力の具体値だけを書いて結果を書かない、または結果らしき記述だけで
  何を入れたときの話か分からない、という片方だけの記述は禁止。
  - 悪い例: `cli_text="traffic-manager accelerator..."` （結果が無い）
  - 良い例: `cli_text="a,b"` → `subports=["a","b"]`

  **禁止2（「〜の場合」止め）:** `〜が入っていた場合` `〜のとき` のように条件の言い換えで終わり、
  結果が書かれていない文は禁止。条件を書くなら必ず結果（例外名・返り値・状態変化）まで矢印で続ける。
  - 悪い例: `` `[]`や`null`が入っていた場合 ``
  - 良い例: `` `[]`や`null` → `StateError` ``

  **禁止3（具体値の無い言い換え）:** 文字列リテラル・数値・実際に送出される例外名などの具体的な
  値が一つも登場せず、コードの条件をそのまま日本語に言い換えただけの文は禁止。必ずどちらかの側に
  `` `...` `` で囲んだ具体的な値を最低1つ置く。
  - 悪い例: `ファイルとして存在/未存在のいずれか`、`巨大な入力でもログを肥大化させない`
  - 良い例: `path="/tmp/x"` → `os.path.isfile=True`、`本文が10000文字 → 先頭200文字に切詰め`

  **禁止4（解説欄との重複）:** 例欄は解説欄が伝えていない「具体的にどう動くか」を補う場所であり、
  解説欄の言い換えや再掲ではない。解説欄が「他関数へのリンクのみ」の行では、例欄にはその関数を
  **呼んだ結果**の具体値を書く（リンクそのものを例欄に繰り返さない）。

  **一貫性:** 同じ変換パターンが複数箇所に出てくる場合、コピー＆編集の過程で具体性を落とさない。
  1箇所目を `{"a":1,"b":2,"c":3}` → `(1,2)` のように具体的に書いたら、同型の2箇所目以降も同じ
  粒度で書く。

  **較正用の対比（架空コード）:**

  | 良い例 | 悪い例 | 該当する禁止 |
  |---|---|---|
  | `n=42` → 境界値と一致し`True` | `境界値と一致する場合` | 禁止2 |
  | `s="a,b,c"` → `["a","b","c"]` | `カンマ区切りで分割` | 禁止3 |
  | `code="X"` → `_build(code)`を呼び`"prefix-X"`を返す | `[_build()](#fn-_build)`（解説欄と同一） | 禁止4 |

- 解説側・例側は通常の Markdown（`` `code` ``・`**bold**`・リンク）が使える。別の関数を呼んでいる
  行では解説側で必ず `[関数名()](#fn-関数名)` でページ内のコールフレームへリンクする（例側に
  このリンクを繰り返さない。上記「禁止4」参照）。
- 抜粋の範囲: 関数が短ければ全体を、長ければ本筋の理解に関わる箇所（分岐・ループ・例外処理・
  戻り値の組み立て等）を中心に引用する（本筋と無関係な定型的な設定値の羅列まで律儀に全部貼らない
  のは構わないが、いったん引用すると決めた行には全て解説を付ける — 引用範囲を絞ることと、
  引用した行の解説を省略することは別の話）。
- ごく稀に、コード自体に ` ::: ` という文字列がそのまま含まれてしまう場合は、その箇所だけ
  `line-notes` を諦め、説明を1行以上添えた通常の ```python コードブロックにフォールバックしてよい。

### 7a. 深掘り補足（構文の仕組み＋なぜこう書くか）

line-notes 行の解説欄（2列目）は今まで通り簡潔な一言のままにする（大原則「簡潔性」「一本道
原則」は変えない）。その代わり、解説欄が非空の行のうち次のいずれかに当てはまるものには、
**折りたたんだ深掘り補足**を添えてよい:

**対象外（基礎文法）:** `for`/`if`/`def`/`class`/`import`/`try-except` のような基礎文法は
7a の深掘り `<details>` の対象にしない。想定読者の引き下げ（本ページ冒頭）で基礎文法にも
不慣れな読者を想定するようになったが、出現頻度が高い基礎文法を毎回ここで厚く解説すると
大原則の「簡潔性」「一本道原則」を壊す。代わりに「9. 用語集ブロックとインライン参照」の
用語集ページへ `[[for文]]` のような軽量リンクを張るだけで済ませる（同じ用語への2回目以降の
リンクも、初出時と同じリンクを再利用してよい — 用語集側の説明を都度書き直さない）。以下の
対象構文リストは基礎文法より出現頻度が低く、その場の設計判断とセットで解説する価値が高い
ものに限る。

- 大原則「構文の非自明性」に挙げた構文（タプルアンパック代入・内包表記・条件式・walrus演算子・
  `*`/`**`アンパック展開・拡張スライス・デコレータ・with文のコンテキストマネージャ）、または
  型ヒントのユニオン型（`|`）・ジェネリクス（`list[str]`等）・デフォルト引数・キーワード引数・
  関数を値として渡す呼び出し（第一級関数）・複数デコレータの重ね掛け・メソッドチェーンが
  1つでも登場する行（構文が非自明な行。以前の版では「2つ以上重なる行」に限定していたが、
  単一要素でも深掘りに値するなら使ってよい — 制約は下記の比例原則側に移した）
- 構文自体は読めても、「なぜこの書き方・この設計を選んだか」という実務上の理由に踏み込む
  価値がある行（例: なぜこの引数のデフォルトを `None` にするか、なぜこのクラスを
  `frozen=True` にするか、なぜここでこのデータ構造・アルゴリズムを選んだか）
- 標準/サードパーティライブラリの関数・クラスの呼び出しで、その呼び出しがそのライブラリを
  一般にどう使うか（主なシグネチャ・よくある引数・戻り値の型・詰まりやすい落とし穴）に触れる
  価値がある行（例: `argparse.ArgumentParser`・`ThreadPoolExecutor`・`sqlite3.connect`・
  `tempfile.mkstemp`・`http.client.HTTPConnection` 等）。ライブラリ関数の呼び出しは読者が
  ここで初めて出会うとは限らず、**このコードを離れて別の場面で使うときの一般知識**を渡す点が
  他の2条件と異なる。

**比例原則はここで働かせる:** 解説欄だけで読者が迷わない自明な行、または深掘りしても
「解説欄をそのまま言い換えただけ」にしかならない行には付けない。付けるかどうかの判断基準は
「この行の直下にチャットで質問されたら、解説欄の一言だけで答えを終えられるか」— 終えられない
なら深掘り補足の対象。ライブラリ呼び出しの場合はこれに加えて「このAPIを他のコードで再利用
しようとしたとき、このコードでの使われ方の説明だけで足りるか」も判断材料にする。足りる
（このコード固有の使い方で完結する、あるいは呼び出しが単純すぎて一般知識を書くほどでもない）
なら深掘りしない。

**中身（チャットでの回答と同じ深さを目安にする）:**

1. 構文が絡む場合は「要素｜意味」の2列 Markdown テーブルを、コード上の出現順に並べる。
2. ライブラリ機能の呼び出しに該当する場合は、続けて **「このAPIの一般的な使い方」** という
   太字ラベルの後に3〜5行程度で書く。含める内容の目安はシグネチャ概要（主な引数・デフォルト値・
   キーワード専用引数の有無）・戻り値の型と意味・よくある落とし穴（例外を投げる条件・リソース
   解放忘れ・スレッド安全性等）。**このコードでの使われ方の説明ではなく、ライブラリ自体の
   一般知識である**ことが分かるように書く（「このコードでは」ではなく「一般に」「通常は」で
   始める文体を目安にする）。構文テーブルだけで十分な行ではこの項目自体を省いてよい。
3. その下に「なぜこう書くか」を1〜3文程度のプレーンな文章で続ける（実務上の理由・設計意図。
   テーブルだけでは伝わらない「なぜ」を補う）。構文の深掘りだけで十分な行（実務上の理由が
   自明、または特に無い）はテーブルだけでよく、この文章は省いてよい。
4. **難所ブロックとの役割分担:** 難所ブロック（`<details open>`、通常は複数行/関数全体に
   またがる設計判断を扱う）が同じ理由をすでに説明している場合は、ここで繰り返さない
   （「難所参照」のように一言で済ませるか、深掘り補足自体を省く）。深掘り補足は1行スコープの
   構文＋設計理由（＋該当すればライブラリの一般知識）に特化し、難所ブロックはより広い粒度の
   設計判断を扱う、という分担を保つ。

**置き場所:** 該当行を含む `line-notes` フェンスが閉じた直後（フェンスの外、次の行）に
`<details>`を1つ置く。`<summary>` には対象のコード行をバッククォートで引用する。1つのフェンスに
該当行が複数あれば、フェンス直後に `<details>` を複数並べてよい（フェンスの途中に割り込ませない）。
`<details>`/`<summary>…</summary>`/`</details>` は各行単独で書く固定書式（本ページ冒頭の
コールフレームの書式と同じ制約）。

**開閉状態は所属するコールフレームに合わせる:** その行が属するコールフレームが本筋
（`<details open>`）なら深掘り補足も `<details open>` にする、脇道（`<details>`）なら
深掘り補足も `<details>`（閉じた状態）にする。本筋は「読者が今読んでいる場所」なので、
深掘りも含めて開いたまま読み進められるようにする — クリックを要求すると、完全展開原則で
増えた情報量の中で「読めば分かる詳細」まで隠してしまい逆効果になるため。脇道は元々クリック
して初めて読む場所なので、深掘りも閉じたままでよい。

例:

```markdown
```line-notes
def main(argv: list[str] | None = None) -> int: ::: `sync-rules` CLI エントリ ::: cron起動 → argv=None
    return pipeline.run_cli("rules", make_fetch_and_transform, read_current, apply, argv) ::: 3つの関数を [pipeline.run_cli()](#fn-run_cli) に注入して委譲(脇道) ::: → 終了コード(int)
```

<details open>
<summary>深掘り: `def main(argv: list[str] | None = None) -> int:`</summary>

| 要素 | 意味 |
|---|---|
| `argv: list[str]` | 型ヒント。「文字列のリスト」であることを示す(実行時に強制はされない) |
| `\| None` | ユニオン型(Python 3.10+の記法)。「`list[str]`か`None`のどちらか」 |
| `= None` | デフォルト引数。呼び出し時に省略すると`None`が入る |
| `-> int` | 戻り値の型ヒント |

**なぜこう書くか:** `argv` を明示的に受け取れるようにしておくと、cronから実際に起動された
ときは省略して内部で `sys.argv` を使う一方、テストコードからは `main(["--dry-run"])` の
ように明示的な引数リストを渡して呼べる。`sys.argv` に直接依存すると自動テストが書きにくく
なるための定石。戻り値を `int` にしているのはUNIXの終了コードの慣習に合わせたもの
（`0`=正常終了、非0=異常）。

</details>

<details open>
<summary>深掘り: `pipeline.run_cli("rules", make_fetch_and_transform, read_current, apply, argv)`</summary>

| 要素 | 意味 |
|---|---|
| `make_fetch_and_transform`（`()` 無し） | 関数呼び出しではなく関数そのものを値として渡している(第一級関数)。`run_cli` 側が後で自分のタイミングで呼び出す |
| `read_current`, `apply` | 同様に関数を値として渡している |

**なぜこう書くか:** 引数解析・ロック取得・例外→終了コード写像といった共通のCLI実行ロジックを
`pipeline.run_cli` に任せ、`rules` 固有の処理だけを3つの関数として注入している。共通の骨格を
1箇所にまとめつつ、`sync-subscribers` 側は別の3関数を渡すだけで同じ骨格を再利用できる
（依存性注入的な設計）。

</details>
```

ライブラリ使用法の分岐を含む例（構文が単純でも、ライブラリの一般知識に触れる価値がある行）:

```markdown
```line-notes
parser = argparse.ArgumentParser(prog=batch_name) ::: コマンドライン引数パーサを作る(`argparse`は標準ライブラリのCLI引数解析機能) ::: `batch_name="rules"` → ヘルプ表示等で`prog`名として使われる
```

<details open>
<summary>深掘り: `parser = argparse.ArgumentParser(prog=batch_name)`</summary>

**このAPIの一般的な使い方:** `argparse.ArgumentParser`はキーワード専用引数を複数取り、
`prog`（ヘルプ表示に出るプログラム名。省略時は`sys.argv[0]`から自動推定）の他に
`description`（ヘルプ冒頭の説明文）・`epilog`（ヘルプ末尾の補足）・`add_help`
（`-h/--help`を自動生成するか、既定`True`）等がよく使われる。生成したパーサに
`add_argument(...)`で引数定義を積み重ね、最後に`parse_args(argv)`を呼ぶと
`argparse.Namespace`（属性アクセスできる名前空間オブジェクト）が返る、という3段階
（生成→定義→解析）が基本の使い方。未知の引数や必須引数の欠落があると、`parse_args`が
usageメッセージを標準エラーに出して`sys.exit(2)`する（例外ではなくプロセス終了である点に
注意）。

**なぜこう書くか:** `prog`を明示しているのは、cronのログや`--help`表示で「どのバッチの
パーサか」が`sync-rules`/`sync-subscribers`のどちらか判別できるようにするため（省略すると
両方とも`sys.argv[0]`由来の同じ表示になり紛らわしい）。

</details>
```

## 8. 変数一覧（フローページの末尾に1つ）

このフローのコード抜粋に登場した変数を、モジュールレベル（定数・設定値）から関数内ローカルまで
一覧化する。自明なループカウンタ（`i` 等）も省略しない。**折りたたまない**（常に展開された表
として直接埋め込む — 変数一覧はレビュー中に頻繁に参照されるため、クリックの手間を要求しない。
旧版にあった `<details>`/`<summary>` による折りたたみは廃止した）。

```markdown
## 変数一覧

| 変数名 | スコープ | 型/値 | 役割 | file:line |
|---|---|---|---|---|
| MAX_RETRY | モジュール定数 | int = 3 | PATCH失敗時のリトライ上限 | fx2/client.py:12 |
| current_state | `sync_rules()` 内ローカル | dict | 装置から取得した現在状態 | sync/rules.py:46 |
```

- 「スコープ」列: モジュールレベルなら「モジュール定数」「モジュールグローバル」、関数内なら
  `` `関数名()` 内ローカル `` と書く。変数名が同じでもスコープが違えば別の行として扱う。
- 「型/値」列: 型ヒントがあればそれを、無ければ代入されている値から型を読み取って書く。

## 9. 用語集ブロックとインライン参照

想定読者の引き下げ（本ページ冒頭）に伴い、基礎文法（`for`/`if`/`def`/`class`/`import`/
`try-except` 等）や Python 特有のイディオム・専門用語（`dataclass`・クロージャ等）を、
本文を厚くせずに説明するための仕組み。用語を1箇所にまとめて定義し、本文からは軽量なリンクで
参照する。

- **`glossary` フェンス（`index.md` にのみ置く）**: 用語集は `index.md` に1箇所だけまとめる。
  フローページ（`flow_*.md`）側では用語を定義しない — 定義が複数箇所に散らばると、どれが
  正典か分からなくなるため。書式は既存の `line-notes` と同じ ` ::: ` 区切り:

  ````markdown
  ```glossary
  dataclass ::: フィールド定義から__init__等を自動生成するクラスの書き方 ::: 詳しい説明(3列目は任意)
  クロージャ ::: 定義時点で周囲の変数を捕まえたまま持ち運べる関数
  ```
  ````

  1列目は用語（参照キー）、2列目はツールチップに出す一言要約（必須）、3列目は用語集ページ
  だけに表示する詳しい説明（任意）。

- **`[[term]]` によるインライン参照**: 本文（`index.md`・`flow_*.md` のどちらでも）から
  wiki リンク風の `[[term]]` で用語集の項目を参照する。表示テキストを変えたい場合は
  `[[term|表示テキスト]]` とする。既存の Markdown リンク `[text](url)` とは構文が異なるため
  衝突しない。

- **掲載基準（比例原則）**: 大原則「用語説明の原則」「構文の非自明性」で挙げる専門用語・
  イディオムに加え、基礎文法と、プログラミング・Web の一般用語（関数・戻り値・例外・JSON・
  REST API・ハッシュ等）・このプロジェクトのドメイン語（指紋ゲート・差分同期等）もここに
  載せる対象になる（基準読者にとってはどれも「知らない言葉」であり区別しない）。基準読者の
  日常語（SSH・cron・コンフィグ等）は載せない。ただし **1つのフローページ内で1回しか
  登場しない用語は、無理に用語集化せず従来通り括弧書きの一言説明で済ませてよい**。用語集は
  「複数箇所で繰り返し使う用語の参照先」であって、登場する専門用語をすべて義務的に載せる場所
  ではない。

  - 悪い例: `for` 文が出てくる Step ごとに「`for` は繰り返し処理を行う構文」という同じ説明を
    一言一句書き直す。
  - 良い例: `index.md` の `glossary` に `for文 ::: 繰り返し処理を行う構文` を一度だけ定義し、
    各 Step の line-notes 解説欄では `[[for文]]` と書くだけで済ませる。

- **コード中に `[[term]]` という文字列自体を表示したい稀なケース**（生成対象コードにたまたま
  この記法と同じ文字列が含まれる等）では、その箇所だけ `line-notes`/インライン参照を諦め、
  通常のコードブロックへフォールバックしてよい（「7. line-notes フェンスの書き方」末尾の
  ` ::: ` フォールバックと同じ考え方）。

## 見出しへの明示的アンカー（`{#id}`）

見出しの末尾に半角スペース+`{#任意のid}`を付けると、自動生成されるスラグではなくそのIDが
アンカー（HTML の `id` 属性）として使われる（表示テキストからは取り除かれる）。コールフレームの
h4 見出しで使うのが主な用途。ID は英数字・ハイフン・アンダースコア（`[\w-]+`。ドット不可 —
命名変換は「4. コールフレーム」のアンカー ID の命名を参照）。同名関数が複数モジュールにある場合は
`モジュール名_関数名` のように衝突しない ID にする。ページ内リンクは `#id` で参照する。

## 書き終えたら

- **基準読者テスト**: 各フローページの「ひとことで言うと」と各 Step の見出し・1文目だけを
  拾い読みし、基準読者（本ページ冒頭）の日常語だけで読めるか確認する。実装用語
  （オーケストレーション・クロージャ・差分同期・写像等）が言い換えなしで見出しや1文目に
  出てくる箇所は、大原則「運用のことばで先に説明する」の2文構成に書き直す。概要にメタ情報
  （生成経緯・以前の版との差分）が混ざっていないかも確認する。
- 使った `file:line` を全部リストアップし、`grep -n` で1つずつ実ソースと突合する。
- スコープ内の全エントリポイントにフローページがあるか、インデックスの「フロー一覧」と生成した
  ファイルが一致しているか確認する。フローに乗らないコードはインデックスの「どのフローにも
  登場しないコード」に列挙されているか確認する。
- **一本道の通し読み確認**: 各フローページを頭から読み下し、(1) 実行順と節の並びが一致しているか
  (2) 展開もリンクも要約も無い呼び出し行が残っていないか (3) 例外の行き先が最後まで追えるかを
  見直す。
- **完全展開原則の確認**: 「呼び出し先はコールフレームとして展開する」対象になっている関数呼び
  出しのうち、コールフレーム化(h4 見出し)されずに1行要約だけで済まされているものが残っていない
  か見直す(標準/外部ライブラリの呼び出しでソースを引用できないものは対象外)。全てのコール
  フレームに「入力/処理/出力」の表があるか、`file:line` が実ソースと一致しているかも合わせて
  確認する。
- `{#fn-関数名}` で用意したアンカーIDと、line-notes 解説欄・フロー全体図のノード id がそれぞれ
  一致しているか確認する（打ち間違えるとリンク/スクロールが無音で外れる）。
- インデックスの「フロー一覧」のリンク先ファイル名・全体構成図の `flow_*.html` と、実際に生成する
  HTML ファイル名が一致しているか確認する。各フローページ冒頭の「← インデックスへ戻る」も同様。
- `<details open>`（本筋）と `<details>`（脇道）の使い分けが方針通りか見直す — 全部開いている・
  全部閉じているページは、本筋/脇道の判断を省略したサイン(完全展開原則により脇道もソースは
  必ず引用されているはずなので、閉じていること自体は問題ない)。「7a. 深掘り補足」の開閉状態が
  所属するコールフレームの開閉状態と一致しているかも確認する。
- 「7a. 深掘り補足」を使った行がある場合、判断基準（「解説欄の一言だけで読者の疑問が終えられ
  ないか」）に照らして過剰適用・過小適用がないか見直す。解説欄をそのまま言い換えただけの
  補足になっていないか、難所ブロックと同じ理由を重複して書いていないか、line-notes の解説欄
  自体が簡潔さを失って深掘り補足に頼っていないかを確認する。「このAPIの一般的な使い方」を
  書いた行では、その内容が本当にライブラリ自体の一般知識になっているか（「このコードでは」で
  終わる、このコードの説明に成り下がっていないか）も見直す。
- `line-notes` の例（3列目）について、次のセルフチェックを行う（目視で見直すべき行を洗い出す
  ための足がかり）:
  - **解説欄（2列目）が非空なのに例（3列目）が無い行がないか確認する（最も見落としやすい）。**
    ```bash
    python3 -c '
    import glob
    for f in glob.glob("*.md"):
        for i, line in enumerate(open(f, encoding="utf-8"), 1):
            if line.count(" ::: ") == 1:
                _, note = line.rstrip("\n").split(" ::: ")
                if note.strip():
                    print(f"{f}:{i}: 解説はあるが例が無い -> {note.strip()}")
    '
    ```
  - 「→」を含まない例がないか確認する（禁止1・禁止2に触れている疑いのある行の候補）。
    ```bash
    grep -n ' ::: .* ::: ' *.md | grep -v '→'
    ```
  - 「〜場合」で終わる例が残っていないか確認する。
    ```bash
    grep -n ' ::: [^:]*場合\s*$' *.md
    ```
  - 解説欄と例欄が一字一句同じになっている行がないか確認する（禁止4）。
    ```bash
    python3 -c '
    import glob
    for f in glob.glob("*.md"):
        for i, line in enumerate(open(f, encoding="utf-8"), 1):
            if line.count(" ::: ") == 2:
                _, note, ex = line.rstrip("\n").split(" ::: ")
                if note.strip() and note.strip() == ex.strip():
                    print(f"{f}:{i}: 解説と例が同一 -> {note.strip()}")
    '
    ```
  - 上記でヒットした行は「本当に禁止パターンか」を目視で判断する（`→`が無くても正当な例が
    ありうる場合は残してよい）。
- 完成した Markdown を見直し、通常のコードブロック（```python 等）が残っていないか確認する
  （コード抜粋は `line-notes` フェンスが原則。素のコードブロックは ` ::: ` を含むコードへの
  フォールバック時のみのはず）。
