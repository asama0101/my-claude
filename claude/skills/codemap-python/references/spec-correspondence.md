# 仕様書/要件との対応表の作り方

## 1. 自動探索

対象プロジェクトのルートから、仕様書らしきものを探す。存在すれば優先度の高い順に採用する。

1. `docs/spec/`、`docs/specs/`、`docs/requirements/` のようなディレクトリ
2. `REQ-` `NFR-` `ADR-` のような ID パターンを含む Markdown ファイル
   （`grep -rlE 'REQ-[0-9]+|NFR-[0-9]+' docs/ 2>/dev/null` のように探す）
3. `docs/adr/` や `docs/decisions/` などの ADR (Architecture Decision Record) 群
4. プロジェクトルートの `CLAUDE.md`/`README.md` に仕様書の参照順や場所が明記されていないか
   （shaper-db の `CLAUDE.md` の「仕様書の参照順」節のような記述）
5. 上記で何も見つからない場合、`README.md` の設計・仕様に関する節（"Design"/"Architecture"/
   "Requirements" 等の見出し）を次点として使う

## 2. ユーザーへの確認

自動探索の結果は**そのまま使わずユーザーに提示して確認を取る**。理由: 汎用スキルとして複数の
プロジェクトで使われるため、探索ロジックが誤検出する可能性がある（例: `docs/spec/` はあるが
実際は別用途、あるいは仕様書が別リポジトリ/Wiki にある等）。

確認の型:
```
仕様書として次を見つけました:
- docs/spec/01_requirements.md ほか5ファイル (REQ-xx/NFR-xx形式のIDあり)

これを仕様の正典として対応表を作ってよいですか？ 違う場所を使うべきであれば教えてください。
```

見つからなかった場合:
```
このプロジェクト内に仕様書らしきものが見つかりませんでした。
参照すべき仕様書やREADMEの場所があれば教えてください。無ければ対応表は「該当なし」として進めます。
```

ユーザーが「無い」「進めてよい」と答えた場合のみ、対応表を「該当なし」として省略してよい
（黙って省略しない）。

## 3. 対応表フォーマット

ID 形式の要件がある場合（例: `REQ-xx`/`NFR-xx`）はそれを使う。無い場合は仕様書の見出し名を
そのまま識別子として使う。

対応表の本体は**インデックスページ**に置く（俯瞰用）。加えて、各フローページの Step 本文
「この段階でやること」にも該当する仕様 ID をインラインで添える（読みながらその場で確認する用。
`flow-map-format.md` 3節参照）。表とインラインの両方をやる — どちらか片方に省略しない。

```markdown
### 仕様書との対応表

| コード (file:line) | 対応する要件/仕様 | 備考 |
|---|---|---|
| `transform/rules.py:42 generate_rule_name` | REQ-12 (ルール名生成規則) | 命名パターンが仕様と一致するか目視確認 |
| `sync/subscribers.py:88 sync_subscribers` | 05_sync-spec.md §2.3 | ID形式の要件がないため見出し名で対応付け |
```

- 対応が見つからない実装（仕様に書かれていない振る舞い）は「対応する仕様記述が見当たらない」と
  明記する。逆に、仕様に書かれているが実装が見当たらないものも同様に注記する。どちらも判定
  （バグかどうか）はせず、レビュアーへの確認事項として提示する。
- 仕様書を実際に Read してから対応付けること。ファイル名や見出しだけを見て推測しない。
