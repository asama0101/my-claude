# codemap-python: Python レビュー地図生成（処理フロー軸）

Python コードベースを初めて読む、または久しぶりに読む人間のレビュアーのために、「処理フローの
一本道＋仕様対応表＋ロジック可視化」を、インデックスページ＋フローページ群のHTMLレポート一式に
まとめて生成する。**このスキル自身はバグの有無を判定しない。** 判定は人間、または
`review-correctness` のような別スキル/エージェントの仕事。このスキルの役割は「見るべき場所を
具体的に指し示し、理解を助けること」に徹する。

**構成の軸はモジュール（ファイル）ではなく処理フロー。** ページをファイル構造で分割すると、
1つの処理を追うのに何ページもリンクで行き来することになり、人間の読解が破綻する（実証済みの
失敗構成）。エントリポイントから実行順に読み下せる「一本道」のページを作り、呼び出し先の
コードはモジュール境界を越えてその場に展開する。詳細な書式は
`references/flow-map-format.md` が正典。

**レビュー観点チェックリストは作らない（全面廃止）。** チェックリストは「判定を促す問いかけ」
であり、このスキルの「判定はしない」という役割と相容れない。判定・チェックが要る場合は
`review-correctness` など別のスキル/エージェントに委ねる（ユーザーから依頼された場合も、
このスキルでは作らずその旨を伝える）。

## いつ使うか

- 初めて読むモジュール/パッケージの処理の流れを把握したいとき
- PR や `git diff` のレビュー準備をしたいとき
- 仕様書とコードの対応が取れているか確認したいとき
- 「地図」「マップ」「俯瞰」「流れを追いたい」「構造を整理して」のような言葉が出てきたとき

## ワークフロー

### 1. スコープを決める

呼び出し時の文脈から、次のどちらかを判断する。曖昧なら質問する。

- **diff/PR スコープ**: `git diff`（未コミット差分、またはユーザー指定のベースブランチとの差分）で
  変更されたコードを起点に、**変更コードを実行時に通るフローだけ**を対象にする（呼び出し元を
  `grep -rn` で遡ってエントリポイントを特定する）。変更に無関係なフローは作らない。
- **全体スコープ**: ユーザーが指定したディレクトリ/パッケージ配下の**全エントリポイント**を対象に
  する。指定が無ければ対象の候補（例: `src/` 配下のトップレベルパッケージ）を挙げてユーザーに
  確認する。

### 2. 仕様書/要件を探す

`references/spec-correspondence.md` の手順に従って自動探索し、見つけた候補をユーザーに提示して
確認を取ってから対応表作成に進む。確認を飛ばさない（誤検出のリスクがあるため）。

### 3. フローを特定し、ユーザーに確認する

エントリポイントを機械的に洗い出す:

- `if __name__ == "__main__"` / `def main(` を `grep -rn` で検索
- `pyproject.toml`/`setup.py`/`setup.cfg` の `console_scripts`・`[project.scripts]`
- cron・systemd・ドキュメント（README・運用手順書）に書かれた起動コマンド
- ライブラリなら公開 API（`__init__.py` の `__all__`、ドキュメント記載の関数）

見つけたエントリポイントを「フロー候補一覧（名前＋入口 `file:line`＋1行説明）」としてユーザーに
提示し、**どのフローをページ化するか確認を取ってから**次に進む。フロー数が多い場合（5本超）は
主要なものに絞ることを提案する。黙って一部だけ処理して「完了」と報告しない — 絞った場合は必ず
その旨をレポート冒頭に明記する。

### 4. コードを読む

各フローについて、エントリポイントから呼び出しを実際に辿りながら Read する（推測で書かない）:

- 実行順の段階分け（Step の切れ目）: 設定読込・データ取得・変換・適用…のような意味的な段階
- 各関数の分岐・例外・戻り値パターン。**例外がどこで最終処理されるか（終了コード写像等）まで
  追いきる** — フローページは一本道なので「この例外はどこへ行くのか」を宙に浮かせない
- 呼び出しの本筋/脇道の区別（本筋=入力を出力に変換する背骨、脇道=バリデーション・ログ・エラー
  整形等）。この区別がコールフレームの開/閉に直結する
- 呼び出し関係は `関数A[file:line] -> 関数B[file:line]` の形で控えておく（フロー全体図に使う）

範囲が広い場合は、複数のサブエージェントに**フロー単位で**分担させて並行調査・執筆してよい
（フローはページとして独立しているため分担に向く）。その場合、各エージェントには担当フローの
エントリポイントと `references/flow-map-format.md` を読むよう指示し、`file:line` 付きの事実に
基づいて書かせる。

### 5. Markdown 中間生成物を書く（フローごと＋インデックス）

`references/flow-map-format.md` の書式に従い、「フロー数＋1」のファイルを書く:

- **インデックスページ**（`index.md`）: 対象範囲、全体構成図（`callgraph`）、フロー一覧、
  仕様書との対応表、どのフローにも登場しないコードの一覧。
- **フローページ**（`flow_<名前>.md`、フローごとに1つ）: 概要 → フロー全体図 → Step 1..N
  （実行順、呼び出し先はコールフレームでその場に展開）→ 変数一覧。

書き終えたら `flow-map-format.md` の「書き終えたら」チェックリストを実施する（`file:line` の
実ソース突合・一本道の通し読み確認・アンカー整合・line-notes セルフチェック）。これを
飛ばさない — このスキルの成果物の信頼性はここにかかっている。

### 6. HTML化

Markdown ファイルの数だけ `build_report.py` を実行する（インデックス1回＋フローごとに1回）。
出力先ディレクトリ名・ファイル名は入力の Markdown と対応させ、同じディレクトリに並べる。
`index.md` の `glossary` フェンス（用語集）を `flow_*.md` 側の `[[用語]]` 参照から解決するため、
**必ず `index.md` を最初にビルドし、そのあとで各 `flow_<名前>.md` をビルドする**（実行順序に
依存関係がある）。

まず `index.md` をビルドする。`--glossary-out <出力先ディレクトリ>/.glossary.json` を指定し、
`index.md` 中の `glossary` フェンスを JSON として書き出す:

```bash
python3 <このスキルのディレクトリ>/scripts/build_report.py \
  --input <index.mdのパス> \
  --output <出力先ディレクトリ>/index.html \
  --title "<対象範囲名>" \
  --target "<対象範囲の説明。例: src/shaper_db/ (全体スコープ)>" \
  --glossary-out <出力先ディレクトリ>/.glossary.json
```

続けて各 `flow_<名前>.md` をビルドする。`--glossary-in` に上記と同じパスを指定し、本文中の
`[[用語]]` 参照を用語集ページへのリンクとして解決する:

```bash
python3 <このスキルのディレクトリ>/scripts/build_report.py \
  --input <flow_<名前>.mdのパス> \
  --output <出力先ディレクトリ>/flow_<名前>.html \
  --title "<フロー名>" \
  --target "<対象範囲の説明。例: src/shaper_db/ (全体スコープ)>" \
  --glossary-in <出力先ディレクトリ>/.glossary.json
```

出力先は対象プロジェクトのルート直下 `docs/codemap/<対象を表す短い語>/`（例:
`docs/codemap/shaper-db-flow/index.html` ＋同ディレクトリに `flow_sync_rules.html` 等）にする。
ディレクトリが無ければ作成する。この地図は使い捨てのレビュー成果物であり、`doc-updater` が
管理する永続的なドキュメント（README・ガイド等）とは別物。プロジェクトルートに `.gitignore` が
あり `docs/codemap/` がまだ無視対象になっていなければ、`docs/codemap/` の1行を追記する（使い捨て
成果物が誤ってコミットされるのを防ぐため）。追記した場合は最終報告でその旨を一言添える。
`--glossary-out`/`--glossary-in` でやり取りする `.glossary.json` はビルド時の中間生成物であり
最終成果物ではない。出力先ディレクトリ（前述の `docs/codemap/` 配下）に残っても実害は無く、
ビルド完了後に削除してもよい。
Artifact 化はしない（対象コードの内容を外部ホストへ公開しないため、常にローカルファイルとして
保存する）。

### 7. 報告する

生成した **インデックス** HTML の保存先パスを一文で伝える（そこから各フローページへ辿れるため、
これが入口になる）。レポート内容の要約（フロー数・対応表の件数など）を簡潔に添えてよいが、
レポート本文の中身をここで繰り返し出力しない（レポートを見れば分かるため）。

## 参照ファイル

- `references/flow-map-format.md` — フローページ/インデックスの構成・Step・コールフレーム・
  難所解説・行ごとのコード注釈（`line-notes`）・`callgraph` 文法のテンプレ定義（書式の正典）
- `references/spec-correspondence.md` — 仕様書の自動探索手順とユーザー確認フロー
- `scripts/build_report.py` — Markdown→HTML変換スクリプト（標準ライブラリのみ、`--help` で使い方確認可。
  `callgraph`/`line-notes` フェンスと見出しの明示アンカー`{#id}`、`<details>` 固定書式に対応）
- `assets/report_template.html` — 上記スクリプトが差し込む自己完結HTMLテンプレート（直接編集する用途ではない）
