_chunks(seq, size) -> Iterator[tuple] — src/shaper_db/sync/subscribers.py:20

意図: docstring「seq を size 件ずつのタプルに分割するジェネレータ。」。04_fx2-api.md§2.4/2.5 の「1リクエスト最大 1,000,000 件」制約に対応するための共通ヘルパ。

呼び出し元: apply()(同モジュール内、src/shaper_db/sync/subscribers.py:103,105)からのみ呼ばれる。

実装の要点:

  • range(0, len(seq), size) によるスライス分割。seq が空なら1回もイテレートしない

(apply() 側の早期return no-op と合わせて、空チャンクが送信されることはない)。

  • 最終チャンクが size 件未満でも問題なくスライスされる(Pythonのスライスは範囲外を

自動的に切り詰めるため、オフバイワンの心配がない)。

  • size に0以下の値を渡すと range() が ValueError を送出する。ただし呼び出し元は常に

モジュール定数 _CHUNK(または system テストで monkeypatch された正の整数)を渡すため、実運用経路でこの分岐に到達することはない。

apply(client, group_name, diff) -> None — src/shaper_db/sync/subscribers.py:84

意図: docstring「差分を装置へ適用する(design §5 手順3、04_fx2-api.md §2.4/2.5・05_sync-spec.md §3.5)。」

呼び出し元: pipeline._sync_one_device(src/shaper_db/sync/pipeline.py:106)から、Apply 型のコールバックとして呼ばれる(read_current と同様、注入された関数オブジェクトとして呼ばれる)。

実装の要点:

  • diff.to_add/diff.to_delete が両方空なら即 return(装置に一切触れない no-op、

test_apply_noop_empty で回帰テスト済み)。

  • 削除を先に行ってから追加する順序を関数内のコード順(103行目→105行目)で保証している。

理由は docstring: flow_id/address が装置全体で一意という制約があり、値の変更(=同じflow_id で address だけ変わるケース)は削除→再登録でしか行えないため、追加が先だと一意制約違反になりうる。

  • to_delete/to_add それぞれを _chunks(..., _CHUNK) で分割し、チャンクごとに

client.delete_flows(flow_ids=[...])/client.post_flows(flows=[...]) を呼ぶ。

  • sync/rules.py の apply() が最後に client.commit_config(...)(PATCH→refresh moff→

save moff の3ステップコミット)を呼ぶのに対し、この関数は一切 commit を呼ばない(flows は即時反映のため。docstring・test_apply_no_commit で明記/回帰テスト済み)。

  • 分割送信の途中(例: 3チャンク中2番目)で Fx2Error が発生した場合、それ以前に送信済みの

チャンクは装置に適用されたまま残り、この関数はロールバックを一切行わない。docstring はこれを意図的な設計(「次周期の事前読取 diffによる自己修復に委ねる」)と説明している。

確認ポイント: 分割送信の部分適用(fail-loud + 自己修復委譲)は 05_sync-spec.md §7 の「失敗・自己修復モデル」を根拠にした設計判断として docstring に明記されている。ただし_sync_one_device(src/shaper_db/sync/pipeline.py:83-109)は例外を捕捉した時点で即座にDeviceOutcome(device.id, "failure", str(exc)) を返すため、「delete は完了したが post が一部だけ失敗した」ような部分適用の詳細(どのチャンクまで成功したか)はエラーメッセージ(str(exc))に残らない。障害対応(06_operations.md §6)の観点でこの粒度で十分か確認する価値がある。

制御フローの可視化: apply()

apply() は分岐こそ1つの早期returnのみで単純だが、「削除→追加」の順序と「commitを呼ばない」という非対称性がレビュー時に見落とされやすい非自明な設計判断のため、簡易図で可視化する。

この図が示す通り、DELETE分割送信 の途中(2チャンク目以降)で失敗しても、既に送信済みの削除チャンクは取り消されない。同様に POST分割送信 の途中で失敗した場合、全ての削除と一部の追加だけが装置に反映された状態で例外が伝播する。apply() の実装コード上の要点にも既述の通り、これは docstring が「意図した設計」と説明している挙動。
