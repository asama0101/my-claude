対象範囲

src/shaper_db/transform/ 配下の全モジュール(3ファイル、__init__.py を除き実質413行)を対象とする。絞り込みは行っていない(モジュール数が3つと少なく、全体スコープで問題ない規模のため)。

  • common.py (120行) — 識別子/subport/flow_id 検証の正規表現、SkipReason・TransformSummary・

行検証共通ヘルパ

  • rules.py (115行) — build_rules: 外部DBのルール表から Rule(FX2 subport 相当)を構築
  • subscribers.py (178行) — build_subscribers: 顧客CSV・エリア表から Subscriber(FX2 flow

相当)をロンゲストマッチで構築

仕様書は docs/spec/03_data-spec.md(データ仕様書。変換規則(REQ-04)・検証規則(REQ-10)・データ不整合時の扱い(REQ-09)を規定)を正典として使用した。プロジェクト直下の CLAUDE.md の「仕様書の参照順」節で明記されている参照順に従い、transform/ の対応仕様として 03 番を採用している。

呼び出し元は src/shaper_db/sync/rules.py・src/shaper_db/sync/subscribers.py の2箇所のみ(いずれも sync/ 配下のエントリポイントから1回ずつ呼ばれる)。

src/shaper_db/transform/rules.py

平たく言うと: 外部DBのルール表(4フィールドの行イテラブル)から FX2 の subport に相当するRule を構築する。1行1行を検証し、問題なければ Rule を1件生成、問題があればスキップして理由をカウントする。

src/shaper_db/transform/subscribers.py

平たく言うと: 顧客CSV(3フィールド)とエリア表(2フィールド)から、ロンゲストマッチで顧客の所属エリアを決定し、FX2 の flow に相当する Subscriber を構築する。rules.py より1段複雑(エリア索引の事前構築・IPv6アドレスのビット演算によるロンゲストマッチが加わる)。

なぜ要るか: 03_data-spec.md 2.1節「所属エリア決定(ロンゲストマッチ)」の変換規則を実装する唯一の場所。高速化のためプレフィックス長ごとのハッシュテーブル方式を採用しており(同仕様書101-106行目)、その実装が _build_area_index/_longest_match/_truncate_to_length に分割されている。

呼ばれ方: src/shaper_db/sync/subscribers.py:58 の1箇所からのみ呼ばれる(result = build_subscribers(customer_rows, db_rows.areas, config.groups))。

主な関数:

_build_area_index(area_rows) -> dict[int, dict[int, str]] — subscribers.py:53

意図: エリア表から {prefix_len: {正規化ネットワークアドレス(int): poi_id}} を構築する(docstring 54-58行目)。poi_id の4桁形式違反・prefix のパース不能行は索引に入れない(ASK-2a,P1-SUB-01)。ビルド毎に1回だけ構築し、顧客ループ内では再構築しない(R8、性能上の要点)。

呼び出し元: build_subscribers()(subscribers.py:119)からのみ呼ばれる非公開ヘルパ。

実装の要点:

  • arity(2フィールド)不一致 → スキップカウントせず黙って continue(subscribers.py:62-63)。

build_rules や build_subscribers 本体のような SkipReason 記録が無い点に注意。

  • poi_id が AREA_ID_PATTERN に不一致 → 同様に黙って continue(subscribers.py:65-66)。
  • ipaddress.IPv6Network(poi_prefix, strict=False) が ValueError → 黙って continue

(subscribers.py:67-70)。

  • 同一 poi_id が複数プレフィックスに対応する場合(仕様書1.2節に記載あり)は、bucket に

複数エントリとして自然に追加される。

確認ポイント: このヘルパはエリア表の不正行を検証している(顧客CSVではない)。03_data-spec.md4節の「データ不整合時の扱い(REQ-09)」はスキップ理由をログに記録する対象を主に顧客行の検証と読めるが、エリア表側の不正行は SkipReason カウントの対象になっていない。この非対称性が仕様上意図されたものか(エリア表はDB取得時点で1.4節の完全性検証を通過済みで、行レベルの形式異常は起こりにくい前提なのか)確認する価値がある。

意図: IPv6アドレス整数値を上位 length ビットへマスクして切り詰める(R1)。ipaddress.IPv6Network オブジェクト生成を避け、整数ビットマスクのみでネットワークアドレスを求める(NFR-01、索引キーと同じ int 正規化、docstring 78-80行目)。

意図: 顧客ネットワークに最長一致するエリアの poi_id を返す(2.1節、P1-SUB-04)。照合対象は顧客プレフィックス長以下のエリア長に限り、長い順に走査する(P1-SUB-05)。

呼び出し元: build_subscribers()(subscribers.py:150)からのみ呼ばれる非公開ヘルパ。

実装の要点:
