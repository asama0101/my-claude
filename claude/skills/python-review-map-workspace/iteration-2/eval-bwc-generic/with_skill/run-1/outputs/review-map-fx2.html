2.1 呼び出し関係の全体図

fx2/client.py の各公開メソッドを、どの呼び出し元（オーケストレーター/API 層）がどの用途で呼んでいるかを示す。矢印は「呼び出し元 → 呼び出し先」の向き。

この図から読み取れること: fx2/client.py は core/ 配下の 4 つのオーケストレーター（subport_orchestrator・flow_id_orchestrator・sync_engine・rollback_engine）と api/devices.py の計 5 箇所から共通に使われる薄い通信層である。FX2Client の生成は必ず core/orchestrator_utils.py:13 make_fx2_client() を経由する（直接 FX2Client(...) を呼ぶ箇所は orchestrator_utils.py:16 の 1 箇所のみ）。

2.2 モジュールブロック

src/bwc_provisioner/fx2/client.py

平たく言うと: FitelNet FX2 装置への REST API 通信を一手に引き受けるクライアント。FX2Client クラス 1 つと、2 つの例外クラス（FX2CommitError / FX2PostCommitError）、およびモジュール定数（PATCH テンプレート・リトライ回数）で構成される。

なぜ要るか: FX2 装置固有の通信プロトコル（3-step commit・PATCH ボディのテンプレート整形・リトライ・discard による取消）をこの 1 ファイルに閉じ込めることで、呼び出し元（4 つのオーケストレーター + API 層）は「CLI 差分文字列を渡すだけ」で FX2 とやり取りできる。device_id（実体は hostname。CLAUDE.md 記載の gotcha）ごとにインスタンスを作り直す設計のため、装置間の状態共有はない。

呼ばれ方: 2.1 の全体図の通り、core/orchestrator_utils.py:13 make_fx2_client() 経由でインスタンス化され、4 つのオーケストレーターと api/devices.py の計 5 箇所から利用される。

実装の要点:

制御フロー（3-step commit の分岐）:

このグラフの「PATCH実行」「refresh実行」「save実行」はそれぞれ内部で最大 3 回のリトライを含む（_patch_config/_cli_command 側のループ。下記参照）。

確認ポイント: spec.html §10（非機能要件）の「障害対応: FX2 接続失敗時 リトライ 3 回後にエラー記録・処理継続」という記述は FX2 通信全般を指しているように読めるが、実際にリトライループを持つのは apply_subport_config が呼ぶ _patch_config/_cli_command のみで、apply_flow_ids/delete_flow_ids/get_subports_raw/get_flow_ids_raw/show_version にはリトライが無い（後述）。この非対称が意図的か確認する？
