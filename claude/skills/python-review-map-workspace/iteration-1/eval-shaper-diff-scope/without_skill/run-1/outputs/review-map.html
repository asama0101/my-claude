3. 関数ごとの詳細

_chunks(seq: tuple, size: int) -> Iterator[tuple]

何をするか: シーケンスを size 件ずつのタプルに分割するジェネレータ。単純なスライスループ(range(0, len(seq), size))。

なぜ要るか: 04_fx2-api.md §2.4/2.5 の「1リクエスト最大1,000,000件」制約を満たすため、apply() が delete/post 両側で使う。

境界: size の倍数でない件数(5件/chunk=2)でのオフバイワンは system テスト test_apply_chunks_both_sides で検証済み(3チャンク: 2,2,1)。

_subscriber_to_row(sub) / _flow_to_row(flow: dict) / _row_to_flow(row: tuple)

何をするか: 3つの相互変換ヘルパー。正準タプル (flow_id, subport, address) を軸に、 Subscriber→タプル、装置応答dict→タプル(count_id 破棄)、タプル→POST用dict、を行う。

なぜ要るか: diff_rows(sqlite3ベース)は列数が両辺で一致していないとcrashする。 装置応答は4キー(count_id 込み)、desired側は3キーのため、_flow_to_row で明示的に3-tuple化しないと 列数不一致でクラッシュする — この一致がこの commit の中核的な正しさの前提。

呼ばれ方: _subscriber_to_row は make_fetch_and_transform 内、 _flow_to_row は read_current 内、_row_to_flow は apply 内(POST時)。

注目点: row のタプル位置(row[0]=flow_id, row[1]=subport, row[2]=address)という「順序による契約」が3関数と apply() の delete処理 (row[0] だけ使う)に暗黙に散らばっている。フィールド取り違えのリグレッションは test_apply_sends_exact_field_mapping が実際の送信bodyのキー・値まで検証することで ミューテーション耐性を担保している(コミットメッセージにも明記)。

make_fetch_and_transform(config: Config) -> FetchAndTransform

何をするか: fetch_db(config.db) + fetch_subscribers(config.csv) → build_subscribers(customer_rows, db_rows.areas, config.groups) → group_name別に 正準タプルへ集約するクロージャを返す。

なぜ要るか: pipeline.run_sync が要求する FetchAndTransform = Callable[[], dict[str, list[tuple]]] 型を満たす「望ましい状態」の生成。

rules.py との非対称: build_rules は config.sync.pir_max_kbps を追加引数に取るが、 build_subscribers は取らない(subscribers に PIR 概念が無いため)。docstring に明記。

未使用の戻り値: fetch_db は DbRows(rules, areas) を返すが、 ここでは db_rows.areas のみ使用し db_rows.rules は破棄する (fetch_db 自体は sync-rules と共用の既存関数で、この commit の変更範囲外)。

read_current(client, group_name, desired_rows) -> list[tuple]

何をするか: group_name/desired_rows 引数を無視し、 client.get_flows([])(空リスト=クエリなし=全件取得)で装置上の全flowを取得、 _flow_to_row で3-tuple化して返す。

なぜ全件か: (1) subport絞り込みは04_fx2-api.md §2.3の「最大10件」制約に抵触しうる、 (2) desiredに無いsubport上の残存flowを取りこぼすとドリフト検知漏れになる。 1装置=1グループ不変は config._validate_devices_unique(config.py:381-406、この commit の変更範囲外)が config読込時に保証するため、全件取得しても他グループのflowと混ざる心配がない。

fail-loud: Fx2Error を捕捉しない。捕捉するのは pipeline._sync_one_device(pipeline.py:83-109)の責務。

apply(client, group_name, diff: DiffResult) -> None

何をするか: 差分ゼロなら即return(no-op)。それ以外は 削除を先に全部実行してから追加を実行する。delete/post いずれも _CHUNK(1,000,000)件単位に分割する。commit_config は呼ばない。

削除→追加の順序の理由: flow_id/address は装置全体で一意制約があり (04_fx2-api.md §2.4)、更新APIが無いため「値の変更」は削除→再登録でしか表現できない。 追加を先にすると一意制約違反になりうる。

rules.py との非対称: rules は subport の CLI コンフィグ差分であり commit_config(PATCH /config → refresh moff → save moff)必須。flows は REST API(POST/DELETE /dynamic-queueing/flows)で即時反映されるため、 commit手順は一切呼ばれない(system テスト test_apply_no_commit が /api/v1/cli・PATCH /api/v1/config への呼び出しゼロを確認)。

失敗時の巻き戻し: チャンク分割送信の途中で失敗しても明示的なロールバックはしない (fail-loud、Fx2Error は捕捉しない)。次周期の事前読取diffによる自己修復に委ねる (05_sync-spec.md §7)。

main(argv=None) -> int

何をするか: pipeline.run_cli("subscribers", make_fetch_and_transform, read_current, apply, argv) に委譲するだけ。 ロジックを持たないため、単体テストは「正しい引数で委譲したか」のみを monkeypatch で検証している(test_main_delegates_to_run_cli)。

4. rules.py との非対称性(意図的な設計判断)

コミットメッセージが「非対称」と明記している箇所を中心に、sync/rules.py(既存・未変更)と 横並びで確認した表。非対称であること自体は正しい設計判断だが、レビュー時は 「対称にすべきなのに漏れている」のか「意図的な非対称」なのかの見極めが要る。

観点rules.py(既存)subscribers.py(今回)非対称の理由
望ましい状態の引数 build_rules(rows, groups, pir_max_kbps) build_subscribers(rows, area_rows, groups) subscribersにPIR概念が無い
現状読取の対象 post_cli("show current.cfg") をパース get_flows([]) でNDJSON取得 rulesはCLIコンフィグ、subscribersはREST資源
適用手段 commit_config(PATCH→refresh→save、3ステップ) delete_flows → post_flows(直接、チャンク分割) flowsは即時反映APIでCLIコンフィグではない
変更(update)の表現 同名subportへの再投入で上書き(no行不要、QA#23) 削除→再登録が必須(更新APIが存在しない、一意制約) API仕様の違い(04_fx2-api §2.4)
適用順序 単一CLIテキストに追加/削除混在(順序は文字列内) 削除を先に全チャンク実行→追加を全チャンク実行 一意制約違反回避のため

5. apply() の処理フロー

apply(client, group_name, diff) │ ├─ diff.to_add と diff.to_delete が両方とも空か? │ └─ Yes → return (装置に一切触れない no-op) │ ├─ diff.to_delete を _CHUNK(1,000,000)件ずつ分割 │ └─ 各チャンクごとに client.delete_flows(flow_ids=[row[0] for row in chunk]) │ ※ 全delete完了 "before" 次のadd開始(先行完了、混在しない) │ ├─ diff.to_add を _CHUNK 件ずつ分割 │ └─ 各チャンクごとに client.post_flows(flows=[_row_to_flow(row) for row in chunk]) │ └─ commit_config は呼ばない(rulesと非対称) ※ どの段階でも Fx2Error は捕捉せずそのまま送出(fail-loud)。 途中で例外が飛べば、残りのチャンクは実行されない = 部分適用の可能性あり。 → 保存指紋は pipeline.run_sync 側で「全装置成功時のみ」更新されるため、 次周期に必ず再読取・再差分・再送される(自己修復、05_sync-spec §7)。
