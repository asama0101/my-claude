1. 全体像・依存関係

transform パッケージは DB/CSV から取得した生の文字列行を、FX2 に投入する「望ましい状態」(Rule/Subscriber)へ変換する純ロジック層。 外部I/O(SSH/SFTP/HTTP)を一切持たず、標準ライブラリのみに依存する(NFR-04)。 正典は docs/spec/03_data-spec.md。設計の詳細根拠は docs/superpowers/specs/2026-07-18-phase1-transform-diff-design.md(以下「design」、コード内コメントの P1-RUL-*・P1-SUB-*・P1-VAL-* はこの design 文書の受入基準ID)。

common.py ── 定数・型のみ(ロジックなし)。rules.py と subscribers.py の両方から import される │ │ SkipReason (enum, 5値) ─ TransformSummary (frozen dataclass) │ SERVICE_CODE_PATTERN / AREA_ID_PATTERN / CUSTOMER_ID_PATTERN │ SUBPORT_NAME_PATTERN / FLOW_ID_PATTERN │ new_skip_counts() / freeze_skip_counts() / finalize_summary() / build_area_group_index() │ ├──► rules.py build_rules(rule_rows, groups, pir_max_kbps) -> RulesResult │ 外部DB「ルール表」の4フィールド行 → Rule(subport相当) │ └──► subscribers.py build_subscribers(customer_rows, area_rows, groups) -> SubscribersResult 顧客CSV + エリア表 → ロンゲストマッチ → Subscriber(flow相当) 呼び出し元: sync/rules.py・sync/subscribers.py (フェーズ4, このレビュー対象の外) config.Group への依存: groups 引数のみ(area_ids: tuple[int])。逆方向の依存(config→transform)はない。

2. common.py — 共通定数・型

transform/common.py rules.py・subscribers.py の DRY 集約先(design P1-VAL-06)。ロジックは持たない。
class SkipReason(str, Enum)

平たく言うと: 行をスキップした理由を表す5値の列挙型(str継承なので辞書キーにそのまま使える)。

値: identifier_format / type_error / address_format / no_area / no_group

なぜ要るか: rules.py・subscribers.py 双方でスキップ理由の文字列表記がバラつくのを防ぐ。仕様書側の対応は §6 を参照。

@dataclass(frozen=True) class TransformSummary

平たく言うと: 変換1回分の結果集計。total_input・emitted・skipped・skipped_by_reason(理由別内訳)・clamped(pir上限クランプ件数)。

呼ばれ方: build_rules/build_subscribers の戻り値(RulesResult.summary/SubscribersResult.summary)に格納される。ログ出力はフェーズ4の責務(design §5)でここでは行わない。

SERVICE_CODE_PATTERN / AREA_ID_PATTERN / CUSTOMER_ID_PATTERN / SUBPORT_NAME_PATTERN / FLOW_ID_PATTERN

平たく言うと: 5つの識別子検証用コンパイル済み正規表現。厳密な対応は §5 の仕様対応表を参照。

new_skip_counts() -> dict[str, int]

平たく言うと: SkipReason 5キーすべてを0で初期化した dict を返す。

なぜ要るか: 各行の失敗時に該当キーを+1するだけで済み、テスト側で「他理由は不変」という assert が KeyError なく書ける、とコメントに明記あり。

freeze_skip_counts(skip_counts: dict) -> Mapping[str, int]

平たく言うと: 集計済み dict を types.MappingProxyType で読み取り専用ビューに包む。

注意点: コメントにある通り「値は参照元 dict のビュー」であり、呼び出し側が参照元をその後変更しない前提。finalize_summary 以外から直接呼ばれる想定は薄い。

finalize_summary(total_input, emitted, skip_counts, clamped=0) -> TransformSummary

平たく言うと: skipped = total_input - emitted を計算し、skip_counts を凍結して TransformSummary を組み立てる。rules.py・subscribers.py の両方が末尾でこれを呼ぶ。

前提条件(コメントに明記): skipped = total_input - emitted という差分計算は、config の area_ids が互いに素で単一グループ割当が保証されている(P1-CFG-01)ことに依存する。config 側の重複検証が壊れると、このサマリの不変条件も壊れる。

build_area_group_index(groups: Sequence[Group]) -> dict[str, str]

平たく言うと: poi_id(4桁ゼロ埋め文字列) → グループ名 の索引を構築する。config の area_ids(int)を f"{n:04d}" で正規化(QA #32)。

呼ばれ方: rules.py・subscribers.py の両方が build 冒頭で1回だけ呼ぶ(ループ内で毎回呼ばない=性能面の配慮)。

暗黙の前提: 「各 poi_id は高々1グループに対応する」。これは config 側で area_ids の重複が拒否される(P1-CFG-01、QA #33)ことに依存しており、この関数自体には重複防御コードが無い。config.py 側の検証実装をレビュー対象に含めるべき。

3. rules.py — ルール(subport)生成

transform/rules.py 仕様: 03_data-spec.md §2.2, §3.1-3.2, §5
@dataclass(frozen=True, slots=True) class Rule

フィールド: name: str(subport名, 12字固定) / pir_kbps: int(クランプ後) / fair_use_enabled: bool / group_name: str

def build_rules(rule_rows, groups, pir_max_kbps) -> RulesResult

平たく言うと: 「サービスコード, 地域エリアID, 上限帯域, 公平制御フラグ」の4フィールド行を1行ずつ検証し、通った行だけ Rule に変換する。docstring 通り6段階の検証パイプライン。

なぜ要るか: 外部DBの生データはそのまま装置には投入できない(識別子形式・型・グループ帰属を保証する必要がある、REQ-09/REQ-10)。

呼ばれ方: sync/rules.py(フェーズ4、本レビュー対象外)から、fetch 済みの生行を渡されて呼ばれる想定(design §2「境界」)。

処理フロー(1行あたり)

  1. arity フィールド数が4でなければ type_error でスキップ
  2. 識別子形式 isp_code/poi_id が正規表現不一致なら identifier_format でスキップ
  3. 型変換 pipe_band を int() 変換できなければ type_error でスキップ
  4. t/f判定 fair_use_flg が "t"/"f" 以外なら type_error でスキップ
  5. グループ割当 poi_id がどのグループにも属さなければ no_group でスキップ
  6. ルール名 = f"{isp_code}-{poi_id}" を組み立て、subport 形式を再検証(識別子形式の組み合わせで理論上常に成立するが DRY のため実施、と docstring に明記)
  7. クランプ pipe_band > pir_max_kbps なら pir_max_kbps にクランプし clamped をインクリメント(スキップではない)
  8. Rule を追加、emitted をインクリメント
境界値・注意点
  • pipe_band の下限チェックは無い(負値もそのまま通る)。仕様上「下限の検証は行わない」(QA #27)と明記されており意図的。テスト test_rul17_negative_pipe_band_no_lower_bound がこれをカバー。
  • クランプ境界は pir_max_kbps と等しい場合はクランプ**されない**(> であり >= ではない)。テスト test_rul05/06 が境界一致・境界未満を確認。
  • fair_use の "t"/"f" 以外の値(例: 大文字 "T"、空文字)はすべて type_error 扱い。真偽値表現の揺れを想定していない=psql の t/f 固定という前提に強く依存。
  • 同一 poi_id が複数グループに該当するケースへの防御はこの関数内にはない(common.py の項参照、config 側の検証に依存)。

4. subscribers.py — 加入者(flow)生成・ロンゲストマッチ

transform/subscribers.py 仕様: 03_data-spec.md §2.1, §2.3, §3.1, §3.3-3.4
@dataclass(frozen=True, slots=True) class Subscriber

フィールド: flow_id: str(加入者ID) / subport: str(所属ルール名) / address: str(UserPrefix原文のCIDR) / group_name: str

def _build_area_index(area_rows) -> dict[int, dict[int, str]]

平たく言うと: エリア表(poi_prefix, poi_id の2フィールド行)から {プレフィックス長: {正規化ネットワークアドレス(int): poi_id}} の二重索引を構築する。

なぜ要るか: 顧客ごとのロンゲストマッチをO(索引長数)で行うための前処理。build 毎に1回だけ構築し、顧客ループ内では再構築しない(コメントに明記、性能上の要件 design R8)。

境界値: poi_id が4桁形式でない行、poi_prefix がIPv6としてパースできない行は索引に**サイレントに**入らない(スキップカウントには計上されない = エリア表自体の完全性検証は fetch 層の責務、とコメントで明示)。同一 poi_id が複数プレフィックスに対応するのは正常(後勝ちで上書きではなく、プレフィックス長ごとに別バケット)。

def _truncate_to_length(addr_int: int, length: int) -> int

平たく言うと: 128bit整数のIPv6アドレスを上位 length ビットだけ残すビットマスク処理。ipaddress.IPv6Network オブジェクト生成を避けて整数演算のみで済ませている(性能配慮、NFR-01)。

def _longest_match(area_index, sorted_area_lengths, addr_int, client_len) -> str | None

平たく言うと: 顧客プレフィックス長**以下**のエリア長を長い順に走査し、最初に一致した poi_id を返す。一致なしは None。

境界値・注意: length > client_len の索引長はスキップ(顧客プレフィックスより長いエリアは照合対象外、03:97-99)。sorted_area_lengths は呼び出し側(build_subscribers)がループの外で1回だけソート済みで渡す前提=この関数自体はソートしない。

def build_subscribers(customer_rows, area_rows, groups) -> SubscribersResult

平たく言うと: 顧客CSV(isp_code, customer_no, user_prefix)の各行に対し、エリア表とのロンゲストマッチで所属エリアを決定し、加入者ID(flow_id)・所属ルール名(subport)を生成する。docstring 通り5段階の検証パイプライン。

呼ばれ方: sync/subscribers.py(フェーズ4、本レビュー対象外)から呼ばれる想定。

処理フロー(1顧客行あたり)

  1. arity フィールド数が3でなければ type_error でスキップ
  2. 識別子形式 isp_code/customer_no が正規表現不一致なら identifier_format でスキップ
  3. アドレス形式 user_prefix がIPv6としてパースできない、または prefixlen が {56, 64} 以外なら address_format でスキップ
  4. ロンゲストマッチ 一致するエリアが無ければ no_area でスキップ
  5. subport = f"{isp_code}-{poi_id}" を組み立て、形式を再検証(理論上常に成立)
  6. flow_id = f"{customer_no}@{user_prefix}@{subport}" を組み立て、文字集合・長さ(1-127)を検証。違反なら identifier_format でスキップ
  7. グループ割当 poi_id がどのグループにも属さなければ no_group でスキップ
  8. Subscriber を追加、emitted をインクリメント
rules.py との非対称点
subscribers には clamped に相当する概念が無い(常に0扱い、finalize_summary のデフォルト引数のまま呼ぶ)。クランプは帯域(pir)固有の概念のため。
flow_id 生成の暗黙前提
03_data-spec.md §3.1 の記述によれば、識別子の形式(サービスコード7字・地域エリアID4字・顧客ID最大15字)から flow_id は理論上必ず127字以内・許可文字種内に収まる設計。それでも実装は「常に満たすが検証は実施する」の方針でコード側の検証を省略していない(DRY方針とのトレードオフとしてレビュー観点になり得る=冗長ではなく防御的)。

6. SkipReason 対応関係

SkipReasonrules.py での発生条件subscribers.py での発生条件
identifier_format isp_code/poi_id の正規表現違反、生成した subport 名の形式違反 isp_code/customer_no の正規表現違反、生成した subport/flow_id の形式違反
