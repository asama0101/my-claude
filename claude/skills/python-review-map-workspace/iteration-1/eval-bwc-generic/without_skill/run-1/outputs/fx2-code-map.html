033-step commit プロトコル

subport 変更系（F-01〜F-04）は apply_subport_config() 1メソッドに集約されており、 以下の3ステップを順に実行する。仕様書対応: spec.html「subport 操作（登録・更新・削除）」 （1566〜1578行）／architecture.md「FX2 3-step Commit プロトコル」（51〜73行）。

STEP 1
PATCH /api/v1/config
CLI テキスト全文（text/plain）を候補設定領域（moff）へ投入。最大3回リトライ（_patch_config）。
→
STEP 2
POST /api/v1/cli
{"command": "refresh moff"} — 候補設定をランタイムに反映。最大3回リトライ（_cli_command）。
→
STEP 3
POST /api/v1/cli
{"command": "save moff"} — ランタイム設定を不揮発領域へ保存。最大3回リトライ（_cli_command）。
STEP 1 失敗時 — _discard() で discard moff を best-effort 送信し、FX2CommitError を送出。discard 自体の失敗は警告ログのみで無視する。
STEP 2 / 3 失敗時 — discard は送らない（PATCH は既に適用済みのため）。FX2PostCommitError を送出し、呼び出し元の subport_orchestrator._apply_with_rollback() がこれを捕捉して逆デルタ CLI を apply_subport_config() に再投入する（F-13 自動ロールバック）。ロールバック自体が失敗した場合は logging.critical で「manual recovery required」を記録し、例外を再送出する。
NOTE flow_id の追加・削除（apply_flow_ids / delete_flow_ids）は単発の POST／DELETE のみで、この3-step commit の対象外。flow_id_orchestrator.py の docstring に「flow_id は HTTP アトミック操作のため FX2 側ロールバック不要（F-14）」と明記されている。

09初見メモ・注意点

リトライ対象の非対称性

3-step commit のPATCH／refresh／saveだけが3回リトライされ、get_subports_raw・get_flow_ids_raw・apply_flow_ids・delete_flow_ids・show_versionは一発勝負（リトライなし）。spec.html の F-11「FX2 接続失敗時リトライ3回」がsubport commitパスのみを指すのか、全エンドポイントに及ぶ意図なのかはこのファイルからは判断できない。レビュー時に仕様側の意図を確認する価値がある。

_patch_config と _cli_command のコード重複

両者はほぼ同じリトライループ（ログ文言とペイロード形式だけが違う）。将来リトライ回数や待機戦略を変える際は2箇所同時に直す必要がある。

ssl=False は全装置に一律適用

_make_session() が証明書検証を無効化しているため、SEC-01の「自己署名証明書でも可」は満たすが、中間者攻撃への耐性はない。運用ネットワークの信頼境界（FX2までの経路）に依存した設計。

__init__.py に再エクスポートなし

全呼び出し元が from bwc_provisioner.fx2.client import FX2Client のようにフルパスで import している。パッケージ内に他モジュールが増えた場合の import パスに注意。

関連プロジェクトからの参照 shaper-db（別リポジトリ）の CLAUDE.md はこの fx2/client.py を「設計の参考にのみ用いる。コードは流用しない」と明示的に引用している。理由は依存方針の違い（shaper-db は標準ライブラリのみ、本モジュールは aiohttp 依存）。また本モジュールは POST /api/v1/cli のペイロードキーに "command" を使っているが、shaper-db 側の正式なFX2 API仕様（04_fx2-api.md）ではキー名が "cmd" である点が名指しで注意喚起されている。3-step commit（PATCH→refresh moff→save moff）という手順自体は shaper-db 側にも踏襲されている。
