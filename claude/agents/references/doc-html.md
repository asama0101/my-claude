# HTML 成果物の役割分担と流用（詳細リファレンス）

> doc-updater エージェント用のオンデマンド参照ファイル（エージェント定義ではない）。
> フォーマットに HTML を選んだ場合の、視覚デザインの主体・流用元ライブラリからの再利用・新デザインの申し送りを扱う。

---

## 役割分担（視覚デザインは frontend-design が主体）

フォーマットに HTML を選んだ場合、**視覚デザインと最終レビューの主体は doc-updater ではなく Main 側の `frontend-design` スキル**である。doc-updater はサブエージェントで `Skill` ツールを持たず、自力で `frontend-design` を起動できないため、役割を次のように分ける。

| 担当 | 範囲 |
|------|------|
| **Main（frontend-design スキル）** | 視覚デザイン（レイアウト・タイポグラフィ・配色・余白）と、提出前の敵対的クロスレビュー |
| **doc-updater** | ドキュメント構成・セクション設計・内容の正確性（ソース整合）・既存スタイルの踏襲／流用テンプレの適用 |

- **クロスレビュー（必須）**: HTML 成果物は提出前に独立サブエージェントで敵対的クロスレビュー→修正→再レビューを通す。ブラウザ不可環境では `node --check` ＋ JS ロジック追跡等の静的解析で代替する（グローバル CLAUDE.md 準拠）。

---

## 流用メカニズム（過去デザインを再利用する）

過去に作成した HTML デザインは流用元ライブラリ `~/.claude/assets/html-templates/` に蓄積される。カタログは同ディレクトリの `INDEX.md` が単一ソース。**HTML を書き始める前に、まず既存テンプレを流用できないか判断する。**

### 判断フロー

1. **カタログを確認**: `~/.claude/assets/html-templates/INDEX.md` を読み、要件（読者・トーン・レイアウト種別）に合うテンプレがあるか調べる。
2. **合うテンプレがある → 流用**: doc-updater が該当 `.html` を Read し、その**構造・CSS を土台に内容を差し込む**。この経路は `frontend-design` の起動不要・Skill 非依存で doc-updater 単独で完結する（プロジェクト固有の中身は捨て、骨格とスタイルだけ再利用する）。
3. **合うものが無い → 新デザイン**: doc-updater 単独では凝った視覚デザインシステムを作り込まない。既存スタイルの踏襲と読みやすさ優先の一貫スタイル（doc-writing.md 2-2）に留め、凝った視覚設計が要るなら **Main に `frontend-design` での新規デザインを申し送る**。完成後、再利用価値があれば下記の手順でライブラリに登録する。

### 登録（取り込み）

既存 HTML のライブラリ登録・カタログ追記は **`html-template-import` スキル**が担う（手作業でのコピーや INDEX.md 直接編集はしない）。

- ファイル名規約: `<内容>-<トーン>.html`（例 `report-dark.html`・`topology-light.html`）
- スキルが `assets/html-templates/` へコピーし、`INDEX.md` に `name / 特徴 / 用途 / path` を1行追記する
