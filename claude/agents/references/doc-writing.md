# ドキュメント執筆（Phase 2・詳細リファレンス）

> doc-updater エージェント用のオンデマンド参照ファイル（エージェント定義ではない）。
> [doc-design.md] の設計承認後に、承認された文書を実際に書くフェーズ。
> 書き終えたら整合確認（[doc-verify.md]）は doc-verifier が別コンテキストで行う。HTML は [doc-html.md]。

---

## 新規作成：Phase 2 — 作成

### 2-1. ソースコードを読んでから書く

ドキュメントを書く前に、記載する事実のソースを必ず確認する。

確認すべき典型的なソース：

```
定数・閾値     → 設定ファイル・コード内の定数定義
ログメッセージ → notify() / logger.xxx() の引数文字列
CLI オプション → argparse / click の定義
DB スキーマ   → DDL（CREATE TABLE）・マイグレーションファイル
設定値        → docker-compose.yml・Dockerfile・.env.example
処理フロー    → 実際のコードの実行順序
計算式        → ソースの数式をそのまま転記
```

**推測や記憶で書かない。すべてソースから取得する。**

### 2-2. 既存のスタイルを踏襲する

既存の HTML/CSS があれば参照してスタイルを統一する。なければ読みやすさ優先で一貫したスタイルを定め、全文書で揃える。HTML の場合の視覚デザインの主体と流用手順は [doc-html.md] を参照。

### 2-3. 作成順序の原則

- **index（目次・入口）は最後に作る** — 他文書が揃ってからリンクを張る
- **ストレス下で参照される文書（障害対応・バックアップ）は短く集中させる** — 読者がパニックの中でも使えることを優先
- 設計で承認された文書は連続して作成してよい（1文書ごとの承認は不要）

### 2-4. 読者UX（スキャナビリティ）を意識して書く

読者は全文を読まない。拾い読みで目的の情報へ到達できるかを優先する。

- **情報粒度を読者に合わせる**: 運用担当には手順を短く具体的に、開発担当には Why と背景を厚く。同じ事実でも読者で粒度を変える（doc-design.md 1-1 の読者定義に対応）。
- **スキャナビリティ**: 見出し階層で全体像を先に示す・各セクション冒頭に1〜2行の要約・長い手順は番号付きリスト・重要な閾値やコマンドは本文に埋めず目立たせる。
- **導線**: 「次に何を読むか」を文書間の関連リンクで示す（index の入口案内とは重複させない）。

## この工程でよくある失敗パターン

- **ソースを読まずに記憶で書く** → 定数・ログメッセージ・計算式が実際と異なる
- **読者が違う文書を同じ粒度で書く** → 運用担当には長すぎ、開発担当には短すぎになる
