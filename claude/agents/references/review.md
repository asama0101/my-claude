# コードレビュー共通手順（review-* 共通リファレンス）

> review-correctness / review-security / review-performance / review-test / review-maintainability の共通骨格。
> エージェント定義ではなくオンデマンド参照ファイル。各 reviewer はこの手順に従い、**自分のドメイン固有のチェックリスト差分のみ**を本文に持つ。

---

## 担当境界

各 reviewer は自分の担当次元のみに集中し、他次元は担当外（別の reviewer が担当）とする。担当外の指摘は出さない。

| reviewer | 担当次元 |
|----------|---------|
| correctness | バグ・論理エラー・エラーハンドリング・境界値・冪等性 |
| security | インジェクション・認証/認可・機密情報漏洩・パストラバーサル |
| performance | アルゴリズム計算量・メモリ効率・DB/I/O 最適化・並列処理 |
| test | テスト品質（カバレッジ・フィクスチャ・信頼性）＋仕様適合（不変条件・スキーマ） |
| maintainability | 命名・構造・複雑さ・DRY/YAGNI＋ドキュメント整合 |

---

## レビュープロセス

> **Bash は読み取り/検証専用**（`git diff`・`grep`・テスト実行等）。ファイル書込み・`sed -i`・`commit`・`git add` 等の変更操作は行わない——reviewer は所見を出す役であり、コードを変更しない。

1. `git diff --staged && git diff` で変更差分を取得（差分なしなら `git log --oneline -5` で直近コミットを対象にする）
2. 変更ファイル全体を Read して周辺コード・呼び出し元・対応テストを把握
3. 対象 repo の `CLAUDE.md` / 仕様書（spec / README / docs）/ 既存テスト・設定を Read し、**プロジェクト固有の規約・不変条件・要件**を把握
4. 自分のドメインのチェックリストを、上記プロジェクト固有ルールに照らして適用する

> チェックリストは言語・ドメインに依存しない汎用観点。プロジェクト固有の規約・閾値・要件は必ずリポジトリ側の `CLAUDE.md` / 仕様書 / 既存コードを真実源として優先する。

## 出力フォーマット

### 詳細ブロック（任意 — 指摘がある重大度のみ）

重大度ごとに、問題のある箇所を次の形式で列挙する。指摘が無ければ省略してよい。

```
[CRITICAL] <問題の概要>
File: path/to/file.py:行番号
問題: <具体的な説明>
修正: <修正方法>

  bad_code()   # BAD
  good_code()  # GOOD
```

### サマリー表（必須）

最後に必ず、担当次元名を冠したサマリーを出力する。

```
## <担当次元>レビューサマリー
| 重大度 | 件数 |
|--------|------|
| CRITICAL | N |
| HIGH     | N |
| MEDIUM   | N |
| LOW      | N |

判定: [承認 / 警告（要注意マージ） / ブロック（修正必須）]
```

> 各 reviewer は必要に応じて、この表の直後に固有の追加行（例: カバレッジ影響・ドキュメント更新要否）を足してよい。

### 判定（3択・必須）

- **承認** — CRITICAL・HIGH の問題なし
- **警告（要注意マージ）** — HIGH 以下の懸念はあるがマージ可能
- **ブロック（修正必須）** — CRITICAL または看過できない HIGH がある

**承認基準の既定**: CRITICAL・HIGH 問題なしで承認。
