# ドキュメント設計（Phase 1・詳細リファレンス）

> doc-updater エージェント用のオンデマンド参照ファイル（エージェント定義ではない）。
> 実装済みシステムに人間向けドキュメントを整備するときの **設計フェーズ**。
> 執筆は [doc-writing.md]、整合確認は [doc-verify.md]（doc-verifier が担当）、HTML は [doc-html.md] を参照。

---

## まず：新規作成か更新か判断する

| 状況 | モード |
|------|--------|
| ドキュメントがまだない・一から作り直したい | **新規作成**（本ファイルの Phase 1 → doc-writing.md → doc-verify.md） |
| 特定の文書を「最新にして」と言われた | **単一文書更新**（doc-verify.md の更新モード A） |
| 機能追加・修正後に「関連文書を洗い出して」と言われた | **影響範囲洗い出し更新**（doc-verify.md の更新モード B） |

---

## 新規作成：Phase 1 — 設計

設計が固まらないうちに書き始めると、後で大量の書き直しが発生する。Phase 1 は手を動かさず対話だけで進める。

### 1-1. 読者を明確にする（最初の質問）

最初に必ず「誰が読むか」を確認する。読者が変わると必要な文書の種類も粒度も変わる。

典型的な読者：
- **承認者・管理者**: システムが何をするか・SLA・責任範囲を把握したい
- **運用担当者**: アラート対応・デプロイ・バックアップを手順書だけで完結させたい
- **開発担当者**: なぜこの設計か・データフロー・DB スキーマを理解したい

読者が複数いる場合、それぞれの「知りたいこと」を分けて整理する。

### 1-2. 既存ドキュメントとの役割分担を決める

CLAUDE.md・README・仕様書など既存ドキュメントがあれば内容を確認し、新しいドキュメントとの重複を避ける。**「人間向けドキュメントは CLAUDE.md とは独立して全部書き直す」か「差分だけ書く」かを明示的に決める。**

### 1-3. フォーマットを決める

- **HTML**: 視覚的な整形が必要・既存 HTML ドキュメントがある場合（→ 役割分担と流用は [doc-html.md]）
- **Markdown**: 更新コストを下げたい・Git で diff を見たい

### 1-4. 文書候補を列挙する

読者別に「何が必要か」を挙げる。この時点では多めに挙げてよい。

### 1-5. 批判的レビュー（必須・省略しない）

文書一覧が揃ったら、以下の観点で問題がないか確認する。**ここで出てきた抜けや矛盾を解消してから承認を求める。**

| 観点 | 確認すること |
|------|------------|
| 維持コスト | 文書数が多すぎないか。コード変更のたびに複数文書を更新する羽目にならないか |
| 使用者不明 | 「誰が・いつ使うか」が不明確な文書はないか。誰も読まない文書は削る |
| 統合可能性 | 同じ読者が読む文書は1本にまとめられないか |
| 入口の欠落 | 複数文書がある場合、何から読めばよいかの案内（index）はあるか |
| インターフェース仕様の欠落 | CLI の引数・入出力形式・命名規則が書かれるか |
| ログ・アラート対応の欠落 | 運用担当が「このログは正常か異常か」を判断できる文書があるか |
| 責任分界点の欠落 | このシステムが保証すること・保証しないことが明記されるか |
| セキュリティの欠落 | 認証情報の管理方針・変更手順が書かれるか |

### 1-6. 各文書の記載内容を詳細化し、承認を得る

各文書について「何を・どのくらいの粒度で書くか」の箇条書きを作り、ユーザーに確認する。**設計の承認を得てから執筆（doc-writing.md）に進む。**

---

## この工程でよくある失敗パターン

- **設計を承認前に書き始める** → 方針変更で大量の書き直しが発生する
- **文書を増やしすぎる** → 更新コストが高くなり放置される
- **設計背景を What（何をするか）から書く** → Why（なぜそうするか）が伝わらず反復修正が発生する。設計理由・判断背景を書くときは必ず Why を先に書き、What は補足に留める
