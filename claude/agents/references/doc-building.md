# 実装ドキュメント整備パターン（詳細リファレンス）

> doc-updater エージェント用のオンデマンド参照ファイル（エージェント定義ではない）。
> 実装済みシステムに対して人間向けドキュメントを **設計 → 作成 → 整合確認** の順で整備する手順。

---

## まず：新規作成か更新か判断する

| 状況 | モード |
|------|--------|
| ドキュメントがまだない・一から作り直したい | **新規作成**（Phase 1 → 2 → 3） |
| 特定の文書を「最新にして」と言われた | **単一文書更新**（更新モード A） |
| 機能追加・修正後に「関連文書を洗い出して」と言われた | **影響範囲洗い出し更新**（更新モード B） |

---

## 新規作成：Phase 1 — 設計

設計が固まらないうちに書き始めると、後で大量の書き直しが発生する。Phase 1 は手を動かさず対話だけで進める。

### 1-1. 読者を明確にする（最初の質問）

最初に必ず「誰が読むか」を確認する。読者が変わると必要な文書の種類も粒度も変わる。

典型的な読者：
- **承認者・管理者**: システムが何をするか・SLA・責任範囲を把握したい
- **運用担当者**: アラート対応・デプロイ・バックアップを手順書だけで完結させたい
- **開発担当者**: なぜこの設計か・データフロー・DB スキーマを理解したい

読者が複数いる場合、それぞれの「知りたいこと」を分けて整理する。

### 1-2. 既存ドキュメントとの役割分担を決める

CLAUDE.md・README・仕様書など既存ドキュメントがあれば内容を確認し、新しいドキュメントとの重複を避ける。**「人間向けドキュメントは CLAUDE.md とは独立して全部書き直す」か「差分だけ書く」かを明示的に決める。**

### 1-3. フォーマットを決める

- **HTML**: 視覚的な整形が必要・既存 HTML ドキュメントがある場合
- **Markdown**: 更新コストを下げたい・Git で diff を見たい

### 1-4. 文書候補を列挙する

読者別に「何が必要か」を挙げる。この時点では多めに挙げてよい。

### 1-5. 批判的レビュー（必須・省略しない）

文書一覧が揃ったら、以下の観点で問題がないか確認する。**ここで出てきた抜けや矛盾を解消してから承認を求める。**

| 観点 | 確認すること |
|------|------------|
| 維持コスト | 文書数が多すぎないか。コード変更のたびに複数文書を更新する羽目にならないか |
| 使用者不明 | 「誰が・いつ使うか」が不明確な文書はないか。誰も読まない文書は削る |
| 統合可能性 | 同じ読者が読む文書は1本にまとめられないか |
| 入口の欠落 | 複数文書がある場合、何から読めばよいかの案内（index）はあるか |
| インターフェース仕様の欠落 | CLI の引数・入出力形式・命名規則が書かれるか |
| ログ・アラート対応の欠落 | 運用担当が「このログは正常か異常か」を判断できる文書があるか |
| 責任分界点の欠落 | このシステムが保証すること・保証しないことが明記されるか |
| セキュリティの欠落 | 認証情報の管理方針・変更手順が書かれるか |

### 1-6. 各文書の記載内容を詳細化し、承認を得る

各文書について「何を・どのくらいの粒度で書くか」の箇条書きを作り、ユーザーに確認する。**設計の承認を得てから Phase 2 に進む。**

---

## 新規作成：Phase 2 — 作成

### 2-1. ソースコードを読んでから書く

ドキュメントを書く前に、記載する事実のソースを必ず確認する。

確認すべき典型的なソース：

```
定数・閾値     → 設定ファイル・コード内の定数定義
ログメッセージ → notify() / logger.xxx() の引数文字列
CLI オプション → argparse / click の定義
DB スキーマ   → DDL（CREATE TABLE）・マイグレーションファイル
設定値        → docker-compose.yml・Dockerfile・.env.example
処理フロー    → 実際のコードの実行順序
計算式        → ソースの数式をそのまま転記
```

**推測や記憶で書かない。すべてソースから取得する。**

### 2-2. 既存のスタイルを踏襲する

既存の HTML/CSS があれば参照してスタイルを統一する。なければ読みやすさ優先で一貫したスタイルを定め、全文書で揃える。

### 2-3. 作成順序の原則

- **index（目次・入口）は最後に作る** — 他文書が揃ってからリンクを張る
- **ストレス下で参照される文書（障害対応・バックアップ）は短く集中させる** — 読者がパニックの中でも使えることを優先
- 設計で承認された文書は連続して作成してよい（1文書ごとの承認は不要）

---

## 新規作成：Phase 3 — 整合確認（必須・省略しない）

文書を書いた後、ソースコードと1件ずつ照合する。「たぶん合ってる」では不十分。

### 3-1. 確認項目を抽出する

各文書から「ソースコードで検証できる事実」を列挙する：

| カテゴリ | 例 |
|---------|-----|
| 定数・閾値 | SLA 秒数・タイムアウト・バッファ・バケット幅 |
| ログメッセージ | `notify(level, f'...')` の文字列フォーマット |
| CLI オプション | `add_argument()` の名前・型・ヘルプ |
| DB スキーマ | カラム名・型・PK・インデックス・圧縮/保持ポリシー |
| 設定値 | 環境変数名・ポート番号・ボリュームパス |
| 処理フロー | コードの実行順序と文書記述の一致 |
| 計算式 | 数値の算出ロジック（誤差・丸め含む） |

### 3-2. grep で照合する

```bash
# 定数・閾値の確認
grep -n "SLA\|TIMEOUT\|INTERVAL\|LIMIT" src/config.py

# ログメッセージの確認
grep -n "notify\|logger\." src/main.py

# CLI オプションの確認
grep -n "add_argument\|option\|flag" src/cli.py
```

### 3-3. 結果を表形式で報告する

各項目について ✓（一致）または 不一致（具体的に何が違うか）で報告し、不一致は即座に修正する。

## 更新モード A — 単一文書の更新

「この文書を最新にして」と言われた場合。

1. **文書の記載内容を把握する** — 変更が必要な箇所を特定するため文書全体を読む
2. **対応するソースを確認する** — 文書内のすべての事実（定数・コマンド・スキーマ等）をソースと照合する
3. **差分のある箇所だけ更新する** — 関係ない箇所は触らない
4. **更新箇所の整合確認** — 更新した箇所について Phase 3 と同じ照合を実施する

---

## 更新モード B — 影響範囲洗い出し更新

「機能追加したのでドキュメントを更新したい」と言われた場合。

### B-1. 変更内容を把握する

`git diff`・PR の説明・変更の概要から「何が変わったか」を確認する。

### B-2. 影響する文書を特定する

| 変更の種類 | 影響する文書 |
|-----------|------------|
| CLI オプションの追加・変更 | 設計文書（インターフェース仕様）・運用手順書 |
| DB スキーマ変更（カラム追加等） | 設計文書（DB スキーマ） |
| ログメッセージの変更 | 障害対応手順書（ログリファレンス） |
| 設定値・環境変数の変更 | 運用手順書（環境要件）・デプロイ手順 |
| 処理フローの変更 | 設計文書（フロー図）・システム概要 |
| SLA・閾値の変更 | システム概要・障害対応手順書 |
| 新機能追加 | 設計文書・インターフェース仕様・index（案内の更新） |

### B-3. 影響文書のみ更新する

特定した文書について、変更後のソースを読んで更新する。更新後は Phase 3 の整合確認を実施する。

---

## よくある失敗パターン

- **設計を承認前に書き始める** → 方針変更で大量の書き直しが発生する
- **ソースを読まずに記憶で書く** → 定数・ログメッセージ・計算式が実際と異なる
- **整合確認を省略する** → 文書が陳腐化しても気づかない
- **文書を増やしすぎる** → 更新コストが高くなり放置される
- **読者が違う文書を同じ粒度で書く** → 運用担当には長すぎ、開発担当には短すぎになる
- **設計背景を What（何をするか）から書く** → Why（なぜそうするか）が伝わらず反復修正が発生する。設計理由・判断背景を書くときは必ず Why を先に書き、What は補足に留める
