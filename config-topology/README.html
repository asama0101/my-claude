config-topology

ネットワーク機器の running-config（テキスト）から、機器間の接続関係を自動推論し、 レイヤー別 YAML（正本）を経てインタラクティブな HTML 構成図を生成する Claude Code スキル。

Cisco IOS / IOS-XE Juniper JunOS (set) Python 3 + PyYAML 自己完結 HTML 出力 決定的・再現可能

概要

複数機器の Config を束ねて読み込み、IP / サブネット一致で機器間リンクを自動推論します。 中間表現は人手で編集・補正できるレイヤー別 YAMLで、編集して再描画する round-trip が可能です。 最終成果物は外部依存ゼロ・file:// で開ける単一 HTML の構成図で、レイヤー別ビューの切替・検索・ズーム/パンに対応します。

Excel 手順書のレビューやパラメータ入力ではなく、Config（テキスト）からのネットワーク図化が対象です。

パイプライン

inbox/*.{cfg,conf,txt} │ scripts/parse_configs.py ベンダー自動判定 → 正規化モデル(Device) ▼ │ scripts/build_topology.py IP/サブネット一致でリンク・セグメント推論、BGP対向解決 ▼ topology/ (レイヤー別YAML正本) ← 中間表現（正確・人手編集可・round-trip） │ scripts/topology_io.py dump/load（YAML ⇄ topology dict・参照整合検証） │ scripts/render_topology.py ▼ topology.html SVG + バニラJS（レイヤー別ビュー / 検索 / ズーム・パン）

メモリ上の topology dict と層別 YAML を topology_io.py が相互変換します。 パース・結線推論・描画の各層は単一責務で、境界は層別 YAML です。

主な機能

マルチベンダー・パース

Cisco IOS/IOS-XE running-config と Juniper JunOS（set 形式）を自動判定。複数機器を一括処理。 ベンダーはプラグイン方式で追加可能。

結線の自動推論

各 IF の IP/サブネットから結線を推論。2 機器=リンク、3 機器以上=共有セグメント、単独=スタブ。 BGP は neighbor_ip で対向解決し ebgp/ibgp を判定。

レイヤー別ビュー

HTML 上部タブで Physical（L1）/ L3（サブネットでグルーピング）/ プロトコル別（BGP・OSPF …）を切替。 ビューごとに配置が変わり、機器数が多くても読みやすい。決定的 force-directed レイアウト＋動的キャンバス（〜150 台目安）。

編集可能な YAML 正本

中間表現はレイヤー別 YAML。人手で補正・注記して再描画できる。読込時に ID 参照整合を検証し、 壊れた参照はファイル・フィールド・値を示してエラー。

使い方（Claude スキルとして起動）

本ツールは Claude Code のスキルです。Claude が SKILL.md に従って各スクリプトを実行します。 人が手動で CLI を叩く前提ではありません——Claude に自然言語で依頼して起動します。

起動の合図（トリガー）

inbox/ に Config（.cfg / .conf / .txt）を置く、または次のように Claude に依頼します：

「このconfigから構成図を作って」／「トポロジー図を描いて」／「ネットワーク図にして」／ 「configから結線を起こして」／「機器の接続関係を図にして」

Claude が行う手順（スキル内部のフロー）

  1. 収集・判定：inbox/ または指定パスの Config を集め、ベンダーを自動判定。
  2. build：結線推論してレイヤー別 YAML 正本を生成。
  3. 確認：YAML を目視し（人手補正も可）、参照整合を検証。
  4. render：YAML から自己完結 HTML 構成図を生成。
  5. クロスレビュー：成果物を点検してから提示。

上記 2・4 で実行されるコマンド（Claude が内部で実行する）：

# build: Config → レイヤー別YAML（-o は出力ディレクトリ、既定 topology/）
python3 scripts/build_topology.py <configs...> -o topology/

# render: 層別YAMLディレクトリ → 自己完結HTML（参照整合を検証して dict 復元）
python3 scripts/render_topology.py topology/ -o topology.html

生成 HTML：ビュー切替タブ・検索ボックス・ホイールズーム・ドラッグパン・F=全体表示・Esc=リセット。

実行環境（依存）

Claude の実行環境に Python 3 と PyYAML が必要です（Claude は python3 でスクリプトを実行）。 多くの環境では PyYAML は既に利用可能です。未導入の環境のみ、一度だけ次を実行します：

pip install -r requirements.txt
# 隔離したい場合: python3 -m venv .venv && .venv/bin/pip install -r requirements.txt

レイヤー別 YAML レイアウト（正本）

出力ディレクトリ（既定 topology/）に層別ファイルを配置します。

topology/ _meta.yaml # schema_version: "1.0", title, generated_from devices.yaml # devices[] + interfaces[] ← 全層が ID 参照する基盤 physical.yaml # links[] + segments[] routing.bgp.yaml # bgp[] （空プロトコルは書き出さない） routing.ospf.yaml # ospf[] routing.static.yaml # static[]
  • devices と interfaces は同居：links / routing が ID を外部キー参照するため基盤に集約。
  • routing はプロトコルごとに 1 ファイル。新プロトコルを足すと routing.<proto>.yaml が自動で増える。
  • 直列化は yaml.safe_dump(sort_keys=True) で決定的。読込は yaml.safe_load のみ（任意オブジェクト復元を禁止）。
機密情報の注意：config の interface description 等に誤って community 文字列やパスワードが 書かれている場合、その値はそのまま YAML・HTML に出力されます（パーサは password/secret/snmp community 行自体は読みません）。生成物の共有・保存時は取り扱いに注意してください。

拡張ポイント

追加したいもの方法
新ベンダー（NX-OS / Arista 等）scripts/parsers/<vendor>.py に detect/parse を実装し registry に登録。build / render は変更不要。
新プロトコル / レイヤーtopology の routing に新キー（→ routing.<proto>.yaml 層が増え、ビュー/トグルも自動生成）、または devices[].sections に汎用テーブル。
新しい結線の由来links[].kind に新値（例 CDP/LLDP 由来）。

スコープ（v1）

対象：機器・IF・IP・サブネット結線（コア）＋ ルーティング（BGP / OSPF / static）。

スコープ外（将来拡張）：VLAN/L2・SVI、冗長（HSRP/VRRP・LAG）、CDP/LLDP 由来リンク、 NX-OS/Arista 等の追加ベンダー、Mermaid/Graphviz 併出力、経路シミュレーション等の下流ツール （ロードマップ docs/plans/2026-06-02-config-topology-roadmap.md 参照）。

テスト

cd config-topology && python3 -m pytest tests/ -q   # ユニットテスト（パーサ/推論/I/O/描画）
# E2E:
python3 scripts/build_topology.py examples/configs/sample-ios-r1.cfg examples/configs/sample-junos-r2.conf -o /tmp/topo
python3 scripts/render_topology.py /tmp/topo -o /tmp/t.html   # /tmp/t.html を file:// で開く

examples/topology/（層別 YAML）は 2 つのサンプル config から生成される期待出力（ゴールデン）です。
