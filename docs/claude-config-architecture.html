この文書は、~/.claude/ の Claude Code 設定がどんな部品で構成され、なぜ TDD 品質ゲートが必要かを、初読者向けに解説する。読む順は §1 四層構造 → §2 入口の分岐 → §3 参照関係図 → §4 実行役 → §5 ゲート詳細 が分かりやすい。

第一条

四層の役割分担 — 憲法・工程書・実行役・正典

この設定は「誰が・何を・どの順で」を4つの層に分けて持つ。各層は自分の責務だけを記述し、詳細は下の層へ委譲する。同じ定義が二箇所に現れないことが、この構成の背骨。

憲 法

~/.claude/CLAUDE.md

理念・行動規範・確認と禁止・作業の進め方（規模で工程の深さを変える入口分岐）・エージェント/スキル索引。手順の詳細は持たず、正典への委譲だけを書く（例: レビュー体制の詳細は工程書側の正典 gates.md に委譲）。

工 程 書

skills（1本 + プラグイン）

tdd-gates＝TDD×10ゲートのオーケストレータ（フロー図の正典）。工程の順序と強制力はここが持つ。

実 行 役

agents（12本）

planner／tdd-generator／tdd-evaluator／5次元 review-*／dev-python／doc-updater／doc-verifier／trivial-executor。各自は差分となる責務だけを持ち、共通手順は references を読む。

正 典

references（2 系統）

agents/references/（11本＝doc-* 5・python-* 4・planner・review）＝深い作法・共通骨格。skills/tdd-gates/references/＝ゲート定義・採点基準・言語プロファイル。定義の「単一の真実源」。

単一ソース規約 — 「その定義はどこが正か」を一箇所に固定する
  • ゲート定義・Critical 条件 → skills/tdd-gates/references/gates.md（CLAUDE.md からも正典委譲）
  • 採点閾値・スコアカード形式 → skills/tdd-gates/references/scoring.md
  • ゲート順序・フロー図 → skills/tdd-gates/SKILL.md（gates.md は重複掲載しない）
  • 行数・ネスト深さ等の数値基準 → review-maintainability（dev-python・planner・gates.md が参照）
  • レビューの共通手順・出力形式 → agents/references/review.md（5本の reviewer が共有）
第二条

作業の進め方 — 規模で工程の深さが決まる

変更規模で最初に分岐し、規模で変えるのは「委任するか否か」ではなくパイプラインの深さ。迷ったら substantial（安全側）に倒す。以降で頻出する「台帳」は、タスクごとに作る監査記録ファイル .tdd-gates/ledger-<slug>.md（詳細は末尾の用語集）。

入口区分定義工程（→ 呼ばれる skill / agent）
trivial 数行・既存パターン踏襲・テスト不要（設定/ドキュメント/コメント/誤記） trivial-executor（haiku）へ一括委任。Plan モード・reviewer 群は省略可
small trivial 超だが substantial 未満（差分小・既存テストあり・公開インターフェース不変） tdd-gates: Gate 1→4→5（Gate 3/8 の要否は Gate 1 の判定次第。判定根拠は台帳に記録）
substantial 新規ロジック・複数ファイル横断・公開インターフェース変更・非自明なバグ修正 Plan モード → tdd-gates フル10ゲート。採点は実装者と別コンテキストで自己承認を排除
レビュー単独 フル工程は不要だがレビューだけ独立して欲しい Main が review-* を並列起動 → tdd-evaluator が集約採点
第三条

参照関係図 — 誰が誰を起動し、誰が何を読むか

図は2枚に分けた。図①は起動フロー（誰が誰を起動するか）、図②は参照フロー（誰がどの正典を読むか）。ノードは上から下へ「憲法 → 工程書 → 実行役 → 正典」。G1・G2 等は §5 の 10 ゲート番号。

図① 起動フロー — 誰が誰を起動するか（朱＝Main 起動・実線＝skill 起動）

朱線＝Main による起動（CLAUDE.md の作業の進め方に従う） 実線＝skill による起動（ゲート番号つき） 破線＝行動規則／スキルの参照

図② 参照フロー — 誰がどの正典を読むか（破線＝Read・点線＝所見の受け渡し）

破線＝正典の参照（Read） 点線＝レビュー所見の受け渡し 破線枠＝ディレクトリのまとまり 白チップ＝個別ファイル
pytest は意図的に「二枚」ある skills/tdd-gates/references/profiles/pytest.md は薄い「ゲート用グルー」（パス→テスト種別・実行コマンド・合格ログ判定）。深い作法は agents/references/python-testing.md が単一ソースで、グルー側が委譲する。tdd-generator は両方、dev-python は後者のみを読む。
新言語は _template.md から拡張する 言語プロファイルは現状 pytest のみ。未対応言語では profiles/_template.md から起草し、テスト実行コマンドと合格ログ形式をユーザー承認後にゲート開始する（pytest 前提で進めない）。
Gate 7 だけはエージェントでなくスキルが担う UI/UX レビュー（Gate 7）は frontend-design スキル＋独立サブエージェントの敵対的クロスレビューで行う（図では小チップで表示）。
tdd-gates を通らない変更でも doc 同期の委任は働く tdd-gates を通らない変更でも、CLAUDE.md のタスク手順が同じく doc-updater への委任を課す（ユーザー受け入れ OK 後に実行）。
数値基準の横断参照は図に引いていない 関数50行・ファイル800行・ネスト4 等の数値基準の正典は review-maintainability で、dev-python・planner・gates.md が参照する（§1 の単一ソース規約）。agent 間の横断エッジは図が混むため、この脚注で代える。

第五条

TDD 10ゲート検査ライン — 順序そのものが防衛

「動くように見えて中身がバグだらけ」を防ぐには、工程を飛ばせないことが第一の防衛になる。大半のゲートは Critical 条件を持ち、1つでも未達なら他が満点でも即 FAIL（Gate 7・10 は非該当時にゲートごとスキップされ Critical を持たない）。番号が朱のゲートは Critical が特に要となる工程。

1

コンテキスト分析・要件整理

CONTEXTsmall でも必須

依頼＋既存コード・ドメイン・制約を分析し、要件を抽出・構造化して抜け漏れなく確定する。変更影響範囲の整理・スコープ単一性の確認もここ。small の簡略可否（Gate 3/8 省略等）はこの Gate 1 で判定し台帳に記録する。

planner 起草tdd-evaluator 採点
Critical: 要件の網羅性 — evaluator が抜け漏れ候補を敵対的に列挙し、非自明な欠落はユーザー確認（Stop & Ask）で解消。コンテキスト把握が的確・スコープが単一。
2

受入基準・テスト設計

SPEC & TEST DESIGNsubstantial 必須

確定した要件を検証可能な受入基準に言語化し、テストシナリオ（正常・異常・境界）へ落とす。想定不具合を列挙し、3層（unit／integration／e2e）のどこで守るか・E2E 要否を判断する。

planner 起草tdd-evaluator 採点
Critical: 受入基準が検証可能で、要件 ↔ 受入基準 ↔ テストシナリオの対応に漏れがない（トレーサビリティ）。
3

事前レビュー

PRE-REVIEWsubstantial 必須

テストを書く前に、既存コード・DB・セキュリティ（権限漏れ含む）・設計をレビューし、業務ロジックが unit テスト可能に分離されているかを検査する。この段階はテスト・差分がまだ無く review-* の材料が乏しいため、evaluator がレビュアーを兼ねる（実差分への多次元レビューは Gate 8）。セキュリティ/性能に敏感な計画のときだけ該当 reviewer を条件付き並列（最大2本）。

tdd-evaluator 単独（既定）＋条件付き security／performance（最大2）
Critical: 重大セキュリティ欠陥・設計破綻がない。
4

RED — 失敗するテストを先に書く

REDsmall でも必須

実装コードは一切書かず、テストだけを書いて実行する。この時点のテスト本文（パス＋内容またはハッシュ）を台帳に固定記録し、後のゲートで照合する。実失敗ログの提出はいかなる場合も省略できない。

tdd-generator
Critical: 実行ログで「実際に・意味のある理由で」失敗していること。assert False 等の無条件失敗、collected 0 items、import ERROR で失敗を装うのは無効。
5

GREEN — テストを通す最小限の実装

GREENsmall は G4–5 中心の簡略可

Gate 4 と同一の Generator を SendMessage で継続し、最小実装で全テストを通す。evaluator が git diff で「テストが検証しない実装分岐（過剰実装）」と「テストの書き換え」を照合する。

tdd-generator（同一継続）
Critical: 全テスト通過の実行ログ＋Gate 4 で固定したテスト本文と差分がないこと。
6

REFACTOR — 振る舞い不変で整理

REFACTORsubstantial 必須

重複除去・命名改善のみ。evaluator が git diff でテストファイル不変と「既存テストから到達不能な新機能の混入なし」を確認する（差分カバレッジで裏取り推奨）。

tdd-generator（同一継続）
Critical: 振る舞い不変 — テスト不変のまま全テスト通過。
7

UI/UX レビュー

UI/UXe2e／ビュー層変更時（条件付き）

AI 臭さの排除・レイアウト・導線を検査。テスト種別が unit と判定されていても、対象にテンプレート/ルーティング/ビュー層の変更が含まれていれば必須化され、その判定による回避はできない。Critical 項目はない（非 E2E かつビュー層変更なしの場合はゲートごとスキップ）。

frontend-design スキル敵対的クロスレビュー
8

差し戻し判定 — 最終ゲート

FINAL REVIEWsubstantial 必須

変更差分全体を5次元並列でレビューし、evaluator が集約採点してマージ可否を判定する。ここで review-test が加わり、偽装テスト（assert なし・常に真・実装の写経）の検出が Critical になる。

correctnesssecurityperformancemaintainabilitytest→ tdd-evaluator 集約
Critical: 仕様不適合ゼロ・既存回帰ゼロ・偽装テスト検出ゼロ。
✓
ユーザー受け入れ確認 — Gate 8 PASS 後・Gate 9/10 の前に必ず挟む。例外として、ユーザーが既に承認済みの複数マイルストーン計画で clean PASS のときは、ここで停止せず進捗報告のみで次のマイルストーンへ自動的に進む。この時 evaluator が受け入れチェックリスト（受入基準＋5次元＋例外処理・権限漏れ・変更影響範囲）を生成して台帳に残し、手動確認を再現可能にする。NG なら修正へ差し戻し、ドキュメントは更新しない。
9

CI 品質ゲート整備

CI GATECI 運用時のみ

CI が必須ステージ（lint・typecheck・build・unit・integration・主要 E2E）を自動実行するよう整備・検証する。プロファイル駆動 — 具体コマンドは言語プロファイルが供給する。ワークフロー実体は各プロジェクトに生成する（GitHub Actions 既定）。「グリーン」は push 後の非同期事象のため、本ゲートは定義の存在・被覆をゲートする。

doc-updater 生成tdd-evaluator 採点
Critical: 必須ステージ（lint・typecheck・build・unit・integration・主要E2E）がすべて CI 定義に含まれている（preview デプロイは任意のため対象外）。
10

ドキュメント同期

DOC SYNCdoc 影響なしならスキップ

README・ガイド・コードマップ・API 仕様を変更内容に同期する。Critical 項目はない（doc 影響がなければゲートごとスキップ）。

doc-updater
第六条

判定と検印 — スコア率を Critical が上書きする

各ゲートの採点は tdd-evaluator が scoring.md 準拠で行う。重要なのは、スコア率がどれだけ高くても Critical 1件で即 FAIL になること — 「点数を稼いで欠陥をすり抜ける」経路を断つ。

合PASS

PASS

スコア率 80% 以上かつ Critical 全達成。台帳の各 PASS には evaluator 自身が再実行した検証コマンドの出力行の添付が必須 — 実行痕跡のない PASS は無効。

条CONDITIONAL

CONDITIONAL

スコア率 60–79%。差し戻して再評価するが上限 2 回。再評価では前回スコアカードの指摘を引用し、台帳に記録した再評価回数と整合確認 — 引用なし・カウンタ不整合は上限到達扱い（無限往復の防止）。

否FAIL

FAIL

スコア率 60% 未満、または Critical 未達（他が満点でも即 FAIL）、または CONDITIONAL 2 回超。工程を差し戻す。

第七条

破り方と防ぎ方 — AI 駆動開発の品質問題への13の防衛

この仕組みは「AI 実装者がどうサボり・どう取り繕うか」を先回りして設計されている。各カードは 破り方（起きうる品質問題）→ それを構造的に潰す仕組み の対。

要件の抜け漏れ — 存在しない要件は誰も検査しない

Gate 1 で evaluator が要件集合の完全性を敵対的に点検し、抜け漏れ候補（未記載・暗黙要件・未対応エッジ）を列挙。真実源はユーザーにあるため、非自明な欠落は自動 FAIL せず Stop & Ask で確認して要件へ反映する。

gates.md（Gate 1 Critical）

自己承認 — 実装者が自分で合格を出す

Generator（実装）と Evaluator（採点）を必ず別サブエージェント・別コンテキストで起動。reviewer の所見は scratchpad 経由で evaluator が直接 Read し、Main が要約・改変する経路を外す（当事者が critical 所見を軟化させる余地を消す）。

gates.md / SKILL.md / tdd-evaluator.md

口先報告 — 「テストは通るはず」

証拠主義: RED / GREEN は実行コマンド＋出力スニペットを台帳に残す。ログのない主張は該当項目 0 点。

SKILL.md / scoring.md

ログ捏造 — それらしい出力を貼る

evaluator は貼られたログ・差分を信頼せず、Critical 項目は自分で Bash 再実行して再現した出力のみを証拠にする。再現不能・テストパス不一致は 0 点。

tdd-evaluator.md / SKILL.md

採点者の不正 — 通したいがためにテストを直す

evaluator は Write/Edit ツールを持たず、Bash でも sed -i・リダイレクト・mv/rm・git add/commit を禁止。読取と検証実行しかできない。

tdd-evaluator.md（frontmatter 制約）

RED 偽装 — 形だけ失敗させる

assert False・トートロジー・例外 raise だけの無条件失敗は Critical 未達。collected 0 items や import ERROR で「失敗した」ことにするのも無効。

gates.md / profiles/pytest.md

assert 骨抜き — GREEN でテストを弱める

Gate 4 時点のテスト本文を台帳に固定記録し、Gate 5・6 で照合。差分があれば Critical 未達。

SKILL.md / gates.md / tdd-evaluator.md

REFACTOR への新機能混入

Gate 6 の Critical は振る舞い不変。evaluator が git diff でテスト不変と、既存テストから到達不能な新規の条件分岐・機能追加がないことを確認する。

gates.md

過剰実装 — テストにない機能の作り込み

Gate 5 で evaluator が git diff にテストが検証しない実装分岐がないか（最小性）を確認する。

gates.md / tdd-evaluator.md

形骸化テスト — 常に PASS する飾り

Gate 8 に偽装テスト検出の Critical: assert なし・常に真・実装の写経を検出したら即 FAIL。

gates.md

スコア稼ぎ — 点数で欠陥を薄める

Critical 即 FAIL がスコア率を上書きする。1 件でも未達なら他が満点でも FAIL。

scoring.md

無限往復 — CONDITIONAL で粘る

再評価は上限 2 回。各再評価で前回指摘の引用と台帳に記録した再評価回数の整合を検査し、引用なし・カウンタリセットは上限到達扱いで FAIL 格上げ。

scoring.md / SKILL.md

ゲートのバイパス — 監査できない工程

タスクごとに一意名の台帳を作り、verdict・スコア率・証拠スニペット・再評価回数・evaluator 再実行出力を記録。工程飛ばしと自己承認を監査証跡で防ぐ。

SKILL.md
第八条

Hooks — モデルの規律を機械で補強する

ここまでの防衛は「モデルが指示に従う」前提の規律。hooks はその外側で、従わなかった場合も機械的に止める最後の層。現在はすべてブロック型の3本。

hookこの仕組みとの関係
bash-guard.sh破壊的コマンド・機密ファイルの読取/持ち出しをブロック（rm はプロジェクト配下の子要素のみ許可）— 禁止事項（憲法）の機械的執行

補遺

用語集 — 本文で使う略語・固有語

本文はグロスを外して用語をここに集約した。初出で意味が要るものは以下を参照。

Main
ユーザーと対話する親セッション（オーケストレータ）。全 agent の起動主体で、~/.claude/CLAUDE.md がその行動規則。
台帳（ledger）
タスクごとに作る監査記録ファイル .tdd-gates/ledger-<slug>.md。verdict・スコア率・証拠・再評価回数・evaluator 再実行出力を記録する。
verdict
各ゲートの判定結果。PASS（80%以上＋Critical全達成）／CONDITIONAL（60–79%）／FAIL（60%未満または Critical 未達）。
Plan モード
実装前に計画だけ立てて承認を得る、Claude Code の読取専用モード。
Generator／Evaluator
実装役 tdd-generator と採点役 tdd-evaluator の本文中の呼称。
opus／sonnet／haiku
Claude モデルの階層。opus＞sonnet＞haiku の順に高機能・高コスト。
ADR
Architecture Decision Record。設計判断の記録。
3層（unit／integration／e2e）
テスト階層。unit＝業務ロジック、integration＝結合検証（最も厚くする層）、e2e＝最小限の主要導線。
SendMessage
既存エージェントへ同一コンテキストのまま追加指示を送る仕組み。
scratchpad
セッションの一時ファイル置き場。reviewer が所見を書き、evaluator が直接 Read する。
frontmatter
エージェント定義ファイル冒頭のメタデータ・制約。
グリーン
CI が成功で完走した状態。push 後の非同期事象。
Stop & Ask
非自明な要件欠落を自動 FAIL せずユーザーに確認して解消する運び。
プロファイル駆動
具体的なテスト実行コマンド等を言語プロファイルが供給する仕組み。
第九条

参照関係図の検証と改善余地

第三条（§3）の図が改訂前に持っていた全18エッジを、ソース（SKILL.md・gates.md・各 agent 定義）と1本ずつ突合した。結果は17本が記載どおり正確で、残る1本（doc-verifier が参照する範囲）は §3 内の注記で是正した。検証に続き、下表の改善6件を 2026-07-06 の改訂で図へ反映した（反映後の図は Main ノードと新規エッジを加えた23本構成）。

改善6件の詳細（検証結果と反映内容）
改善点理由・根拠反映内容
Main（親セッション）ノードの不在 dev-python と doc-verifier に入エッジがなく「使われない agent」に誤読される。実際の起動元は Main（dev-python は Python 実装時に随時起動、doc-verifier は doc-updater の執筆後に Main が別コンテキストで起動 — doc-updater.md）。 最上段に Main ノードを新設し CLAUDE.md はその左へ移動。朱線5本すべての起点を Main とし、dev-python・doc-verifier 行き先の朱線を新設した。
review-* → tdd-evaluator の所見フローが未描画 Gate 3/8 の実データフロー（reviewer が scratchpad の所見ファイルに書き、evaluator がそれを直接 Read する — SKILL.md）に対応する矢印がなく、「レビュー単独（evaluator 集約）」ラベルの実体が図にない。 review-* 左辺 → tdd-evaluator 右辺に点線＋専用マーカーで所見フローを追加し、凡例にも「点線＝レビュー所見の受け渡し」を追記した。
review-maintainability の数値基準（関数50行等）を dev-python・planner・gates.md が参照する横断関係が未描画 第一条（§1）の単一ソース規約と図の対応が欠ける。 agent 間の横断エッジは追加せず、脚注カード「数値基準の横断参照は図に引いていない」を新設して代えた。
下段の X 字交差 L3 のチップ並び（planner・review・python-*・doc-*）が L2 のノード並びと逆転しており、交差が構造的に発生する。 チップ順を planner.md → python-* ×4 → review.md → doc-* ×5 へ並び替え、交差を dev-python↔python-* と review-*↔review.md の1箇所のみに削減した。
L2 ノードの並び順が第四条（§4）の表（review-* → dev-python → trivial-executor → doc-updater → doc-verifier）と不一致 図と表を突き合わせるとき、並び順の食い違いで探し直しが起きる。 L2 ノードを planner → tdd-generator → tdd-evaluator → review-* → dev-python → doc-updater → doc-verifier → trivial-executor の順に統一し、§4 の表・§1 の列挙も同順へ揃えた。
Gate 7 の欠番感 図では G7（frontend-design スキル担当）だけ行き先の矢印がなく、脚注頼みになっている。 SKILL.md 脇に小チップ「G7 → frontend-design」を追加し、脚注の文言も「図では小チップで表示」へ更新した。
