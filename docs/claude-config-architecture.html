第一条

四層の役割分担 — 憲法・工程書・実行役・正典

この設定は「誰が・何を・どの順で」を4つの層に分けて持つ。各層は自分の責務だけを記述し、詳細は下の層へ委譲する。同じ定義が二箇所に現れないことが、この構成の背骨。

憲 法

~/.claude/CLAUDE.md

基本理念・禁止事項・比例ルール（作業の入口分岐）・エージェント/スキル索引。手順の詳細は持たず、正典への委譲だけを書く（例: reviewer 構成の正典は gates.md）。

工 程 書

skills（1本 + プラグイン）

tdd-gates＝TDD×9ゲートのオーケストレータ（フロー図の正典）。工程の順序と強制力はここが持つ。

実 行 役

agents（11本）

planner／gate-generator／gate-evaluator／5次元 *-reviewer／python-dev／trivial-executor／doc-updater。各自は差分となる責務だけを持ち、共通手順は references を読む。

正 典

references（二層）

agents/references/（7本）＝深い作法・共通骨格。skills/tdd-gates/references/＝ゲート定義・採点基準・言語プロファイル。定義の「単一の真実源」。

単一ソース規約 — 「その定義はどこが正か」を一箇所に固定する
  • ゲート定義・Critical 条件 → skills/tdd-gates/references/gates.md（CLAUDE.md からも正典委譲）
  • 採点閾値・スコアカード形式 → skills/tdd-gates/references/scoring.md
  • ゲート順序・フロー図 → skills/tdd-gates/SKILL.md（gates.md は重複掲載しない）
  • 行数・ネスト深さ等の数値基準 → maintainability-reviewer（python-dev・planner・gates.md が参照）
  • レビューの共通手順・出力形式 → agents/references/review.md（5本の reviewer が共有）
第二条

比例ルール — 入口はここで分岐する

変更規模で最初に分岐し、規模で変えるのは「委任するか否か」ではなくパイプラインの深さ。迷ったら substantial（安全側）に倒す。

規模定義工程（→ 呼ばれる skill / agent）
trivial 数行・既存パターン踏襲・テスト不要（設定/ドキュメント/コメント/誤記） trivial-executor（haiku）へ一括委任。Plan モード・reviewer 群は省略可
small trivial 超だが substantial 未満（差分小・既存テストあり・公開 IF 不変） tdd-gates の Gate 4–5（RED→GREEN）中心の簡略パイプ。Gate 3/8 の省略可否は Gate 1 で明示承認し台帳に記録
substantial 新規ロジック・複数ファイル横断・公開 IF 変更・非自明なバグ修正 Plan モード → tdd-gates フル9ゲート。採点は実装者と別コンテキストで自己承認を排除
レビュー単独 フル工程は不要だがレビューだけ独立して欲しい Main が *-reviewer を並列起動 → gate-evaluator が集約採点
第三条

参照関係図 — 誰が誰を起動し、誰が何を読むか

朱線＝CLAUDE.md 発のルーティング（比例ルールの分岐）、実線＝skill によるエージェント起動、破線＝正典の参照（Read）。上から下へ「憲法 → 工程書 → 実行役 → 正典」。

CLAUDE.md 発のルーティング（比例ルール） skill によるエージェント起動（ゲート番号つき） 正典の参照（Read）
pytest は意図的に「二枚」ある skills/tdd-gates/references/profiles/pytest.md は薄い「ゲート用グルー」（パス→テスト種別・実行コマンド・合格ログ判定）。深い作法は agents/references/pytest.md が単一ソースで、グルー側が委譲する。gate-generator は両方、python-dev は後者のみを読む。
新言語は _template.md から拡張する 言語プロファイルは現状 pytest のみ。未対応言語では profiles/_template.md から起草し、テスト実行コマンドと合格ログ形式をユーザー承認後にゲート開始する（pytest 前提で進めない）。
Gate 7 だけはエージェントでなくスキルが担う UI/UX レビュー（Gate 7）は frontend-design スキル＋独立サブエージェントの敵対的クロスレビューで行う（図では省略）。
受け入れ後のドキュメント同期は憲法直轄でもある tdd-gates 外の変更でも、ユーザー受け入れ OK 後の関連ドキュメント更新は CLAUDE.md のタスク実行手順が doc-updater へ委任する。

第五条

TDD 9ゲート検査ライン — 順序そのものが防衛

「動くように見えて中身がバグだらけ」を防ぐには、工程を飛ばせないことが第一の防衛になる。各ゲートは Critical 条件を持ち、1つでも未達なら他が満点でも即 FAIL。番号が朱のゲートは Critical が特に要となる工程。

1·2

計画

PLANsubstantial 必須

受入基準の言語化・テストシナリオ設計・対象ファイルとテスト種別の対応付け。planner が素材を起草し、最終計画は Main が組む。簡略化の可否（small の Gate 3/8 省略等）はここで明示承認し台帳に記録する。

planner 起草gate-evaluator 採点
3

事前レビュー

PRE-REVIEWsubstantial 省略不可

テストを書く前に、既存コード・DB・セキュリティ・設計を4次元並列でレビューする（test-reviewer はまだ入らない）。実装前の唯一の実レビュー層。

correctnesssecurityperformancemaintainability→ gate-evaluator 集約
4

RED — 失敗するテストを先に書く

RED実失敗ログはいかなる場合も省略不可

実装コードは一切書かず、テストだけを書いて実行する。この時点のテスト本文（パス＋内容またはハッシュ）を台帳に固定記録し、後のゲートで照合する。

gate-generator
Critical: 実行ログで「実際に・意味のある理由で」失敗していること。assert False 等の無条件失敗、collected 0 items、import ERROR で失敗を装うのは無効。
5

GREEN — テストを通す最小限の実装

GREENsmall は G4–5 中心の簡略可

Gate 4 と同一の Generator を SendMessage で継続し、最小実装で全テストを通す。evaluator が git diff で「テストに対応しない実装ブランチ（過剰実装）」と「テストの書き換え」を照合する。

gate-generator（同一継続）
Critical: 全テスト通過の実行ログ＋Gate 4 で固定したテスト本文と差分がないこと。
6

REFACTOR — 振る舞い不変で整理

REFACTOR

重複除去・命名改善のみ。evaluator が git diff でテストファイル不変と「既存テストから到達不能な新機能の混入なし」を確認する（差分カバレッジで裏取り推奨）。

gate-generator（同一継続）
Critical: 振る舞い不変 — テスト不変のまま全テスト通過。
7

UI/UX レビュー

UI/UXe2e 時のみ（条件付き強制あり）

AI 臭さの排除・レイアウト・導線を検査。unit と自己申告してもテンプレート/ルーティング/ビュー層に変更があれば必須化され、申告による回避はできない。

frontend-design スキル敵対的クロスレビュー
8

差し戻し判定 — 最終ゲート

FINAL REVIEWsubstantial 省略不可

変更差分全体を5次元並列でレビューし、evaluator が集約採点してマージ可否を判定する。ここで test-reviewer が加わり、偽装テスト（assert なし・常に真・実装の写経）の検出が Critical になる。

correctnesssecurityperformancemaintainabilitytest→ gate-evaluator 集約
Critical: 仕様不適合ゼロ・既存回帰ゼロ・偽装テスト検出ゼロ。カバレッジ 80%（行・ブランチ）未満は test-quality 次元で減点。
✓
ユーザー受け入れ確認 — Gate 8 PASS 後・Gate 9 の前に必ず挟む。NG なら修正へ差し戻し、ドキュメントは更新しない。

DOC SYNCdoc 影響なしならスキップ

README・ガイド・コードマップ・API 仕様を変更内容に同期する。

第六条

判定と検印 — スコア率を Critical が上書きする

各ゲートの採点は gate-evaluator が scoring.md 準拠で行う。重要なのは、スコア率がどれだけ高くても Critical 1件で即 FAIL になること — 「点数を稼いで欠陥をすり抜ける」経路を断つ。

合PASS

PASS

スコア率 80% 以上かつ Critical 全達成。台帳の各 PASS には evaluator 自身が再実行した検証コマンドの出力行の添付が必須 — 実行痕跡のない PASS は無効。

条CONDITIONAL

CONDITIONAL

スコア率 60–79%。差し戻して再評価するが上限 2 回。再評価では前回スコアカードの指摘を引用し台帳カウンタと整合確認 — 引用なし・カウンタ不整合は上限到達扱い（無限往復の防止）。

否FAIL

FAIL

スコア率 60% 未満、または Critical 未達（他が満点でも即 FAIL）、または CONDITIONAL 2 回超。工程を差し戻す。

第七条

破り方と防ぎ方 — AI 駆動開発の品質問題への12の防衛

この仕組みは「AI 実装者がどうサボり・どう取り繕うか」を先回りして設計されている。各カードは 破り方（起きうる品質問題）→ それを構造的に潰す仕組み の対。

自己承認 — 実装者が自分で合格を出す

Generator（実装）と Evaluator（採点）を必ず別のサブエージェント・別コンテキストで起動。さらに reviewer の所見は scratchpad ファイル経由で evaluator が直接 Read し、Main が要約・改変する経路を外す（工程当事者が critical 所見を軟化させる余地を消す）。

gates.md / SKILL.md / gate-evaluator.md

口先報告 — 「テストは通るはず」

証拠主義: RED / GREEN は実行コマンド＋出力スニペットを台帳に残す。ログのない主張は該当項目 0 点。

SKILL.md / scoring.md

ログ捏造 — それらしい出力を貼る

evaluator は貼られたログ・差分を信頼せず、Critical 項目は自分で Bash 再実行して再現した出力のみを証拠にする。再現不能・テストパス不一致は 0 点。

gate-evaluator.md / SKILL.md

採点者の不正 — 通したいがためにテストを直す

evaluator は Write/Edit ツールを持たず、Bash でも sed -i・リダイレクト・mv/rm・git add/commit を禁止。読取と検証実行しかできない。

gate-evaluator.md（frontmatter・制約）

RED 偽装 — 形だけ失敗させる

assert False・トートロジー・例外 raise だけの無条件失敗は Critical 未達。collected 0 items や import ERROR で「失敗した」ことにするのも無効。

gates.md / profiles/pytest.md

assert 骨抜き — GREEN でテストを弱める

Gate 4 時点のテスト本文を台帳に固定記録し、Gate 5・6 で照合。差分があれば Critical 未達。

SKILL.md / gates.md / gate-evaluator.md

REFACTOR への新機能混入

Gate 6 の Critical は振る舞い不変。evaluator が git diff でテスト不変と、既存テストから到達不能な新規ブランチ・機能追加がないことを確認する。

gates.md

過剰実装 — テストにない機能の作り込み

Gate 5 で evaluator が git diff にテスト非対応の実装ブランチがないか（最小性）を確認する。

gates.md / gate-evaluator.md

形骸化テスト — 常に PASS する飾り

Gate 8 に偽装テスト検出の Critical: assert なし・常に真・実装の写経を検出したら即 FAIL。

gates.md

スコア稼ぎ — 点数で欠陥を薄める

Critical 即 FAIL がスコア率を上書きする。1 件でも未達なら他が満点でも FAIL。

scoring.md

無限往復 — CONDITIONAL で粘る

再評価は上限 2 回。各再評価で前回指摘の引用と台帳カウンタの整合を検査し、引用なし・カウンタリセットは上限到達扱いで FAIL 格上げ。

scoring.md / SKILL.md

ゲートのバイパス — 監査できない工程

タスクごとに一意名の台帳（ledger） tdd-gates-ledger-<slug>.md を作り、verdict・スコア率・証拠スニペット・再評価回数・evaluator 再実行出力を記録。工程飛ばしと自己承認を監査証跡で防ぐ。

SKILL.md
第八条

Hooks — モデルの規律を機械で補強する

ここまでの防衛は「モデルが指示に従う」前提の規律。hooks はその外側で、従わなかった場合も機械的に止める・促す最後の層。

hookこの仕組みとの関係
tdd-gates-nudge.shtests/ 外の Python 実装ファイル編集を検知し、tdd-gates の使用を非ブロッキングで促す — 比例ルールの飛ばし忘れ対策
bash-guard.sh / workspace-guard.sh破壊的コマンド・プロジェクト外への書込をブロック — 禁止事項（憲法）の機械的執行
venv-guard.shvenv 外への pip install をブロック — 環境汚染の防止
context7-plan-remind.sh計画スキル実行前に context7 でのライブラリ事実確認を促す — 計画段階の思い込み対策
要するに — CLAUDE.md（憲法）が規模で入口を分岐し、tdd-gates（工程書）がゲートの順序を強制し、agent（実行役）は実装と採点を別人格に分け、references（正典）が定義の重複を許さない。その全工程を台帳と検印（証拠つき PASS / Critical 即 FAIL）が監査し、hooks が外側から機械的に支える。「動くように見える」だけのコードが通り抜ける穴を、層ごとに一つずつ塞いだ構成になっている。
