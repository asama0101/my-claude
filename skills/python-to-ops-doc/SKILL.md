# python-to-ops-doc

Pythonプロジェクトのソースコード・スクリプト・設定ファイル・マイグレーションを読み解き、**運用者・監視者向けの引継書**をMarkdownで生成するスキル。

## このスキルが目指すもの

引継書の読者は「コードを読まない運用担当者」であり、システムを**安全に運転し、正常に動いているか確認し、異常時に適切に動ける**ことを目的とする。設計思想やコード解説ではなく、**実機オペレーションに直結する情報**だけを書く。

迷ったら次の問いを自分に投げかける：
- 「このページだけ印刷して夜中の障害対応に持っていって、現場が動けるか？」
- 「コードを開かずに、この手順だけで起動・停止・状態確認ができるか？」

答えがNoなら情報が足りていない。逆に、コードの内部実装や設計判断の理由は引継書には不要。

## 入力

- ユーザーが指定したPythonプロジェクトのディレクトリ・ファイル群
- 典型的に含まれるもの: `src/`配下のPythonコード、`scripts/`のシェル/Python補助スクリプト、`config/`の環境変数テンプレート、`migrations/`のDBマイグレーション、`docker-compose.yml`、`README.md`、`pyproject.toml`、cron設定例など

## 出力

**単一の Markdown ファイル** を 1 本生成する。出力先は次の優先順位で決定する。すべての相対パスはプロジェクトルートから解決する。

### 出力先の決定ルール

| # | ユーザー指定 | 例 | 出力先 |
|---|---|---|---|
| 1 | 完全パス（ディレクトリ＋ファイル名） | `docs/runbook.md` / `/abs/path/runbook.md` | そのパスに従う |
| 2 | ディレクトリのみ | `docs/operations/` | 指定ディレクトリ + デフォルトファイル名 `operations.md` |
| 3 | ファイル名のみ | `runbook.md` | デフォルト配置先 `docs/` + 指定ファイル名 |
| 4 | 指定なし | — | `docs/operations.md` |

ディレクトリは末尾 `/` の有無、または「指定がディレクトリとして既存である」ことで判定する。判別が紛らわしい場合（例: `docs/runbook` のように拡張子なしの文字列）は、生成前にユーザーへ「ディレクトリ扱いとファイル名扱いのどちらか」を確認する。

### ディレクトリ自動作成

出力先の親ディレクトリ（`docs/` など）が存在しない場合は **作成してから書き込む**。プロジェクト構成を尊重したいので、`docs/` 以外の新規ディレクトリを勝手に切らない（例: `docs/operations/` を新設するのは、ユーザーが明示的に指定したときだけ）。

### 既存ファイルの上書きガード

出力先のファイルが **既に存在する場合は、上書き前に必ずユーザーに確認** する。運用引継書は版が重要であり、誤った上書きで前任者の手当てや「未確認」項目への手書きメモが消える事故は致命的になる。確認なしで続行してよいのは、ユーザーが事前に「上書きしてよい」と明言した場合のみ。

確認時には次の情報を併記する:

- 既存ファイルの最終更新日（`stat` 等で取得可能なら）
- 既存ファイルの先頭数行（プロジェクト固有の追記がないか把握できる程度）
- 提案する代替案（例: `operations_YYYYMMDD.md` として別名保存して差分確認、追記マージ、上書き）

## 開発者向け／運用者向けの切り分け（**収録範囲を決めるための最重要ルール**）

引継書は **本番運用で実際に使う情報だけ** を載せる場所であり、開発時の補助ツール・テスト・ベンチマークは収録しない。  
ファイル名や置き場所だけで判断せず、**必ず中身を確認** してから取捨選択する（「`scripts/` だから運用手順」「`init_*` だから運用手順」のような早とちりが最も多い失敗パターン）。

### 判定の優先順位

1. **その操作は本番環境で実施されるか？** Yes ならば運用向け。
2. **その操作は開発・QA・性能評価のために存在するか？** Yes ならば開発向け（収録しない）。
3. **判別が難しい場合**は、運用引継書には載せず「未確認（開発用か運用用か要ヒアリング）」と明記する。憶測で運用手順として書いてしまうと、運用者がテストスクリプトを本番で実行する事故につながる。

### よくある分類のヒント

**運用向け（引継書に収録すべき）**:
- 5 分ごと・日次など **定期実行されるエントリポイント**（cron 行・systemd タイマ・ワーカープロセス）
- **本番デプロイ手順**（`docker compose up` など、本番環境で実行されるもの）
- **マイグレーション** `alembic upgrade head`（本番のスキーマ変更で使われる）
- **DB バックアップ／リストア** の本番手順
- **サンプルデータ生成** など本番では使わないものは含めない（運用者には不要）

**開発向け（引継書に収録しない、ただし「これらは収録しない」旨は注記してよい）**:
- `tests/` ディレクトリ配下のすべて（`pytest` 用のテストコード）
- `scripts/test_*.sh` / `scripts/test_*.py` のように **手動テスト用** スクリプト（`docs/testing_*.md` に対応するもの含む）
- `scripts/measure_*.sh` / `scripts/bench_*` などの **性能測定** スクリプト
- `scripts/gen_sample_*.py` などの **サンプルデータ生成** スクリプト
- `dev/`、`examples/`、`benchmarks/` ディレクトリ
- `pyproject.toml` の `[tool.ruff]` `[tool.pytest]` などの **開発ツール設定**
- ローカル開発用の `docker-compose.override.yml` など

### 判定の手順

1. プロジェクトを俯瞰したら、`scripts/`・`tests/`・`bin/`・`tools/` など補助ディレクトリの **各ファイル冒頭の docstring・コメント・README 該当箇所を読む**。
2. README・`docs/` 内に「テスト」「性能測定」「動作確認」などの語と一緒に紹介されているスクリプトは、まず開発向けの可能性が高い。
3. **本番 cron 例や systemd ユニット** に登場するコマンドは運用向け。それ以外で「`bash scripts/foo.sh` を打つ」だけのものは、本番運用ループに組み込まれていない可能性が高い。
4. `if __name__ == "__main__":` を持つ Python ファイルでも、用途は様々（運用ワーカー／CLI ツール／開発スクリプト）。docstring と引数で判別する。
5. 迷ったときは **収録しない** 側に倒し、「未確認」項目に転記する。引継書の信頼性は「載っていることが正しい」で担保され、過剰収録は混乱の原因になる。

### 章ごとの当てはめ

- 「3. 起動・停止手順」「4. 定期実行ジョブ」: **本番で動かすコマンドだけ**。開発用の手動取り込みは入れない（あるいは「※本番では使用しない」と明示）。
- 「9. バックアップ・リカバリ手順」: 本番の運用手順だけ。`scripts/init_db.sh` のような **データを破壊する初期化スクリプト** は、運用向けに見えても扱い注意。引継書には「※本番DBに対しては絶対に実行しない（開発環境の再初期化用）」と必ず注記する。
- 「2-3. ディレクトリ構成」: ツリーには出してよいが、開発用エントリは「（開発専用）」と注釈する。運用者が触る必要があるのは **どれか** をひと目で分かるようにする。

## 必須: 冒頭の目次

ファイル先頭（タイトル直後）に **目次（Markdown リンク付き）** を必ず置く。引継書は分量が多くなるので、現場が必要な章にすぐ飛べることが重要。各章タイトルを `- [N. 章タイトル](#N-章タイトル)` 形式で並べる（GitHub-Flavored Markdownのアンカー仕様で動作する形）。

## 引継書の構成

以下の順序で章立てする。**`連絡先・エスカレーション`は記載しない**（このスキルでは扱わない）。
当てはまる情報がコードから読み取れない章は、章ごと省略するのではなく **「該当なし」または「未確認（要ヒアリング）」と明記** する。運用引継ぎでは「無いことが分かっていること」自体が重要な情報になる。

### 1. システム概要
- 一言サマリ（このシステムは何をしているか／何のためにあるか、運用視点で）
- 全体像（入力→処理→出力 のフロー、関連する外部システム）
- 構成要素の対応表（コンポーネント名 / 実体 / 役割）
  - 例: 「取り込みワーカー / `src/ingest/worker.py` / 5分おきにCSVをDBへ取り込む」
- 想定する処理サイクル（常時稼働 / cron / イベントドリブン など）

### 2. 前提知識（このシステムを動かす前に押さえること）

引継書は **これ単体で読めば運用者・監視者が動ける** ことを目的とするため、**システムに関する事前知識ゼロでも理解できる教育資料的な節** を本章に置く。コードを開かなくてもシステムの「形」が頭に入るよう、図と表を使って書く。

#### 2-1. データフロー図
入力（どこから何が来る）→ 処理（どのコンポーネントが何をする）→ 出力（どこに何が残る）の流れを **ASCII 図または Mermaid図** で示す。
矢印には何が流れるか（ファイル名・データ種別）を、ボックスには「誰が何をするか」を1行で書く。

例:
```
[帯域制御装置]
   │ SFTP/SCP で .csv.gz を 5 分ごとに送信
   ▼
[incoming/]  ─── cron で起動 ───▶  [取り込みワーカー]
                                       │ パース → DB INSERT
                                       ▼
                                   [TimescaleDB]
                                       │
                                       ▼
                              [外部システムが SQL でクエリ]
```

#### 2-2. 正常時／異常時のファイル遷移
ファイルやレコードが「処理されていく途中でどこにいるか」を **状態遷移図** で示す。  
正常パス（成功）と異常パス（失敗）の両方を描く。各状態に「自動で消えるか／手動対応が必要か」を必ず併記する。

例:
```
[incoming/<file>.csv.gz]
   │
   ├─ 正常 ─▶ [processing/]（処理中・自動移動）
   │             │ DB 書き込み成功
   │             ▼
   │           [自動削除]
   │
   └─ 異常 ─▶ [error/<file>.csv.gz]（隔離・**手動対応が必要**）
                  │ Syslog アラート送信
                  ▼
              [運用者が原因確認 → incoming/ に戻して再投入]
```

#### 2-3. ディレクトリ構成
プロジェクトのトップレベル〜2階層程度のディレクトリ／重要ファイルをツリー形式で列挙し、**各エントリに「運用者から見た役割」を一行で注釈** する（コードの解説ではなく、「ここに何が入る／ここを触っていいか」の運用視点）。

例:
```
traffic-reports-db/
├── incoming/           # CSV.gz の着地先（5分おきに装置から自動配置される）
│   ├── processing/     # 処理中ファイル（自動管理。基本さわらない）
│   └── error/          # 取り込み失敗ファイル（**運用者が中身を確認・対処**）
├── config/app.env      # 実環境用の設定ファイル（接続先・閾値）
├── docker-compose.yml  # コンテナの定義（DB と取り込みアプリ）
└── docs/operations.md  # 既存の運用ドキュメント
```

#### 2-4. 実行環境（コンテナ／プロセス／ホスト構成）
- **どこで動いているか**: ホストOS／コンテナ／クラウドリソースの構成を箇条書きで明記
- **コンテナ構成があれば** 各サービスの役割表を作る（サービス名 / イメージ / 起動方法 / 永続化先）
  - 例: 「`db` / `timescale/timescaledb-ha:pg16` / 常駐 / Docker volume `pgdata`」
  - 例: 「`app` / 自前ビルド / cron 起動・1回実行で終了 / `incoming/` を bind mount」
- **プロセスの起動主体**: cron／systemd／手動／コンテナのどれが起動の発火点か
- **ホスト側で稼働するもの／コンテナ側で稼働するもの** の境界を明確にする（運用者が SSH ログインして見るのはどちらか）
- 関連ボリューム・bind mount・ネットワーク（ポート開放）も含める

#### 2-5. 用語集（任意・必要に応じて）
コード由来の専門語（`hypertable`、`Continuous Aggregate`、`flock`、`冪等処理`、`bind mount` など）が引継書本文で避けられない場合は、ここで **運用者目線の1〜2行** で説明する。「正確な定義」より「これを知らないと運用判断ができない事項」を優先。

### 3. 起動・停止手順
- 起動コマンド（コピペで動く形）
- 停止コマンド（同上）
- 通常起動と緊急停止の違いがあれば両方
- 起動後すぐに確認すべきこと（ログ・プロセス・接続）

### 4. 定期実行ジョブ
- cron / スケジューラ / Continuous Aggregate などの一覧
- 各ジョブの **実行頻度・実行コマンド・想定実行時間・失敗時の挙動** を表で整理
- ジョブが重なったとき・遅延したときの動作（多重起動防止の仕組みなど）

### 5. 正常性確認方法（**重要**）
日次／時間ごとの監視作業として、運用者がそのまま実行できるチェックリスト形式で書く。

- **プロセスが生きているか**: 確認コマンドと期待される出力
- **データが流れているか**: 直近N分の処理件数、最終処理時刻のクエリ
- **エラーが出ていないか**: ログ・キュー・エラーディレクトリ等の確認方法
- **リソースに余裕があるか**: ディスク使用量・DB容量・処理遅延
- 各項目に「正常な状態の例」と「異常を疑う閾値」を併記

### 6. 監視ポイント
- ログの出力先（ファイルパス・Syslog・標準出力など）と、注目すべきログ行のパターン
- メトリクス（取得方法・取得元・典型値）
- 監視推奨項目（アラート対象にすべき条件）
- ログのローテーション設定があればその挙動

### 7. エラー・例外一覧と対応
- ソースコード中で送出される **運用者が遭遇しうる主要なエラー** を抽出
- 各エラーについて **発生条件・観測方法・対応手順** を記述
- アラート（Syslog送信などコード上で能動的に通知しているもの）は別途明記

### 8. よくあるトラブルと対処（FAQ／ランブック）
- コードを読み取って分かる典型的な障害シナリオ
  - 例: 「処理待ちファイルが溜まり続ける」「マイグレーション失敗時の戻し方」「DBが繋がらない」
- 各シナリオについて **症状 → 切り分け手順 → 復旧手順** を書く
- 該当する設定値（閾値・パスなど）を本文中に直接書く（「config/app.envを参照」で済まさない）

### 9. バックアップ・リカバリ手順
- データの永続化先（DB・ファイル）
- バックアップが取られているか / 取り方が定まっているか（コードから読み取れる範囲で）
- 既知のリカバリ手順（マイグレーションのロールバック、DB初期化スクリプト等）
- 取り込み済みファイルや中間ファイルの再投入方法

### 10. 設定ファイル・環境変数の説明
- すべての環境変数を表で列挙: **変数名 / デフォルト値 / 必須か / 用途 / 変更時の影響**
- 設定ファイルのパスと、**運用中に変えてよい項目／変えてはいけない項目** を区別する
- 変更時に再起動が必要かを明記

### 11. 外部依存（DB・API・クラウドリソース）
- 接続先一覧: 種別 / ホスト / ポート / 用途 / 障害時の影響範囲
- 認証情報の管理方法（運用者が触る必要があるか）
- 外部依存が落ちたときに本システムがどう振る舞うか（リトライ・エラー隔離など）

### 12. 変更履歴
- このドキュメント自身の変更履歴を残せるよう、空の表（日付・変更者・変更内容）を用意する
- 初版生成時は「YYYY-MM-DD 初版作成 / python-to-ops-doc による自動生成」を記入

## 文体・粒度のルール

- **すべて日本語**で書く。コマンド・ファイルパス・コードシンボルだけ英語のまま。
- **コードを読まない読者向け**: 「`asyncpg.Pool`を使って…」のような実装語彙は避け、「DB接続の管理は自動で行われる」のように動作で記述する。
- **コマンドは必ずコピペ可能な完全形**で書く。`<your-host>`などのプレースホルダを使う場合は、すぐ下に「※`<your-host>`はDBサーバーのホスト名を入れる」と注釈を付ける。
- **「〜できる」より「〜する」**: 手順は断定的に。曖昧表現は運用ミスを生む。
- **テーブル/箇条書きを多用**: 散文より、コマンドや設定値は表で並べた方が現場で読みやすい。
- **コードの引用は最小限**に。引用するときも「ここを見れば実装が分かる」ではなく、運用判断に必要な値（閾値・タイムアウト・リトライ回数など）を抜き出して載せる。

## 進め方

1. **プロジェクトを俯瞰する**: `README.md` / `pyproject.toml` / `docker-compose.yml` / ディレクトリ構成 / `scripts/` / `config/` をまず読む。プロジェクトの輪郭をつかむ。
2. **開発向け／運用向けを切り分ける**: `scripts/` `tests/` `bin/` 配下の各ファイルの冒頭 docstring・コメント・README 該当箇所を **必ず読み**、「本番で実行される」ものだけを引継書の収録対象にする。詳細は **冒頭の「開発者向け／運用者向けの切り分け」節** を参照。判別不能なものは「未確認」に倒す。
3. **エントリポイントを特定する**: `src/`配下で `if __name__ == "__main__"` を持つもの、`pyproject.toml` の `[project.scripts]`、Dockerfileの`CMD`、シェルスクリプトから呼ばれるPythonモジュールなど。これが「何をどう動かすか」の起点になる（**ただし開発用エントリは除外**）。
4. **運用に効く情報を拾う**: ログ出力箇所、`raise`される例外、`alert`/`syslog`などの通知箇所、環境変数の参照箇所（`os.getenv`）、cron例（READMEや本番デプロイ用設定内）、ファイル/ディレクトリの監視先など。

## 例: 良い記述／避けたい記述

**避けたい**:
> ワーカーは `asyncpg` のコネクションプールを利用してCOPYコマンドで効率的に書き込みを行う。

**良い**:
> 取り込みワーカーは1ファイルあたり数秒〜数十秒でDBに書き込みを完了する。書き込み中はDBに一時的な負荷がかかる（CPU使用率が一時的に上昇する場合がある）。

---

**避けたい**:
> エラー時はSyslogアラートが送信される。

**良い**:
> 取り込みに失敗するとSyslog（`<SYSLOG_HOST>:514`、`LOG_USER`ファシリティ・`LOG_ERR`レベル）に次の形式でメッセージが送信される:
> ```
> ingest failed for FLOW-router01-20260101.csv.gz: <エラー内容>
> ```
> 監視サーバー側でこのメッセージをトリガーにオペレーターに通知する設定が必要。

---

**避けたい**:
> ファイルが溜まったらアラートが上がる。

**良い**:
> `incoming/error/` 配下のファイル数が `INGEST_ERROR_THRESHOLD`（デフォルト 10 件）以上になると、Syslogに「error backlog: N files」というメッセージが送信される。この閾値は `config/app.env` で変更でき、変更後は次回ワーカー起動時から反映される（再起動不要）。
