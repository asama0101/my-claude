### 指摘事項

#### 🔴 重要（誤り・手順不能・参照不能）

**#1 [内容の正確性]** §2-4（ネットワーク）行 179、§6-1（ログ一覧）行 414、§11（外部依存）行 705

Syslog のデフォルトポートとして「514」と記載されているが、コード（`src/alerts.py`）および `config/app.env.example` のデフォルト値はいずれも **5514**（非特権ポート）。この誤記により、運用者が「デフォルト 514 で動いているはず」と誤解し、監視・ファイアウォール設定を誤る可能性がある。

誤記箇所の詳細:
- **§2-4 行 179**: `Syslog ポート（既定 514/UDP）`
- **§6-1 行 414**: `既定 localhost:514 UDP`
- **§11 行 705（表の「ポート」列）**: `SYSLOG_PORT（既定 514）`

→ 修正案:
  - 行 179: `Syslog ポート（既定 **5514**/UDP）`
  - 行 414: `既定 localhost:**5514** UDP`
  - 行 705: `SYSLOG_PORT（既定 **5514**）`

補足: §10-2 環境変数表（行 686）は `5514（example）／514（実環境）` と正しく記載されているため、他箇所との整合性として「example の既定は 5514」に統一すること。

**#2 [内容の正確性]** §8-6（マイグレーション失敗時の戻し方）行 594

「圧縮済 chunk が存在する状態で downgrade すると失敗する場合がある（**migration 0010 の docstring を参照**）」と記載されているが、`migrations/versions/0010_extend_ingest_log.py` は `ingest_log` テーブルの hypertable 化・PK 変更を扱うマイグレーションであり、圧縮 policy とは無関係。さらにこの migration の downgrade は "irreversibility" を理由に `NotImplementedError` を発生させる設計（`raise NotImplementedError("downgrade of 0010 is intentionally not supported")`）であり、「圧縮を解除してから downgrade する」という前段と噛み合わない。

圧縮ポリシーを追加しているのは `migrations/versions/0005_add_compression.py` であり、downgrade 時の decompress 処理もそのファイルに実装されている。

→ 修正案:
```
> 圧縮済 chunk が存在する状態で downgrade すると失敗する場合がある（migration 0005 の downgrade 関数を参照）。圧縮を解除してから downgrade する必要がある。
```

#### 🟡 中程度（抜け穴・混乱の原因）

**#3 [内容の正確性]** §5-2（データが流れているか）行 335 — `files` 列の正常値説明

表の「正常な状態」セルの記述が次のように破損している:

```
600 ファイル/時間（300 装置 × 2 種 × 12 区間/時間 ÷ 12 × ??）= 約 7,200 ファイル/時間を期待。
```

「600 ファイル/時間」という冒頭値が間違い（300 × 2 × 12 = **7,200**）であり、「÷ 12 × ??」は計算式として意味をなさない。運用者がこの値を参照して閾値判断を行う重要箇所であるため、計算式の誤りは監視ミスにつながる。

→ 修正案:
```
7,200 ファイル/時間（300 装置 × 2 種 × 12 区間/時間）を期待。1 時間に **3,600 ファイル未満**は装置側転送異常の疑い
```

#### 🟢 軽微（改善提案・スタイル）

**#4 [記述漏れ・抜け穴]** §4-2（TimescaleDB 内部ジョブ）— `subport_stats_5min` の compression policy 記載なし

§4-2 のバックグラウンドジョブ一覧表に `policy_compression` の対象として `flow_stats`・`subport_stats`・`flow_stats_hourly` の 3 件が列挙されているが、`subport_stats_5min` は compression policy が定義されていない（`migrations/versions/0009_add_subport_stats_5min.py` に `add_compression_policy` の記述なし）。これは仕様通りであり、ドキュメント上も表に記載がないのは正しい。ただし読者が「なぜ 5min だけ圧縮設定がないのか」と疑問に思う可能性がある。

→ 修正案（オプション）: 表の下に注釈を追加する。
```
> ※ `subport_stats_5min` には圧縮ポリシーが設定されていない（93 日保持の期間内に外部システムが参照するため）。
```

**#5 [構成・読みやすさ]** §9-2（バックアップ） `pg_dump` コマンドの `-f` オプション

`pg_dump` サンプルコマンド（行 624-625）:

```bash
docker compose exec -T db pg_dump -U user -F c -f /tmp/traffic.dump traffic
docker compose cp db:/tmp/traffic.dump ./backup/traffic-$(date +%Y%m%d).dump
```

この手順は機能的に正しいが、コンテナ内 `/tmp` に中間ファイルが残るため、大容量（数百 GB 規模）のダンプをコンテナ内 `/tmp` に置くとコンテナのファイルシステム容量を圧迫するリスクがある。また `docker compose cp` コマンドは Docker Compose v2.x 以降が必要な点（一般に普及しているが）も読者によっては確認が必要。

→ 修正案（補足注記として）:
```
> DB サイズが数百 GB に達する本番環境では、コンテナ内 `/tmp` ではなくホスト側にパイプ出力する方式（`docker compose exec -T db pg_dump -U user -F p traffic > ./backup/traffic-$(date +%Y%m%d).sql`）や、volume スナップショットを検討すること。
```
