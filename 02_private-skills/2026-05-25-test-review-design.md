# テスト品質レビュー設計 2026-05-25

## Context

トラヒック統計取り込みシステムは 145件テスト・93% カバレッジ（2026-05-24）に達しているが、
lessons.md に P0（フレーキー3件）・P1（欠落テスト5件）・P2（設計改善）の既知課題が蓄積している。
さらにテスト手法・テストフェーズの役割分担・カバレッジの質（リスクベース）という戦略的観点は
これまで体系的にレビューされていない。

本設計は「品質レビュー」を目的とし、既存テストの問題点を洗い出して Markdown レポートとして出力する。
修正・追加実装は別セッションで行う。

---

## アーキテクチャ：2ラウンド × 10エージェント

### ファイルグループ定義

| グループ | 対象ファイル | テスト件数 | 行数 |
|---------|------------|----------|------|
| Group 1 | `tests/test_flow.py` | 68件 | 1,134行 |
| Group 2 | `tests/test_subport.py` + `tests/test_db.py` + `tests/test_cli.py` + `tests/test_common.py` | ~39件 | 785行 |
| Group 3 | `tests/test_perf_timers.py` + `tests/test_perf_validate.py` + `tests/test_query_bench.py` | ~53件 | 1,086行 |

### Round 1：テストコードレビュー（10エージェント並列）

```
A×B マトリクス（9本）
                  reviewer-correctness  reviewer-test  reviewer-maintainability
Group1: test_flow      Agent 1-A           Agent 1-B        Agent 1-C
Group2: ETL tests      Agent 2-A           Agent 2-B        Agent 2-C
Group3: perf/query     Agent 3-A           Agent 3-B        Agent 3-C

クロスカッティング（1本）
Agent-S：テスト戦略レビュー（全ファイル横断）
```

### Consolidate

メインClaude が10エージェントの出力を受け取り、ドラフトレポートを作成する。

### Round 2：ドラフトレポート品質検査（10エージェント並列）

同じ10エージェントがドラフトレポートをレビューし、漏れ・誤分類・不整合を指摘する。

**Round 2 の各エージェントへのインプット：**
- ドラフトレポート（全文）← 主要インプット
- Round 1 で担当したテストファイル（参照用。自分の担当分の記述を照合する）
- Agent-S のみ：`scripts/` 配下の ST/E2E スクリプト（テスト戦略セクション検証用）

### Final

メインClaude が Round 2 の指摘を反映して最終レポートを確定する。

## 各エージェントの責務

### reviewer-correctness（各グループ）

**担当**：テストが実際にバグを検出できるか

- 常にパスする「偽陽性テスト」（アサーションが不十分で実装バグを見逃す）
- 境界値の判定ロジックの誤り（`>` vs `>=`、`<` vs `<=`）
- モックが実装の副作用を隠蔽して誤検知を防げていないケース
- 論理エラー（期待値の計算ミス、集計ロジックの誤り）

### reviewer-test（各グループ）

**担当**：テストの信頼性・カバレッジ・フィクスチャ品質

- フレーキーテスト（時刻・タイムゾーン・外部リソース依存）
- フィクスチャの分離不足（テスト間の状態汚染）
- エッジケース・境界値の欠落（未テストの分岐パス）
- `monkeypatch` の適切な使用（過剰モック・不十分なモック）
- `clean_db` autouse フィクスチャが全テストで機能しているか

### reviewer-maintainability（各グループ）

**担当**：テストの読みやすさ・構造・一貫性

- テスト名が検証内容を正確に表しているか
- Arrange-Act-Assert パターンの遵守
- フィクスチャ・ヘルパーの重複（DRY 違反）
- テスト間の一貫性（同じ操作に異なるアプローチが混在）
- コメント・docstring の有無と品質

### Agent-S：テスト戦略レビュー（クロスカッティング）

**担当**：テスト手法・フェーズ設計・カバレッジの質

対象：`tests/` 全ファイル + `scripts/run_st.sh` + `scripts/run_e2e_test.py` + `scripts/run_perf_test.py`

- **テスト手法**：unit / integration の境界が適切か。mock vs 実 DB の使い分けが合理的か。ProcessPool/ThreadPool のテストに適切な手法を使っているか
- **テストフェーズ**：pytest（unit/integration）・ST・E2E の役割分担が明確か。重複や抜けがないか
- **カバレッジの質**：93% カバレッジの「リスクベース判断」。高リスク経路（JST/UTC 変換・COPY 失敗・インデックス復元・遅延ファイル検出）が優先的にカバーされているか。残り 7% の未カバー箇所が許容できるか
- **pytest マーカー分類**：`perf` マーカーの運用が適切か。CI での除外設計が機能しているか

## エージェントに提供するコンテキスト

各エージェントには以下を提供する：

1. **対象テストファイル**（グループ該当分を全文）
2. **対応 ETL ソースコード**（何をテストしているか把握用）
   - Group 1 → `etl/flow.py`
   - Group 2 → `etl/subport.py` + `etl/db.py` + `etl/cli.py` + `etl/common.py`
   - Group 3 → `scripts/run_perf_test.py` + `scripts/query_bench.py`
3. **`tests/conftest.py`**（全エージェント共通）
4. **既知の問題リスト**（`lessons.md` の P0/P1/P2 抜粋）—既知 issues との照合用
5. **プロジェクト固有の制約**（JST/UTC 変換・COPY コマンド・現在時バケット除外・ProcessPool 制約など）

### P1（重要）
- ...

### P2（改善推奨）
- ...

### 問題なし
- 特記事項がある場合のみ記述

## Round 2 の各エージェントが確認すること

| エージェント種別 | Round 2 確認観点 |
|--------------|----------------|
| reviewer-correctness × 3 | 発見内容に事実誤認・偽陽性がないか。深刻なバグの見逃しがないか |
| reviewer-test × 3 | 重要なカバレッジ漏れがないか。P0/P1/P2 分類にずれがないか |
| reviewer-maintainability × 3 | 自グループの記述が正確か。グループ間で記述粒度に一貫性があるか |
| Agent-S × 1 | テスト戦略セクションに漏れや矛盾がないか |

---

## 検証方法

1. `docs/superpowers/specs/2026-05-25-test-review.md` が生成されること
2. エグゼクティブサマリーに P0/P1/P2 の件数が明記されていること
3. lessons.md に記録された既知の P0・P1 課題がすべてレポートに言及されていること
4. テスト戦略セクションに unit/integration/ST/E2E の評価が含まれること
5. 各グループのセクションに少なくとも 1件以上の発見または「問題なし」の明示があること
