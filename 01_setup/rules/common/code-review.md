# コードレビュー基準

## 目的

コードレビューは、コードがマージされる前に品質・セキュリティ・保守性を確保します。このルールはいつ・どのようにコードレビューを行うかを定義します。

## レビュー対象

**必須レビュートリガー:**

- コードを書いた・変更した後
- 共有ブランチへのコミット前
- セキュリティに関わるコードが変更された場合（認証、決済、ユーザーデータ）
- アーキテクチャの変更が行われた場合
- プルリクエストをマージする前

**レビュー事前要件:**

レビューを依頼する前に確認:

- 自動チェック（CI/CD）がすべてパスしている
- マージコンフリクトが解消されている
- ブランチがターゲットブランチと同期されている

## レビューチェックリスト

コードを完了とマークする前に:

- [ ] コードが読みやすく適切に命名されている
- [ ] 関数が集中している（50行未満）
- [ ] ファイルが凝集している（800行未満）
- [ ] 深いネストがない（4レベル以下）
- [ ] エラーが明示的に処理されている
- [ ] ハードコードされた秘密情報や認証情報がない
- [ ] console.log やデバッグ文がない
- [ ] 新機能にテストが存在する
- [ ] テストカバレッジが最低80%を満たしている

## セキュリティレビュートリガー

**以下の場合はSTOPして code-reviewer エージェントを使用（セキュリティ観点で）:**

- 認証・認可のコード
- ユーザー入力の処理
- データベースクエリ
- ファイルシステム操作
- 外部API呼び出し
- 暗号化操作
- 決済・金融コード

## レビューの重大度レベル

| レベル | 意味 | アクション |
|--------|------|-----------|
| CRITICAL | セキュリティ脆弱性またはデータ損失リスク | **ブロック** — マージ前に必ず修正 |
| HIGH | バグまたは重大な品質問題 | **警告** — マージ前に修正を推奨 |
| MEDIUM | 保守性の懸念 | **情報** — 修正を検討 |
| LOW | スタイルまたは軽微な提案 | **メモ** — 任意 |

## エージェントの使用

コードレビューに以下のエージェントを使用:

| エージェント | 用途 |
|-------------|------|
| **code-reviewer** | 一般的なコード品質、パターン、ベストプラクティス |

## レビューワークフロー

```
1. git diff で変更を理解する
2. まずセキュリティチェックリストを確認
3. コード品質チェックリストを確認
4. 関連テストを実行
5. カバレッジが80%以上であることを確認
6. 詳細レビューには適切なエージェントを使用
```

## よくある問題の確認事項

### セキュリティ

- ハードコードされた認証情報（APIキー、パスワード、トークン）
- SQLインジェクション（クエリ内での文字列結合）
- XSS脆弱性（エスケープされていないユーザー入力）
- パストラバーサル（サニタイズされていないファイルパス）
- CSRF保護の欠如
- 認証バイパス

### コード品質

- 大きな関数（50行以上）— 小さく分割する
- 大きなファイル（800行以上）— モジュールを抽出する
- 深いネスト（4レベル超）— 早期リターンを使用する
- エラー処理の欠如 — 明示的に処理する
- ミューテーションパターン — イミュータブルな操作を優先する
- テストの欠如 — テストカバレッジを追加する

### パフォーマンス

- N+1クエリ — JOINやバッチ処理を使用する
- ページネーションの欠如 — クエリにLIMITを追加する
- 無制限のクエリ — 制約を追加する
- キャッシングの欠如 — 高コストな操作をキャッシュする

## 承認基準

- **承認**: CRITICAL および HIGH の問題がない
- **警告**: HIGH の問題のみ（注意してマージ）
- **ブロック**: CRITICAL の問題が見つかった
